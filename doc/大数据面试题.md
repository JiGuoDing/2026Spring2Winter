## 1. Spark 面试题

### 1.1 Spark SQL 有哪几种 Join 策略？各自的触发条件与适用场景是什么？

这个问题在流处理和离线一体化面试中很常见。高质量回答的关键不是只背策略名，而是说明 Spark Optimizer (Catalyst + CBO + AQE) 是如何在代价和运行时统计信息基础上选择 Join 物理执行计划的。

先给一个总览：Spark SQL 常见 Join 策略包括 `Broadcast Hash Join (BHJ)`、`Sort Merge Join (SMJ)`、`Shuffle Hash Join (SHJ)`、`Broadcast Nested Loop Join (BNLJ)`、`Shuffle Replicate Nested Loop Join (SRNL)`。其中前三个最核心，后两个主要用于非等值 Join 或笛卡尔场景。

#### 一、Spark SQL Join 策略总览

| Join 策略                                 | 典型触发条件                                                                         | 适用场景                                 | 主要代价特征                          |
| ----------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------- | ------------------------------------- |
| Broadcast Hash Join (BHJ)                 | 小表可广播，通常满足 `size <= spark.sql.autoBroadcastJoinThreshold`，且多为等值 Join | 大表 Join 小维表 (维表关联、标签补全)    | 避免大表 shuffle，网络广播小表        |
| Sort Merge Join (SMJ)                     | 等值 Join，双方较大，或无法广播；Join Key 可排序                                     | 大表 Join 大表 (事实表互关联)            | 需要 shuffle + sort，稳定但开销较高   |
| Shuffle Hash Join (SHJ)                   | 等值 Join，无法广播，但某侧分区后能装入内存哈希表；AQE 可能动态改写到 SHJ            | 中等规模 Join，分区后单分片较小          | 需要 shuffle，但可避免全量排序        |
| Broadcast Nested Loop Join (BNLJ)         | 非等值 Join 或缺少可用 Join Key，且一侧可广播                                        | 范围 Join、小表过滤大表、非等值条件 Join | 复杂度高于哈希 Join，依赖广播侧足够小 |
| Shuffle Replicate Nested Loop Join (SRNL) | 通常由显式 Hint 触发 (如 `SHUFFLE_REPLICATE_NL`)，将一侧复制到所有分区               | 特殊场景的内连接或笛卡尔相关 Join        | 网络与计算开销很高，生产慎用          |

#### 二、核心策略的触发机制细节

##### 1. Broadcast Hash Join (BHJ)

触发要点：

1. 通常是等值 Join (`a.id = b.id`)。
2. 优化器估算某侧表足够小，低于 `spark.sql.autoBroadcastJoinThreshold` (默认常见是 10 MB，实际以集群配置为准)。
3. 可以通过 Hint 强制倾向：`/*+ BROADCAST(dim) */`。

执行机制：

1. 将小表广播到每个 Executor。
2. 在每个分区本地构建哈希表并探测大表记录。
3. 大表侧通常不需要因 Join 再次全量 shuffle (若上游分区已满足要求时收益更明显)。

适用场景：事实表 Join 小维表，是最常见、性价比最高的策略。

##### 2. Sort Merge Join (SMJ)

触发要点：

1. 等值 Join。
2. 双方都不够小，无法广播。
3. Join Key 可排序。
4. 在大表 Join 大表中通常是默认稳妥选择。

执行机制：

1. 双方按 Join Key shuffle 到同一分区。
2. 分区内各自排序。
3. 通过类似归并过程线性扫描匹配。

适用场景：海量数据等值关联，追求可扩展和稳定性。

##### 3. Shuffle Hash Join (SHJ)

触发要点：

1. 等值 Join。
2. 无法广播，但分区后某侧数据足够小，可以在任务内存中构建哈希表。
3. AQE (Adaptive Query Execution) 开启后，可能根据运行时分区统计将原计划 SMJ 动态改写为 SHJ。

执行机制：

1. 双方先 shuffle。
2. 在每个目标分区内，对较小一侧建哈希表。
3. 另一侧流式探测。

适用场景：中等规模 Join，且排序成本高于建哈希成本时效果更好。

##### 4. BNLJ 与 SRNL (补充)

1. `Broadcast Nested Loop Join` 常用于非等值 Join (如 `a.ts between b.start and b.end`) 且一侧很小可广播。
2. `Shuffle Replicate Nested Loop` 会把一侧复制到多个分区，代价高，通常只在特殊 Hint 或特定计划下使用。

#### 三、面试中必须提到的影响因素

1. 统计信息质量：`ANALYZE TABLE` 后 CBO 才能更准估算大小，否则可能选错策略。
2. AQE 动态改写：开启 `spark.sql.adaptive.enabled=true` 后，运行时可以把静态计划优化成更合适的 Join。
3. 数据倾斜：即使策略选对，热点 Key 也会拖慢任务，需要配合 `AQE skew join`、加盐或预聚合。
4. Hint 优先级：Hint 可以影响优化器，但若语义不兼容或资源不允许，最终计划仍可能退化到其他策略。

#### 四、代码示例 (含注释)

```sql
-- 1) 小维表广播，典型 BHJ
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          f.user_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;

-- 2) 大表 Join 大表，常见走 SMJ (是否最终改写取决于 AQE)
SELECT a.id, a.metric, b.tag
FROM big_a a
JOIN big_b b
   ON a.id = b.id;

-- 3) 非等值 Join，可能走 BNLJ (若小表可广播)
SELECT /*+ BROADCAST(rule_dim) */
          e.event_id,
          r.rule_id
FROM event_stream e
JOIN rule_dim r
   ON e.score >= r.min_score
 AND e.score <  r.max_score;
```

如果想在面试里证明你会落地排查，可以补一句：使用 `EXPLAIN FORMATTED` 查看最终物理计划，确认是否是 `BroadcastHashJoinExec`、`SortMergeJoinExec`、`ShuffledHashJoinExec` 或 `BroadcastNestedLoopJoinExec`。

#### 五、面试时可以怎么总结

可以这样回答：Spark SQL 选 Join 策略的核心是代价最小化。小表优先广播走 BHJ，大表对大表通常走 SMJ，中等规模且分区后可建哈希表时可能走 SHJ，非等值 Join 常见 BNLJ，SRNL 属于高代价特殊策略。实际生产中要结合统计信息、AQE 和数据倾斜治理一起看，不能只看静态 SQL 语句本身。

#### 知识扩展

- AQE (Adaptive Query Execution)：可在运行时做 Join 策略改写，与静态 CBO 形成互补。
- CBO (Cost-Based Optimizer)：Join 选择依赖表大小和基数估算，统计信息质量直接影响策略正确性。
- Skew Join 优化：Join 变慢往往不只因为策略，还因为热点 Key，需配合倾斜治理。
- Dynamic Partition Pruning (DPP)：可减少扫描分区数据量，间接影响 Join 代价与最终策略。

### 1.2 Sort Merge Join 的执行流程是怎样的？在 Spark/Presto/Hive 中分别有哪些关键步骤？

这个问题的核心不是背“SMJ 需要排序”这么一句话，而是要说明为什么它能在大表 Join 大表时保持稳定，以及不同引擎在“排序 + 归并”这条链路上各自做了什么。先给结论：Sort Merge Join (SMJ) 的本质是先让两侧数据按照 Join Key 变成有序流，再用双指针归并的方式线性扫描匹配，因此它特别适合等值 Join、超大数据集、且不适合构建内存哈希表的场景。

#### 一、SMJ 的通用执行思想

SMJ 可以拆成 4 个阶段：

1. 按 Join Key 做重分区 (shuffle/repartition)，让相同 Key 落到同一个分区。
2. 在每个分区内分别排序，保证两侧数据都按 Join Key 有序。
3. 以归并的方式同时扫描左右两侧数据，比较当前 Key 的大小关系。
4. 当 Key 相等时，输出匹配行；如果存在重复 Key，则会把一侧的同 Key 记录按组展开后与另一侧做笛卡尔式配对。

它的优势在于：不需要为整张大表构建大哈希表，内存压力更可控；缺点也很明显：shuffle 和 sort 的代价很高，如果数据已经天然有序或可以复用排序结果，SMJ 的性价比才会明显提升。

#### 二、Spark 中的关键步骤

Spark 的 SMJ 是最典型、最标准的实现，物理计划通常对应 `SortMergeJoinExec`。

1. `Catalyst` 先根据等值 Join 条件判断能否使用 SMJ。
2. 执行 `Exchange`，把左右两侧数据按 Join Key shuffle 到相同分区。
3. 在每个分区内执行 `SortExec`，分别对左右两侧排序。
4. `SortMergeJoinExec` 在分区内以流式方式做归并扫描。
5. 如果一侧 Join Key 重复，Spark 会先缓存当前 Key 的等值行，再与另一侧对应组做展开匹配。
6. 在 Whole-stage Codegen 打开时，Spark 会把比较和推进逻辑尽量代码生成，减少解释执行开销。

Spark 的一个典型特征是：即使最终走 SMJ，也不代表完全不会使用内存，排序过程本身仍然可能 spill 到磁盘，所以它更偏向“用可控的顺序 IO 和排序成本，换取对大数据量的稳定性”。

伪代码可以这样理解：

```text
left = next ordered row from left side
right = next ordered row from right side

while left and right both exist:
   if left.key < right.key:
      advance left
   else if left.key > right.key:
      advance right
   else:
      collect all rows with the same key from left group
      collect all rows with the same key from right group
      output the cross product of the two groups
      advance both sides to the next distinct key
```

#### 三、Presto/Trino 中的关键步骤

这里要先补一个面试中很重要的前提：Presto/Trino 的主流 Join 算子通常是 Hash Join (broadcast join 或 repartitioned join)，并不是像 Spark 或 Hive 那样把 SMJ 当作最核心的默认路径。所以如果面试官直接问 Presto 的 SMJ，最好先说明“Presto/Trino 更常见的是 Hash Join；如果出现排序归并类 Join，其前提是输入已经有序，或者执行计划先把输入整理成有序流”。

如果把它理解成“排序后归并”的执行思路，那么关键步骤是：

1. 调度器先生成分布式执行计划，决定数据是广播还是重分区。
2. 如果是排序归并路径，两侧输入必须先满足有序性，可能来自上游算子已经输出排序数据，或者先执行显式排序。
3. Worker 端按顺序读取两侧有序流，用游标逐行比较 Join Key。
4. 当 Key 相等时，输出结果；当 Key 不相等时，推进较小的一侧。
5. 如果运行时发现内存不足，Presto/Trino 更常见的是依赖 Hash Join spill 机制，而不是把 SMJ 作为首选兜底策略。

因此，Presto/Trino 的面试回答重点不是“它的 SMJ 和 Spark 一样怎么做”，而是要说清楚：它的默认优化方向更偏向 Hash Join，排序归并只是从算法视角理解其 Join 执行的一种补充表达。

#### 四、Hive 中的关键步骤

Hive 的 Join 实现要结合执行引擎来看，经典场景通常出现在 MapReduce 或 Tez 上。Hive 里和 SMJ 最相关的是两类情况：普通的 reduce-side sort merge join，以及更工程化的 Sort Merge Bucket Map Join (SMB MapJoin)。

1. 如果没有利用到 bucket 和 sort 属性，Hive 会在 shuffle 阶段把相同 Join Key 的数据汇聚到同一个 reducer。
2. reducer 端收到的是按 Key 排序后的数据流，然后按归并方式完成 Join。
3. 如果表已经按 Join Key 分桶并排序，Hive 可以减少甚至跳过部分 shuffle 和排序成本，直接在对应 bucket 上做局部归并。
4. 在 SMB MapJoin 场景下，如果 bucket 数和排序属性满足要求，Join 可以进一步下沉到 map 端执行，避免 reducer 成为瓶颈。

Hive 的关键点在于：它非常依赖表的物理组织方式。也就是说，Hive 中的 SMJ 不只是一个执行算子问题，更是“表是否分桶、是否排序、bucket 是否对齐”的问题。表设计做得好，SMJ 的代价会显著下降；表设计不好，即使逻辑上能 Join，执行代价也会很高。

#### 五、三者对比总结

| 引擎 | 典型定位 | SMJ 的关键链路 | 面试时要强调的点 |
| --- | --- | --- | --- |
| Spark | 经典 SMJ 的代表 | Shuffle -> Sort -> Merge | 物理计划常见为 `SortMergeJoinExec`，适合大表等值 Join |
| Presto/Trino | 主流是 Hash Join | 若出现排序归并，则必须先保证有序流，再线性归并 | 先说明它并不以 SMJ 为主，避免答偏 |
| Hive | 强依赖表布局 | Shuffle/Sort -> Reducer 归并，或利用 bucketed sorted table 做 SMB Join | 要把 bucket、sort、MapReduce/Tez 一起讲 |

#### 六、常见误区

1. SMJ 不等于“没有内存开销”，排序阶段仍可能 spill。
2. SMJ 不是“天然比 Hash Join 慢”，在大表场景下它往往更稳，因为哈希表可能更容易爆内存。
3. Hive 的 Sort Merge Bucket Join 不是普通逻辑 Join，而是建立在表的分桶和排序属性之上的物理优化。
4. Presto/Trino 不要硬说它“默认走 SMJ”，这在面试中很容易被追问细节。

#### 七、面试时可以怎么总结

可以这样回答：Sort Merge Join 的本质是先把两侧数据按 Join Key 排序，再通过双指针归并完成匹配。Spark 里最标准，流程就是 Shuffle、Sort、Merge；Hive 里要结合 MapReduce/Tez 和 bucketed sorted table 来讲，很多优化都依赖表的物理布局；Presto/Trino 则要先说明它主流并不是 SMJ，而是 Hash Join，如果谈排序归并 Join，需要强调它必须先保证输入有序，然后再做流式归并。

#### 知识扩展

- Hash Join：和 SMJ 的核心对比点是“建哈希表”还是“排序后归并”，面试里常拿来比较内存和 IO 代价。
- Bucketed Table：Hive 中是否能高效执行 Join，往往取决于分桶设计是否合理。
- Spill 机制：无论是排序还是哈希 Join，内存不足时如何落盘是判断执行引擎成熟度的关键。
- AQE (Adaptive Query Execution)：动态决定 Join 策略时，可能在运行时把计划从一种 Join 改写成另一种 Join。

### 1.3 Broadcast Hash Join 的完整执行流程是怎样的？什么时候会失效或变慢？

这是 Spark SQL 面试里的高频追问题。回答时不能只说“小表广播到大表”，而要讲清楚从优化器决策到运行时广播、建哈希表、探测匹配的完整链路，以及什么情况下 BHJ 会退化甚至失效。

先给结论：Broadcast Hash Join (BHJ) 的核心是把小表复制到每个 Executor，在任务本地构建哈希表，再用大表流式探测。它通常可以避免大表侧 shuffle，因此在“事实表 + 小维表”等值 Join 场景中非常高效；但如果小表估算失真、广播超时、内存不足或数据倾斜严重，BHJ 会明显变慢，甚至被迫回退到其他 Join 策略。

#### 一、BHJ 的完整执行流程

可以按“规划阶段 + 执行阶段”来理解。

##### 1. 规划阶段 (Catalyst/CBO/AQE)

1. 解析 Join 条件，优先判断是否为等值 Join。
2. 基于统计信息估算左右表大小，结合 `spark.sql.autoBroadcastJoinThreshold` 判断是否可广播。
3. 如果写了 Hint (如 `/*+ BROADCAST(dim) */`)，优化器会优先尝试 BHJ。
4. 生成物理计划时，通常出现 `BroadcastExchangeExec` + `BroadcastHashJoinExec`。
5. 如果开启 AQE (Adaptive Query Execution)，运行时仍可能根据真实分区统计改写策略。

##### 2. 执行阶段 (Driver/Executor)

1. Spark 先计算被广播侧数据。
2. 通过 `BroadcastExchangeExec` 将小表物化为可广播对象 (常见是哈希关系结构)。
3. Driver 把广播变量分发到各个 Executor。
4. 每个 Executor 在本地反序列化广播数据并建立哈希结构。
5. 大表分区数据在本地流式读取，逐行用 Join Key 探测哈希表。
6. 命中则输出匹配结果；未命中按 Join 类型 (inner/left 等) 执行对应语义。

它的关键收益在于：Join 阶段通常不需要对大表再次全量 shuffle，网络代价主要是“一次小表广播”而不是“大表双边重分区”。

#### 二、触发条件与适用场景

1. 最常见是等值 Join (`fact.user_id = dim.user_id`)。
2. 一侧数据量明显更小，且可放入 Executor 可用内存。
3. 维表相对稳定、字段较窄、重复复用频繁时，BHJ 性价比很高。

典型场景：订单事实表关联用户维表、商品维表、地域维表等。

#### 三、什么时候会失效

这里的“失效”指的是 BHJ 无法按预期执行，或最终没有采用 BHJ。

1. 小表估算超过广播阈值：优化器直接不选 BHJ。
2. 统计信息不准：静态计划选了 BHJ，但运行时小表远大于预期，导致广播或内存阶段出问题。
3. 广播超时：`spark.sql.broadcastTimeout` 内未完成广播，任务失败或回退。
4. 内存不足：Executor 无法容纳广播侧哈希结构，出现 OOM 或频繁 GC。
5. Join 语义/条件不匹配：复杂非等值条件下，通常不会走标准 BHJ。

#### 四、什么时候会变慢

1. 小表“勉强可广播”但过大：网络分发和反序列化成本高，收益被抵消。
2. 广播侧列很多、行很宽：对象膨胀导致内存和 GC 压力大。
3. 大表 Join Key 严重倾斜：热点 Key 让部分 task 极慢，拖慢全作业。
4. 集群网络拥塞或 Executor 数量多：广播 fan-out 成本上升。
5. 频繁重复触发广播：同一批次多次 Join 同一维表但未被有效复用，导致额外开销。

面试里可以补一句：BHJ 快，不等于“永远比 SMJ 快”。当广播侧变大或倾斜严重时，SMJ 的稳定性可能更好。

#### 五、排查与优化建议 (可落地)

1. 先看物理计划：`EXPLAIN FORMATTED` 是否出现 `BroadcastHashJoinExec`。
2. 校准统计信息：定期 `ANALYZE TABLE`，减少 CBO 误判。
3. 控制广播规模：只选择必要列，提前过滤维表，降低广播体积。
4. 合理设置参数：关注 `spark.sql.autoBroadcastJoinThreshold` 与 `spark.sql.broadcastTimeout`。
5. 处理倾斜：用 AQE skew join、加盐、预聚合等方式打散热点 Key。
6. 对比策略效果：在关键 SQL 上对比 BHJ 与 SMJ 的实际耗时和资源曲线，而不是只看理论。

#### 六、代码示例 (含注释)

```sql
-- 典型 BHJ：事实表关联小维表
-- 通过 BROADCAST Hint 显式提示优化器优先广播 dim_user
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          f.user_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;

-- 建议配合 EXPLAIN FORMATTED 检查是否真的走了 BroadcastHashJoinExec
EXPLAIN FORMATTED
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;
```

#### 七、面试时可以怎么总结

可以这样回答：Broadcast Hash Join 的完整流程是，优化器先根据统计信息和阈值判断小表是否可广播，然后在运行时通过 BroadcastExchange 把小表发到每个 Executor，本地建哈希表后由大表流式探测匹配。它在大表 Join 小维表时通常最快，因为避免了大表 shuffle。但当小表过大、统计失真、广播超时、Executor 内存不足或 Join Key 严重倾斜时，BHJ 会失效或明显变慢，生产上必须结合 EXPLAIN、统计信息和倾斜治理一起看。

#### 知识扩展

- AQE (Adaptive Query Execution)：可在运行时把静态 Join 决策改写为更合适的策略。
- CBO (Cost-Based Optimizer)：BHJ 是否被选中强依赖统计信息质量。
- Skew Join 优化：热点 Key 会放大 BHJ 的长尾问题，需要配套倾斜治理。
- Dynamic Partition Pruning (DPP)：能减少大表扫描数据量，间接提升 BHJ 效率。

### 1.4 left semi join / left anti join 在实现上和 inner join 有什么不同？引擎层面有什么优化？

这个问题的关键不在于记住 SQL 语义，而在于说明为什么 semi join 和 anti join 往往比 inner join 更省资源。它们都属于“只关心是否存在匹配关系”的 Join：left semi join 只保留左表中能够匹配到右表的行，left anti join 则只保留左表中无法匹配到右表的行，因此输出结果天然只需要左表列，不需要把左右两侧做成完整的笛卡尔式配对。

先给一个直观对比：

| Join 类型 | 是否输出右表列 | 是否需要生成左右行配对 | 关注点 |
| --- | --- | --- | --- |
| Inner Join | 是 | 是 | 匹配到的完整结果集 |
| Left Semi Join | 否 | 否，找到一个匹配即可 | 左表行是否存在右表匹配 |
| Left Anti Join | 否 | 否，找到一个匹配即可排除 | 左表行是否不存在右表匹配 |

#### 一、和 Inner Join 的本质区别

1. Inner Join 的目标是输出左右两边的连接结果，因此如果右表同一个 Join Key 有多行，左表一行可能扩展成多行结果。
2. Left Semi Join 只要求“是否存在至少一条匹配”，所以对右表重复 Key 不敏感，命中一次就足够，不需要继续展开右表后续记录。
3. Left Anti Join 则相反，它的判断标准是“是否完全没有匹配”，一旦找到任意一条匹配，就可以立即丢弃当前左表行。
4. 因为输出只依赖左表，semi/anti join 通常可以做更激进的列裁剪和数据裁剪，只保留 Join Key 和过滤条件所需列。

#### 二、执行层面的差异

以 Spark SQL 为例，left semi / anti join 仍然可以走 BHJ、SMJ、SHJ 等物理策略，但算子的行为和 inner join 不同：

1. 构建侧通常只保留用于匹配的 key 和必要过滤列，而不是整行 payload。
2. 探测侧逐行扫描左表时，只要确认“存在”或“不存在”即可，不需要产生左右行组合。
3. 对于 left semi join，探测到第一条匹配后就可以 short-circuit，避免继续扫描右表同 key 的后续行。
4. 对于 left anti join，同样可以在确认存在任意匹配后直接判定失败，减少无效比较。

这意味着：当右表存在大量重复 Key 时，inner join 可能会产生明显的数据膨胀，而 semi/anti join 只做存在性判断，代价会稳定得多。

#### 三、引擎层面的优化点

##### 1. Broadcast Semi / Anti Join

如果右表较小，优化器可以把右表广播到各个 Executor，本地只构建 key 集合或轻量哈希结构，然后用左表逐行探测：

1. left semi join：命中右表 key 就保留左表行。
2. left anti join：未命中右表 key 才保留左表行。

这类执行路径特别适合 `EXISTS` / `NOT EXISTS` 子查询改写后的场景，因为本质上就是存在性判断。

##### 2. Join Filter / Projection Pruning

semi/anti join 可以更早裁剪无关列：

1. 右表如果只是做过滤存在性判断，很多引擎只会读取 Join Key，而不会加载完整宽表列。
2. 左表如果后续列不参与输出，也会被尽量裁剪掉。
3. 这会降低网络 shuffle、反序列化和内存占用。

##### 3. Runtime Filter 和 Dynamic Partition Pruning (DPP)

在数仓和湖仓引擎里，semi join 常被当成“过滤器传播”的基础能力：

1. 先从小表提取可匹配的 key 集合。
2. 将 key 集合下推到大表扫描阶段，提前过滤掉不可能命中的分区或文件。
3. Spark 中对应的典型优化就是 DPP，很多时候它本质上就是一种半连接过滤的思想。

##### 4. Null 语义优化

left anti join 在 `NOT IN` 场景下需要特别小心 null 语义。因为只要右表中出现 null，`NOT IN` 的三值逻辑就可能让结果变得“不确定”，因此引擎通常会使用更严格的 Null-Aware Anti Join (NAAJ) 处理这类情况，避免和普通 anti join 混淆。

#### 四、典型 SQL 示例

```sql
-- 1) Left semi join：只保留在 dim_user 中存在的订单用户
SELECT f.order_id, f.user_id
FROM fact_order f
LEFT SEMI JOIN dim_user d
   ON f.user_id = d.user_id;

-- 2) Left anti join：找出在 dim_user 中不存在的脏数据用户
SELECT f.order_id, f.user_id
FROM fact_order f
LEFT ANTI JOIN dim_user d
   ON f.user_id = d.user_id;

-- 3) 等价思路：EXISTS / NOT EXISTS 通常会被优化器改写成 semi / anti join
SELECT f.order_id
FROM fact_order f
WHERE EXISTS (
   SELECT 1
   FROM dim_user d
   WHERE f.user_id = d.user_id
);
```

#### 五、常见误区

1. Semi join 不是 inner join 的语法糖，二者的结果集语义不同，尤其是右表重复行时差异非常明显。
2. Anti join 不是“inner join 再取反”，它需要严格处理 null 语义，特别是 `NOT IN` 和 `NOT EXISTS` 不完全等价。
3. Semi/anti join 并不一定更快，如果右表很大或者无法利用广播与过滤下推，仍然会出现 shuffle 和 sort 开销。

#### 六、面试时可以怎么总结

可以这样回答：left semi join 和 left anti join 和 inner join 的核心区别在于，它们只关心右表是否存在匹配，而不关心右表的实际输出，因此不需要生成左右表的完整配对结果。实现上，semi/anti join 通常可以在探测到第一条匹配后短路，避免重复展开右表数据；引擎层面则会结合广播、小表 key 集合、投影裁剪、runtime filter、DPP 和 Null-Aware Anti Join 等优化，把“存在性判断”尽量前移到扫描和过滤阶段，从而显著降低网络、内存和计算开销。

#### 知识扩展

- EXISTS / NOT EXISTS：通常是 semi / anti join 的 SQL 等价表达，优化器经常会把子查询改写成对应物理 Join。
- Dynamic Partition Pruning (DPP)：利用小表 key 过滤大表分区，本质上就是半连接过滤思想的落地。
- Runtime Filter：在 Flink、Trino、Spark 等引擎中都常见，和 semi join 一样强调“先过滤，再 Join”。
- Null 语义：`NOT IN`、`NOT EXISTS`、anti join 在 null 处理上的差异是面试中很容易追问的细节。

### 1.5 在什么情况下会选择 Nested Loop Join（或 Broadcast Nested Loop Join）？为什么？

这是面试中常被追问的一个工程化问题。高质量回答不仅要说明“什么时候用”，还要解释“为什么用”以及引擎如何把这种看似暴力的算法做得尽量可用。Nested Loop Join (NLJ) 的核心思想很直白：对左右两侧逐行做嵌套循环并应用 Join 条件，复杂度在最坏情况下接近 O(N*M)。因此它通常不是首选，但在若干场景下是合理或必要的。

#### 一、适用场景（何时选择 NLJ / BNLJ）

1. 非等值 Join 或复杂表达式：当 Join 条件不是等值匹配（例如范围 Join、近似匹配、复杂 UDF 判定、模糊匹配等），哈希或排序归并等等值优化无法直接适用，NLJ 通常是可行的实现路径。
2. 右表/任意一侧非常小（可广播）：如果一侧足够小，可以广播到每个 Executor（或每个 worker），则 Broadcast Nested Loop Join (BNLJ) 的实际成本接近扫描大表一次并在内存中用小表做快速探测，代价可接受。
3. 强过滤或高选择性条件：当嵌套循环中能很快通过谓词淘汰大部分组合（例如左表先做强过滤，或右表通过索引/布隆过滤快速判定否定），NLJ 的实际工作量会远小于理论复杂度。
4. 非常规 Join（如笛卡尔积后筛选）：当 SQL 显式要求笛卡尔或等价复杂逻辑，NLJ 能直接表达这类语义（尽管代价高，要谨慎使用）。

#### 二、为什么在这些场景下选择 NLJ？

1. 算法通用性：NLJ 不依赖于 Key 的可哈希或可排序性质，能够执行任意可计算的 Join 条件。
2. 广播后成本可控：BNLJ 在小表广播时，右表在内存可随机访问，左表只需单次扫描并对每行探测小表，IO 与网络成本低于对两表做大规模 shuffle。
3. 早期短路与过滤：BNLJ/Block NLJ 能在满足条件后短路（short-circuit）或按块（block）批量判断，显著减小实际比较次数。

#### 三、引擎层面的实现与优化手段

虽然 NLJ 算法看似朴素，但各个引擎做了许多优化来控制其实际成本：

1. Broadcast + Build Small Key Structure
   - 广播时只序列化并广播必要的 Join Key 或经简化的探测结构（例如哈希集合、位图或布隆过滤器），避免传输宽列。
   - 在 Executor 本地用紧凑的数据结构（hash set / bitmap）来快速做存在性判断。

2. Block Nested Loop（块式嵌套循环）
   - 将外层若干行打包成一个块，对内层做批量探测，减少函数调用和 IO 次数，并提高缓存命中率。

3. Bloom Filter / Runtime Filter
   - 在广播或构建阶段同时生成 Bloom filter，把可能不存在的外部行提前过滤掉，从而降低后续比较次数。

4. 短路评估与谓词下推
   - 在匹配到满足条件的内表记录后立即短路（对于 semi join/exists 场景更明显）。
   - 尽量把可单行判断的过滤条件下推到扫描阶段，减少进入 NLJ 的候选行数。

5. 向量化和 CodeGen
   - 对内循环做向量化比较（批量比较多个 key），或利用 whole-stage codegen 减少解释开销。

6. Null-aware / Null-safe 处理
   - 在实现 anti/NOT IN 语义时，使用 Null-Aware 反连接逻辑避免三值逻辑错误。

7. 引擎级 Hint 与改写
   - 优化器可能提供 Hint（或基于成本模型）将非等值子查询改写为 BNLJ（如果小表可广播）或保留为嵌套循环执行。

8. 利用索引（在支持索引的存储引擎）
   - 如果内表（被循环的一侧）支持索引，可把内层查找转成索引查找，复杂度从线性降为对数级或常数级，极大加速 NLJ。

#### 四、典型 SQL 示例（示意）

```sql
-- 1) 非等值范围 Join（NLJ 是合理选择）
SELECT a.id, b.label
FROM events a
JOIN windows b
   ON a.ts >= b.start_ts AND a.ts < b.end_ts;

-- 2) 小表广播后的存在性判断（BNLJ）
SELECT f.order_id
FROM fact_order f
WHERE EXISTS (
   SELECT 1 FROM small_ref s WHERE some_complex_udf(f.col, s.col)
);
-- 优化器可将上述 EXISTS 改写为 BNLJ（广播 small_ref）并利用构建侧的紧凑探测结构
```

#### 五、何时避免使用 Nested Loop Join

1. 两表都很大且无可广播一侧：NLJ 会退化为 O(N*M)，不可接受。
2. 广播侧非但不小且宽列很多：广播成本和反序列化成本会让 BNLJ 变慢或 OOM。
3. 可用等值 Join（Hash）或 SMJ 更优：若条件允许，优先选择 BHJ/SHJ/SMJ。

#### 六、面试时可以怎么总结

可以这样回答：Nested Loop Join（及其广播变体 BNLJ）适用于非等值或复杂 Join 条件、或一侧非常小可以广播的场景。它的优势是通用性强、实现简单；劣势是最坏复杂度高。各引擎通过广播小表、块式批量处理、布隆过滤器、索引查找、向量化和 codegen 等手段，把 NLJ 的实际成本降到可接受范围，从而在特定场景成为最合适的执行策略。

#### 知识扩展

- Block Nested Loop：块式处理以提高缓存和 IO 利用率。
- Bloom Filter / Runtime Filter：用于把无效候选提前剔除，常与 NLJ/Broadcast 配合。
- Index Nested Loop Join (INLJ)：在支持索引的存储引擎中，把内层查找转为索引查找，极大加速 NLJ。
- AQE 与计划改写：运行时统计可能把原本计划改写为更适合的策略（例如当发现可广播小表时把某些路径改成 BNLJ）。

### 1.6 为什么说 Shuffle 通常是作业的性能瓶颈？你会从哪些点入手排查和调优？

Shuffle（重分区/洗牌）通常是 Spark 作业里最容易成为瓶颈的环节之一。面试中如果只说“因为要跨节点传数据”还不够，最好能进一步说明它为什么贵、贵在哪里，以及如何系统地排查和优化。Shuffle 之所以常常成为瓶颈，本质上是因为它同时放大了网络、磁盘、CPU、内存和调度五类成本。

#### 一、为什么 Shuffle 容易成为瓶颈

1. 网络传输量大：Shuffle 需要把 map 端的数据发送到目标分区，跨节点网络 IO 往往是最直接的瓶颈。
2. 序列化与反序列化开销高：数据在发送前要序列化，接收后要反序列化，CPU 开销明显。
3. 磁盘 spill 频繁：内存不够时，shuffle 数据会落盘，磁盘读写和排序成本会迅速上升。
4. 文件和连接数多：Shuffle 会产生大量中间文件和网络连接，小文件过多还会拖慢元数据处理。
5. 数据倾斜放大长尾：某些热点 key 可能让个别 partition 特别大，导致少数 task 跑得极慢，拖住整个 stage。
6. GC 压力增大：中间对象、缓冲区和反序列化对象会提升 GC 频率，进一步影响吞吐和稳定性。

#### 二、如何排查 Shuffle 问题

1. 看执行计划：用 `EXPLAIN FORMATTED` 或 `EXPLAIN COST` 确认是否存在 `Exchange`，以及哪一步引入了 shuffle。
2. 看 Spark UI：重点看 Stage 的 `Shuffle Read`、`Shuffle Write`、Task Duration、Spill（内存/磁盘）和失败重试情况。
3. 看长尾 task：如果少数 task 明显慢于其他 task，优先怀疑数据倾斜或单分区数据过大。
4. 看 Executor 指标：观察网络带宽、磁盘 IO、CPU、GC、Heap 使用率，判断瓶颈主要在网络还是存储或内存。
5. 看日志：关注 OOM、shuffle fetch failed、fetch timeout、spill 过多、序列化异常等关键错误信息。
6. 看 AQE 结果：如果开启了 `spark.sql.adaptive.enabled=true`，需要确认是否发生了分区合并、倾斜拆分或 Join 策略改写。

#### 三、如何调优 Shuffle

调优时通常遵循一个原则：先减少 Shuffle 的数据量，再降低 Shuffle 的单次成本，最后处理倾斜和资源瓶颈。

##### 1. 从 SQL / 算法层减少 Shuffle

1. 优先使用 Broadcast Join：小表能广播时，避免大表双边 shuffle。
2. 早过滤、少传输：提前做过滤和列裁剪，只保留参与计算的字段。
3. 预聚合：在 map 端先做局部聚合，再汇总到 reduce 端，减少传输的数据量。
4. 避免无谓的 repartition：不要频繁对中间结果做重复重分区。

##### 2. 从分区与并行度入手

1. 调整 `spark.sql.shuffle.partitions`：让并行度与数据量匹配，避免分区过少导致单 task 太大，或分区过多导致任务调度开销过高。
2. 合理使用 `repartition()` / `coalesce()`：在输出或中间阶段控制分区数量。
3. 让 key 分布更均匀：必要时通过重新设计分区键来避免热点分区。

##### 3. 从序列化与压缩入手

1. 使用 Kryo 序列化：`spark.serializer=org.apache.spark.serializer.KryoSerializer`，并注册常用类型。
2. 开启 shuffle 压缩：降低网络传输量，但要权衡 CPU 开销。
3. 选择合适的压缩编码：例如 `lz4` 在吞吐和 CPU 之间通常较平衡。

##### 4. 从内存和 spill 入手

1. 给 shuffle 留足执行内存：避免频繁 spill 到磁盘。
2. 观察 spill 指标：如果 spill 很多，说明内存配置或数据组织方式有问题。
3. 控制单个 task 的数据量：通过更合理的分区和预聚合降低单 task 压力。

##### 5. 从数据倾斜入手

1. 加盐 (salting)：对热点 key 进行打散，降低单分区压力。
2. AQE 倾斜优化：开启 `spark.sql.adaptive.skewJoin.enabled=true`，让 Spark 在运行时拆分倾斜分区。
3. 提前拆热点：对特别大的 key 单独处理，再和主路径结果合并。

##### 6. 从系统与网络入手

1. 检查磁盘性能：本地盘太慢会直接放大 shuffle spill 成本。
2. 检查网络带宽和丢包：Shuffle 对网络稳定性很敏感。
3. 合理配置 shuffle 相关 buffer：例如减少频繁小块传输带来的开销。

#### 四、一个简单的排查思路

可以按下面顺序快速定位：

1. 先看是不是 shuffle 很多：`EXPLAIN` 里是否有多个 `Exchange`。
2. 再看是不是某些 task 特别慢：如果是，优先查倾斜。
3. 再看是不是 spill 很严重：如果是，优先查内存和分区大小。
4. 再看是不是网络慢：如果 `Shuffle Read` 很高但 CPU 不高，可能是网络或磁盘问题。
5. 最后再做参数优化：在确认瓶颈类型后，再调整分区数、压缩、序列化器和 AQE。

#### 五、常见误区

1. 误区一：分区越多越好。过多分区会带来任务调度成本和小文件问题。
2. 误区二：只调一个参数就能解决问题。Shuffle 瓶颈往往是综合问题，通常要结合 SQL、分区、内存和数据倾斜一起看。
3. 误区三：压缩一定会变慢。压缩会增加 CPU，但在网络受限场景往往是划算的。

#### 六、面试时可以怎么总结

可以这样回答：Shuffle 之所以常常成为性能瓶颈，是因为它会同时引入跨节点网络传输、序列化反序列化、磁盘 spill、GC 压力和数据倾斜长尾等成本。排查时要先看执行计划和 Spark UI，再结合 Executor 指标、日志和 AQE 结果定位问题来源；调优时则优先从减少 shuffle 数据量、控制分区和并行度、优化序列化与压缩、处理数据倾斜以及改善磁盘和网络环境入手。

#### 知识扩展

- AQE (Adaptive Query Execution)：运行时自动合并分区、拆分倾斜分区，并可能改写 Join 策略。
- Broadcast Join：很多时候本质上是在减少 Shuffle，而不是单纯优化 Join 算法。
- 数据倾斜治理：加盐、拆热点、预聚合、改 key 设计，都是围绕 Shuffle 长尾问题展开的。
- 文件治理：小文件合并和分区设计会直接影响后续 Shuffle 和下游读取效率。
