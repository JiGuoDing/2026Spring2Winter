## 1. Spark 面试题

### 1.1 Spark SQL 有哪几种 Join 策略？各自的触发条件与适用场景是什么？

这个问题在流处理和离线一体化面试中很常见。高质量回答的关键不是只背策略名，而是说明 Spark Optimizer (Catalyst + CBO + AQE) 是如何在代价和运行时统计信息基础上选择 Join 物理执行计划的。

先给一个总览：Spark SQL 常见 Join 策略包括 `Broadcast Hash Join (BHJ)`、`Sort Merge Join (SMJ)`、`Shuffle Hash Join (SHJ)`、`Broadcast Nested Loop Join (BNLJ)`、`Shuffle Replicate Nested Loop Join (SRNL)`。其中前三个最核心，后两个主要用于非等值 Join 或笛卡尔场景。

#### 一、Spark SQL Join 策略总览

| Join 策略                                 | 典型触发条件                                                                         | 适用场景                                 | 主要代价特征                          |
| ----------------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------- | ------------------------------------- |
| Broadcast Hash Join (BHJ)                 | 小表可广播，通常满足 `size <= spark.sql.autoBroadcastJoinThreshold`，且多为等值 Join | 大表 Join 小维表 (维表关联、标签补全)    | 避免大表 shuffle，网络广播小表        |
| Sort Merge Join (SMJ)                     | 等值 Join，双方较大，或无法广播；Join Key 可排序                                     | 大表 Join 大表 (事实表互关联)            | 需要 shuffle + sort，稳定但开销较高   |
| Shuffle Hash Join (SHJ)                   | 等值 Join，无法广播，但某侧分区后能装入内存哈希表；AQE 可能动态改写到 SHJ            | 中等规模 Join，分区后单分片较小          | 需要 shuffle，但可避免全量排序        |
| Broadcast Nested Loop Join (BNLJ)         | 非等值 Join 或缺少可用 Join Key，且一侧可广播                                        | 范围 Join、小表过滤大表、非等值条件 Join | 复杂度高于哈希 Join，依赖广播侧足够小 |
| Shuffle Replicate Nested Loop Join (SRNL) | 通常由显式 Hint 触发 (如 `SHUFFLE_REPLICATE_NL`)，将一侧复制到所有分区               | 特殊场景的内连接或笛卡尔相关 Join        | 网络与计算开销很高，生产慎用          |

#### 二、核心策略的触发机制细节

##### 1. Broadcast Hash Join (BHJ)

触发要点：

1. 通常是等值 Join (`a.id = b.id`)。
2. 优化器估算某侧表足够小，低于 `spark.sql.autoBroadcastJoinThreshold` (默认常见是 10 MB，实际以集群配置为准)。
3. 可以通过 Hint 强制倾向：`/*+ BROADCAST(dim) */`。

执行机制：

1. 将小表广播到每个 Executor。
2. 在每个分区本地构建哈希表并探测大表记录。
3. 大表侧通常不需要因 Join 再次全量 shuffle (若上游分区已满足要求时收益更明显)。

适用场景：事实表 Join 小维表，是最常见、性价比最高的策略。

##### 2. Sort Merge Join (SMJ)

触发要点：

1. 等值 Join。
2. 双方都不够小，无法广播。
3. Join Key 可排序。
4. 在大表 Join 大表中通常是默认稳妥选择。

执行机制：

1. 双方按 Join Key shuffle 到同一分区。
2. 分区内各自排序。
3. 通过类似归并过程线性扫描匹配。

适用场景：海量数据等值关联，追求可扩展和稳定性。

##### 3. Shuffle Hash Join (SHJ)

触发要点：

1. 等值 Join。
2. 无法广播，但分区后某侧数据足够小，可以在任务内存中构建哈希表。
3. AQE (Adaptive Query Execution) 开启后，可能根据运行时分区统计将原计划 SMJ 动态改写为 SHJ。

执行机制：

1. 双方先 shuffle。
2. 在每个目标分区内，对较小一侧建哈希表。
3. 另一侧流式探测。

适用场景：中等规模 Join，且排序成本高于建哈希成本时效果更好。

##### 4. BNLJ 与 SRNL (补充)

1. `Broadcast Nested Loop Join` 常用于非等值 Join (如 `a.ts between b.start and b.end`) 且一侧很小可广播。
2. `Shuffle Replicate Nested Loop` 会把一侧复制到多个分区，代价高，通常只在特殊 Hint 或特定计划下使用。

#### 三、面试中必须提到的影响因素

1. 统计信息质量：`ANALYZE TABLE` 后 CBO 才能更准估算大小，否则可能选错策略。
2. AQE 动态改写：开启 `spark.sql.adaptive.enabled=true` 后，运行时可以把静态计划优化成更合适的 Join。
3. 数据倾斜：即使策略选对，热点 Key 也会拖慢任务，需要配合 `AQE skew join`、加盐或预聚合。
4. Hint 优先级：Hint 可以影响优化器，但若语义不兼容或资源不允许，最终计划仍可能退化到其他策略。

#### 四、代码示例 (含注释)

```sql
-- 1) 小维表广播，典型 BHJ
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          f.user_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;

-- 2) 大表 Join 大表，常见走 SMJ (是否最终改写取决于 AQE)
SELECT a.id, a.metric, b.tag
FROM big_a a
JOIN big_b b
   ON a.id = b.id;

-- 3) 非等值 Join，可能走 BNLJ (若小表可广播)
SELECT /*+ BROADCAST(rule_dim) */
          e.event_id,
          r.rule_id
FROM event_stream e
JOIN rule_dim r
   ON e.score >= r.min_score
 AND e.score <  r.max_score;
```

如果想在面试里证明你会落地排查，可以补一句：使用 `EXPLAIN FORMATTED` 查看最终物理计划，确认是否是 `BroadcastHashJoinExec`、`SortMergeJoinExec`、`ShuffledHashJoinExec` 或 `BroadcastNestedLoopJoinExec`。

#### 五、面试时可以怎么总结

可以这样回答：Spark SQL 选 Join 策略的核心是代价最小化。小表优先广播走 BHJ，大表对大表通常走 SMJ，中等规模且分区后可建哈希表时可能走 SHJ，非等值 Join 常见 BNLJ，SRNL 属于高代价特殊策略。实际生产中要结合统计信息、AQE 和数据倾斜治理一起看，不能只看静态 SQL 语句本身。

#### 知识扩展

- AQE (Adaptive Query Execution)：可在运行时做 Join 策略改写，与静态 CBO 形成互补。
- CBO (Cost-Based Optimizer)：Join 选择依赖表大小和基数估算，统计信息质量直接影响策略正确性。
- Skew Join 优化：Join 变慢往往不只因为策略，还因为热点 Key，需配合倾斜治理。
- Dynamic Partition Pruning (DPP)：可减少扫描分区数据量，间接影响 Join 代价与最终策略。

### 1.2 Sort Merge Join 的执行流程是怎样的？在 Spark/Presto/Hive 中分别有哪些关键步骤？

这个问题的核心不是背“SMJ 需要排序”这么一句话，而是要说明为什么它能在大表 Join 大表时保持稳定，以及不同引擎在“排序 + 归并”这条链路上各自做了什么。先给结论：Sort Merge Join (SMJ) 的本质是先让两侧数据按照 Join Key 变成有序流，再用双指针归并的方式线性扫描匹配，因此它特别适合等值 Join、超大数据集、且不适合构建内存哈希表的场景。

#### 一、SMJ 的通用执行思想

SMJ 可以拆成 4 个阶段：

1. 按 Join Key 做重分区 (shuffle/repartition)，让相同 Key 落到同一个分区。
2. 在每个分区内分别排序，保证两侧数据都按 Join Key 有序。
3. 以归并的方式同时扫描左右两侧数据，比较当前 Key 的大小关系。
4. 当 Key 相等时，输出匹配行；如果存在重复 Key，则会把一侧的同 Key 记录按组展开后与另一侧做笛卡尔式配对。

它的优势在于：不需要为整张大表构建大哈希表，内存压力更可控；缺点也很明显：shuffle 和 sort 的代价很高，如果数据已经天然有序或可以复用排序结果，SMJ 的性价比才会明显提升。

#### 二、Spark 中的关键步骤

Spark 的 SMJ 是最典型、最标准的实现，物理计划通常对应 `SortMergeJoinExec`。

1. `Catalyst` 先根据等值 Join 条件判断能否使用 SMJ。
2. 执行 `Exchange`，把左右两侧数据按 Join Key shuffle 到相同分区。
3. 在每个分区内执行 `SortExec`，分别对左右两侧排序。
4. `SortMergeJoinExec` 在分区内以流式方式做归并扫描。
5. 如果一侧 Join Key 重复，Spark 会先缓存当前 Key 的等值行，再与另一侧对应组做展开匹配。
6. 在 Whole-stage Codegen 打开时，Spark 会把比较和推进逻辑尽量代码生成，减少解释执行开销。

Spark 的一个典型特征是：即使最终走 SMJ，也不代表完全不会使用内存，排序过程本身仍然可能 spill 到磁盘，所以它更偏向“用可控的顺序 IO 和排序成本，换取对大数据量的稳定性”。

伪代码可以这样理解：

```text
left = next ordered row from left side
right = next ordered row from right side

while left and right both exist:
   if left.key < right.key:
      advance left
   else if left.key > right.key:
      advance right
   else:
      collect all rows with the same key from left group
      collect all rows with the same key from right group
      output the cross product of the two groups
      advance both sides to the next distinct key
```

#### 三、Presto/Trino 中的关键步骤

这里要先补一个面试中很重要的前提：Presto/Trino 的主流 Join 算子通常是 Hash Join (broadcast join 或 repartitioned join)，并不是像 Spark 或 Hive 那样把 SMJ 当作最核心的默认路径。所以如果面试官直接问 Presto 的 SMJ，最好先说明“Presto/Trino 更常见的是 Hash Join；如果出现排序归并类 Join，其前提是输入已经有序，或者执行计划先把输入整理成有序流”。

如果把它理解成“排序后归并”的执行思路，那么关键步骤是：

1. 调度器先生成分布式执行计划，决定数据是广播还是重分区。
2. 如果是排序归并路径，两侧输入必须先满足有序性，可能来自上游算子已经输出排序数据，或者先执行显式排序。
3. Worker 端按顺序读取两侧有序流，用游标逐行比较 Join Key。
4. 当 Key 相等时，输出结果；当 Key 不相等时，推进较小的一侧。
5. 如果运行时发现内存不足，Presto/Trino 更常见的是依赖 Hash Join spill 机制，而不是把 SMJ 作为首选兜底策略。

因此，Presto/Trino 的面试回答重点不是“它的 SMJ 和 Spark 一样怎么做”，而是要说清楚：它的默认优化方向更偏向 Hash Join，排序归并只是从算法视角理解其 Join 执行的一种补充表达。

#### 四、Hive 中的关键步骤

Hive 的 Join 实现要结合执行引擎来看，经典场景通常出现在 MapReduce 或 Tez 上。Hive 里和 SMJ 最相关的是两类情况：普通的 reduce-side sort merge join，以及更工程化的 Sort Merge Bucket Map Join (SMB MapJoin)。

1. 如果没有利用到 bucket 和 sort 属性，Hive 会在 shuffle 阶段把相同 Join Key 的数据汇聚到同一个 reducer。
2. reducer 端收到的是按 Key 排序后的数据流，然后按归并方式完成 Join。
3. 如果表已经按 Join Key 分桶并排序，Hive 可以减少甚至跳过部分 shuffle 和排序成本，直接在对应 bucket 上做局部归并。
4. 在 SMB MapJoin 场景下，如果 bucket 数和排序属性满足要求，Join 可以进一步下沉到 map 端执行，避免 reducer 成为瓶颈。

Hive 的关键点在于：它非常依赖表的物理组织方式。也就是说，Hive 中的 SMJ 不只是一个执行算子问题，更是“表是否分桶、是否排序、bucket 是否对齐”的问题。表设计做得好，SMJ 的代价会显著下降；表设计不好，即使逻辑上能 Join，执行代价也会很高。

#### 五、三者对比总结

| 引擎 | 典型定位 | SMJ 的关键链路 | 面试时要强调的点 |
| --- | --- | --- | --- |
| Spark | 经典 SMJ 的代表 | Shuffle -> Sort -> Merge | 物理计划常见为 `SortMergeJoinExec`，适合大表等值 Join |
| Presto/Trino | 主流是 Hash Join | 若出现排序归并，则必须先保证有序流，再线性归并 | 先说明它并不以 SMJ 为主，避免答偏 |
| Hive | 强依赖表布局 | Shuffle/Sort -> Reducer 归并，或利用 bucketed sorted table 做 SMB Join | 要把 bucket、sort、MapReduce/Tez 一起讲 |

#### 六、常见误区

1. SMJ 不等于“没有内存开销”，排序阶段仍可能 spill。
2. SMJ 不是“天然比 Hash Join 慢”，在大表场景下它往往更稳，因为哈希表可能更容易爆内存。
3. Hive 的 Sort Merge Bucket Join 不是普通逻辑 Join，而是建立在表的分桶和排序属性之上的物理优化。
4. Presto/Trino 不要硬说它“默认走 SMJ”，这在面试中很容易被追问细节。

#### 七、面试时可以怎么总结

可以这样回答：Sort Merge Join 的本质是先把两侧数据按 Join Key 排序，再通过双指针归并完成匹配。Spark 里最标准，流程就是 Shuffle、Sort、Merge；Hive 里要结合 MapReduce/Tez 和 bucketed sorted table 来讲，很多优化都依赖表的物理布局；Presto/Trino 则要先说明它主流并不是 SMJ，而是 Hash Join，如果谈排序归并 Join，需要强调它必须先保证输入有序，然后再做流式归并。

#### 知识扩展

- Hash Join：和 SMJ 的核心对比点是“建哈希表”还是“排序后归并”，面试里常拿来比较内存和 IO 代价。
- Bucketed Table：Hive 中是否能高效执行 Join，往往取决于分桶设计是否合理。
- Spill 机制：无论是排序还是哈希 Join，内存不足时如何落盘是判断执行引擎成熟度的关键。
- AQE (Adaptive Query Execution)：动态决定 Join 策略时，可能在运行时把计划从一种 Join 改写成另一种 Join。

### 1.3 Broadcast Hash Join 的完整执行流程是怎样的？什么时候会失效或变慢？

这是 Spark SQL 面试里的高频追问题。回答时不能只说“小表广播到大表”，而要讲清楚从优化器决策到运行时广播、建哈希表、探测匹配的完整链路，以及什么情况下 BHJ 会退化甚至失效。

先给结论：Broadcast Hash Join (BHJ) 的核心是把小表复制到每个 Executor，在任务本地构建哈希表，再用大表流式探测。它通常可以避免大表侧 shuffle，因此在“事实表 + 小维表”等值 Join 场景中非常高效；但如果小表估算失真、广播超时、内存不足或数据倾斜严重，BHJ 会明显变慢，甚至被迫回退到其他 Join 策略。

#### 一、BHJ 的完整执行流程

可以按“规划阶段 + 执行阶段”来理解。

##### 1. 规划阶段 (Catalyst/CBO/AQE)

1. 解析 Join 条件，优先判断是否为等值 Join。
2. 基于统计信息估算左右表大小，结合 `spark.sql.autoBroadcastJoinThreshold` 判断是否可广播。
3. 如果写了 Hint (如 `/*+ BROADCAST(dim) */`)，优化器会优先尝试 BHJ。
4. 生成物理计划时，通常出现 `BroadcastExchangeExec` + `BroadcastHashJoinExec`。
5. 如果开启 AQE (Adaptive Query Execution)，运行时仍可能根据真实分区统计改写策略。

##### 2. 执行阶段 (Driver/Executor)

1. Spark 先计算被广播侧数据。
2. 通过 `BroadcastExchangeExec` 将小表物化为可广播对象 (常见是哈希关系结构)。
3. Driver 把广播变量分发到各个 Executor。
4. 每个 Executor 在本地反序列化广播数据并建立哈希结构。
5. 大表分区数据在本地流式读取，逐行用 Join Key 探测哈希表。
6. 命中则输出匹配结果；未命中按 Join 类型 (inner/left 等) 执行对应语义。

它的关键收益在于：Join 阶段通常不需要对大表再次全量 shuffle，网络代价主要是“一次小表广播”而不是“大表双边重分区”。

#### 二、触发条件与适用场景

1. 最常见是等值 Join (`fact.user_id = dim.user_id`)。
2. 一侧数据量明显更小，且可放入 Executor 可用内存。
3. 维表相对稳定、字段较窄、重复复用频繁时，BHJ 性价比很高。

典型场景：订单事实表关联用户维表、商品维表、地域维表等。

#### 三、什么时候会失效

这里的“失效”指的是 BHJ 无法按预期执行，或最终没有采用 BHJ。

1. 小表估算超过广播阈值：优化器直接不选 BHJ。
2. 统计信息不准：静态计划选了 BHJ，但运行时小表远大于预期，导致广播或内存阶段出问题。
3. 广播超时：`spark.sql.broadcastTimeout` 内未完成广播，任务失败或回退。
4. 内存不足：Executor 无法容纳广播侧哈希结构，出现 OOM 或频繁 GC。
5. Join 语义/条件不匹配：复杂非等值条件下，通常不会走标准 BHJ。

#### 四、什么时候会变慢

1. 小表“勉强可广播”但过大：网络分发和反序列化成本高，收益被抵消。
2. 广播侧列很多、行很宽：对象膨胀导致内存和 GC 压力大。
3. 大表 Join Key 严重倾斜：热点 Key 让部分 task 极慢，拖慢全作业。
4. 集群网络拥塞或 Executor 数量多：广播 fan-out 成本上升。
5. 频繁重复触发广播：同一批次多次 Join 同一维表但未被有效复用，导致额外开销。

面试里可以补一句：BHJ 快，不等于“永远比 SMJ 快”。当广播侧变大或倾斜严重时，SMJ 的稳定性可能更好。

#### 五、排查与优化建议 (可落地)

1. 先看物理计划：`EXPLAIN FORMATTED` 是否出现 `BroadcastHashJoinExec`。
2. 校准统计信息：定期 `ANALYZE TABLE`，减少 CBO 误判。
3. 控制广播规模：只选择必要列，提前过滤维表，降低广播体积。
4. 合理设置参数：关注 `spark.sql.autoBroadcastJoinThreshold` 与 `spark.sql.broadcastTimeout`。
5. 处理倾斜：用 AQE skew join、加盐、预聚合等方式打散热点 Key。
6. 对比策略效果：在关键 SQL 上对比 BHJ 与 SMJ 的实际耗时和资源曲线，而不是只看理论。

#### 六、代码示例 (含注释)

```sql
-- 典型 BHJ：事实表关联小维表
-- 通过 BROADCAST Hint 显式提示优化器优先广播 dim_user
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          f.user_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;

-- 建议配合 EXPLAIN FORMATTED 检查是否真的走了 BroadcastHashJoinExec
EXPLAIN FORMATTED
SELECT /*+ BROADCAST(dim_user) */
          f.order_id,
          d.user_level
FROM fact_order f
JOIN dim_user d
   ON f.user_id = d.user_id;
```

#### 七、面试时可以怎么总结

可以这样回答：Broadcast Hash Join 的完整流程是，优化器先根据统计信息和阈值判断小表是否可广播，然后在运行时通过 BroadcastExchange 把小表发到每个 Executor，本地建哈希表后由大表流式探测匹配。它在大表 Join 小维表时通常最快，因为避免了大表 shuffle。但当小表过大、统计失真、广播超时、Executor 内存不足或 Join Key 严重倾斜时，BHJ 会失效或明显变慢，生产上必须结合 EXPLAIN、统计信息和倾斜治理一起看。

#### 知识扩展

- AQE (Adaptive Query Execution)：可在运行时把静态 Join 决策改写为更合适的策略。
- CBO (Cost-Based Optimizer)：BHJ 是否被选中强依赖统计信息质量。
- Skew Join 优化：热点 Key 会放大 BHJ 的长尾问题，需要配套倾斜治理。
- Dynamic Partition Pruning (DPP)：能减少大表扫描数据量，间接提升 BHJ 效率。
