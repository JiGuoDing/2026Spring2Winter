## 1. 操作系统面试题

### 1.1 操作系统中的进程有哪几种状态

经典的进程状态模型是三态和五态，面试中最常用的是五态模型。

- 新建 (New): 进程刚被创建，PCB (Process Control Block) 已建立，但还未进入就绪队列
- 就绪 (Ready): 进程具备运行条件，只差 CPU 时间片
- 运行 (Running): 正在 CPU 上执行
- 阻塞 (Blocked/Waiting): 等待某个事件 (I/O 完成、锁释放、信号到达)
- 终止 (Terminated): 进程执行结束，等待系统回收资源

很多系统会细化为七态，在五态基础上增加：

- 就绪挂起 (Ready Suspended): 可运行但被换出到外存
- 阻塞挂起 (Blocked Suspended): 阻塞且被换出到外存

状态迁移核心逻辑：

- Running -> Blocked: 主动等待资源
- Running -> Ready: 时间片用完或被更高优先级任务抢占
- Blocked -> Ready: 等待事件到达
- Ready -> Running: 被调度器选中

知识扩展：
状态迁移与调度算法强相关。例如抢占式调度会显著增加 Running -> Ready 的切换频率，这会影响上下文切换成本与缓存命中率。

### 1.2 线程和进程有什么区别

核心结论：进程是资源分配单位，线程是 CPU 调度单位。

关键区别：

- 地址空间: 进程间隔离，线程共享所属进程地址空间
- 资源拥有: 进程拥有独立资源 (虚拟内存、文件描述符表等)，线程主要共享这些资源
- 切换开销: 进程切换通常比线程切换更重，因为涉及地址空间切换和 TLB (Translation Lookaside Buffer，用于加速虚拟地址到物理地址转换的缓存) 影响
- 通信方式: 进程常用 IPC，线程可直接读写共享内存 (但要同步)
- 故障影响: 一个进程崩溃通常不直接影响其他进程；线程崩溃可能导致整个进程异常

工程视角：

- 需要强隔离和稳定性时优先多进程
- 需要低延迟共享数据时优先多线程

知识扩展：
用户态线程和内核线程模型 (1:1、M:1、M:N) 会影响并发能力和阻塞行为，是理解 Go 协程调度器的基础。

### 1.3 什么是用户态和内核态

用户态和内核态是 CPU 权限级别的划分。

- 用户态 (User Mode): 权限受限，不能直接执行特权指令，不能直接访问硬件
- 内核态 (Kernel Mode): 拥有最高权限，可访问硬件、管理内存、调度进程

为什么要分态：

- 安全隔离: 防止应用直接破坏系统
- 稳定性: 应用故障不应直接破坏内核核心数据结构
- 受控访问: 通过系统调用统一访问内核服务

典型切换路径：

1. 用户程序发起系统调用 (如 read, write)
2. 触发陷入 (trap) 进入内核态
3. 内核执行服务逻辑
4. 返回用户态继续执行

代价：态切换伴随寄存器保存恢复和流水线影响，因此频繁小 I/O 需要通过缓冲减少系统调用。

知识扩展：
用户态/内核态切换与中断处理、系统调用优化 (如 vDSO、io_uring) 有直接关联。

### 1.4 进程之间的通信方式有哪些

常见 IPC (Inter-Process Communication) 方式：

- 管道 (pipe): 半双工，常用于父子进程
- 命名管道 (FIFO): 支持无亲缘关系进程
- 消息队列: 以消息为单位，支持优先级
- 共享内存: 性能最高，需自行同步 (锁/信号量)
- 信号量: 主要用于同步与互斥，不直接传递大数据
- 信号 (signal): 异步通知机制，语义简单
- 套接字 (socket): 最通用，可跨主机
- 内存映射文件 (mmap): 文件与内存映射，适合大数据共享

面试重点对比：

- 性能: 共享内存通常最快
- 可靠性和易用性: 消息队列更结构化
- 跨机能力: socket 唯一通用选择

知识扩展：
高并发服务常把“共享内存 + 无锁队列”用于同机低延迟通信，把“socket + 序列化协议”用于跨机 RPC。

### 1.5 什么是 CPU 缓存一致性协议 MESI

MESI 是多核系统下保证缓存一致性的经典协议，定义了缓存行的四种状态：

- M (Modified): 本核已修改，数据仅在本核缓存，和主内存不一致
- E (Exclusive): 仅本核持有，和主内存一致
- S (Shared): 多核共享，和主内存一致
- I (Invalid): 无效状态

核心目标：当某个核写入共享数据时，其他核上的旧缓存副本必须失效或更新，避免读到脏数据。

典型流程：

- 某核在 S 状态写入 -> 触发总线消息使其他核该行失效 -> 本核变 M
- 其他核读取该地址时，需要从拥有最新数据的缓存或内存获取

工程影响：

- 伪共享 (false sharing) 会导致频繁缓存行抖动，严重影响并发性能
- 高并发计数器常通过缓存行填充避免伪共享

知识扩展：
MESI 与内存模型 (如 x86 TSO) 和原子指令成本密切相关，是理解锁和无锁算法性能差异的关键。

### 1.6 进程的调度算法有哪些

常见调度算法可以分为非抢占和抢占两类。

- FCFS (First Come First Served): 先来先服务，简单但可能导致短作业饥饿
- SJF (Shortest Job First): 优先短作业，平均等待时间优，但需要预估运行时长
- SRTF (Shortest Remaining Time First): SJF 抢占版
- RR (Round Robin): 时间片轮转，响应快，适合分时系统
- Priority Scheduling: 按优先级调度，需防止低优先级饥饿
- Multilevel Queue: 不同队列对应不同任务类型
- Multilevel Feedback Queue: 经典通用方案，动态调整优先级，兼顾吞吐与响应

现代 Linux 的 CFS (Completely Fair Scheduler) 核心思想是“按虚拟运行时间公平调度”，不是传统固定时间片轮转。

知识扩展：
调度策略与系统目标相关。交互系统强调低延迟，批处理系统强调吞吐，实时系统强调截止期可预测性。

### 1.7 操作系统在进行线程切换时需要进行哪些动作

线程切换本质是保存当前上下文并恢复下一个线程上下文。

主要动作：

1. 保存当前线程 CPU 上下文 (通用寄存器、PC、SP、标志寄存器)
2. 更新线程状态并放入相应队列 (就绪队列或阻塞队列)
3. 调度器选择下一个可运行线程
4. 恢复目标线程上下文
5. 必要时处理地址空间相关结构 (同进程线程通常无需切页表，跨进程会更重)
6. 返回执行目标线程

性能成本来源：

- 上下文保存恢复
- 缓存局部性破坏
- TLB 影响 (跨进程更明显)

知识扩展：
减少切换开销的常见方法包括绑核、减少锁竞争、批量系统调用、协程化 I/O。

### 1.8 什么是软中断，什么是硬中断

- 硬中断 (Hardware Interrupt): 由外设触发 (网卡、磁盘、时钟)，异步打断 CPU 当前执行流
- 软中断 (Software Interrupt): 由软件机制触发的延后处理机制，常用于把重工作从硬中断上下文搬到更安全环境

为什么需要软中断：

- 硬中断上下文要求短小快速，不能做复杂阻塞逻辑
- 把后续处理下沉到软中断/内核线程，提高系统可抢占性和响应性

Linux 中常见关联机制：hardirq、softirq、tasklet、workqueue。

知识扩展：
网络高吞吐场景常依赖 NAPI 结合软中断批量处理包，降低中断风暴与上下文切换。

### 1.9 为什么要有虚拟内存

虚拟内存的核心价值是抽象、隔离和扩展。

- 地址空间隔离: 每个进程看到独立虚拟地址空间，增强安全性
- 按需分配: 不是一次性加载全部数据，减少物理内存压力
- 扩展可用空间: 通过换页机制让程序“看起来”可用更大内存
- 简化编程模型: 进程不需要关心真实物理地址布局

如果没有虚拟内存：

- 多进程互相干扰风险大
- 程序必须处理复杂物理内存管理
- 内存利用率与可移植性显著下降

代价：

- 地址翻译开销
- 缺页中断开销
- 过度换页会导致抖动 (thrashing)

知识扩展：
TLB (Translation Lookaside Buffer) 是虚拟内存性能关键，页表设计和大页机制都在围绕减少地址翻译成本。

### 1.10 什么是分段，什么是分页

两者都是内存管理机制，但切分维度不同。

- 分段 (Segmentation): 按逻辑意义划分，如代码段、数据段、栈段，段大小可变
- 分页 (Paging): 按固定大小页划分，页大小固定 (如 4KB)

对比：

- 分段便于体现程序逻辑，但会有外部碎片
- 分页易于物理内存管理，主要有内部碎片
- 现代系统多以分页为主，分段作用弱化 (x86-64 下更明显)

地址转换：

- 分段: 逻辑地址 = 段号 + 段内偏移
- 分页: 线性地址 = 页号 + 页内偏移

知识扩展：
页式管理与多级页表、反向页表、HugePage 等机制相关，决定了大内存场景的性能与页表开销。

### 1.11 CPU 使用率和 CPU 负载指的是什么，它们之间有什么关系

- CPU 使用率 (CPU Utilization): 某时间窗口内 CPU 忙碌时间占比
- CPU 负载 (Load Average): 一段时间内处于可运行态 (R) 和不可中断睡眠态 (D) 的平均任务数

两者区别：

- 使用率看“CPU 忙不忙”
- 负载看“系统排队长不长”

典型现象：

- 高使用率 + 高负载: CPU 可能是瓶颈
- 低使用率 + 高负载: 可能大量任务卡在 I/O 不可中断等待 (D 状态)
- 高使用率 + 低负载: 可能是少量计算密集任务吃满核

面试常用判断：

- 单核机器 load 接近 1 通常接近满载
- N 核机器 load 接近 N 通常接近满载

知识扩展：
排障时要联合看 load、us/sys/iowait、上下文切换、run queue 长度，避免只看单一指标误判。

### 1.12 常用的 Linux 指令

按面试高频场景整理如下。

系统与资源：

- `top` / `htop`: 实时查看 CPU、内存、进程
- `uptime`: 查看 load average
- `vmstat`: 看上下文切换、内存、I/O
- `free -h`: 看内存和 swap

进程与线程：

- `ps -ef` / `ps aux`: 查看进程
- `pgrep` / `pkill`: 按名称查杀进程
- `pidstat -p <pid> 1`: 观察指定进程资源变化

文件与文本：

- `ls`, `cp`, `mv`, `rm`, `find`
- `cat`, `less`, `head`, `tail -f`
- `grep`, `awk`, `sed`, `sort`, `uniq`, `cut`

网络排查：

- `ss -lntp` / `netstat -anp`: 看端口与连接
- `curl`: 调接口
- `ping`, `traceroute`, `dig` / `nslookup`

磁盘与 I/O：

- `df -h`: 磁盘容量
- `du -sh *`: 目录体积
- `iostat -x 1`: 磁盘 I/O 指标
- `lsof`: 查看文件句柄与端口占用

知识扩展：
生产环境定位问题通常是命令组合拳，例如 `top + pidstat + ss + iostat + lsof`，而不是单命令判断。

### 1.13 为什么网络 I/O 会被阻塞

网络 I/O 阻塞的根因是“当前读写条件不满足”。

- 读阻塞: 接收缓冲区没有数据可读
- 写阻塞: 发送缓冲区满，无法继续写入
- 连接阻塞: 三次握手或 DNS 解析尚未完成

从内核视角看：

- 进程发起阻塞式系统调用后，如果条件未满足，会被挂起并从运行队列移除
- 当网卡中断到来、协议栈处理完成，内核再唤醒对应进程

关键点：

- 阻塞不是“浪费 CPU 忙等”，而是让出 CPU
- 但线程被挂起会带来调度延迟和上下文切换成本

知识扩展：
这也是 I/O 多路复用出现的原因，用一个线程监听多个 fd，减少线程数量与阻塞开销。

### 1.14 I/O 模型有哪些

Unix/Linux 常见五种 I/O 模型：

- 阻塞 I/O (blocking I/O): 调用直到完成才返回
- 非阻塞 I/O (non-blocking I/O): 立即返回，未就绪返回 EAGAIN
- I/O 多路复用 (select/poll/epoll): 先等待“就绪事件”，再执行读写
- 信号驱动 I/O: 就绪时内核发信号通知进程
- 异步 I/O (AIO): 提交后立即返回，内核完成后通知“已完成”

面试易混点：

- 多路复用是“同步 I/O”，因为数据拷贝阶段通常仍由进程主动调用 read/write
- 异步 I/O 才是内核完成整个 I/O 后通知用户进程

知识扩展：
Reactor 基于 I/O 多路复用，Proactor 更贴近真正异步 I/O 完成通知模型。

### 1.15 同步和异步的区别

同步/异步描述的是“调用结果通知方式”。

- 同步: 调用方主动等待或轮询结果
- 异步: 被调方完成后主动通知调用方 (回调、信号、事件)

关键不是“是否阻塞线程”，而是“谁来推进结果获取”。

示例：

- 同步非阻塞: 不断轮询状态
- 同步阻塞: 一直等到返回
- 异步非阻塞: 提交后继续做别的，完成后收到通知

知识扩展：
同步/异步和阻塞/非阻塞是两个正交维度，很多误区来自把它们混为一谈。

### 1.16 阻塞和非阻塞的区别

阻塞/非阻塞描述的是“调用当下线程是否被挂起”。

- 阻塞: 条件不满足时，线程睡眠等待
- 非阻塞: 条件不满足时，立刻返回错误码或状态

影响：

- 阻塞编程简单，但大量连接需要大量线程
- 非阻塞需要事件循环或轮询，编程更复杂

知识扩展：
高并发网络服务通常使用非阻塞 fd + I/O 多路复用，把阻塞点收敛到 event loop。

### 1.17 同步、异步、阻塞、非阻塞的 I/O 的区别

可以用二维表理解：

- 维度 1: 同步/异步 (结果通知机制)
- 维度 2: 阻塞/非阻塞 (调用期间线程状态)

常见组合：

- 同步阻塞: 经典 read 阻塞直到数据到达
- 同步非阻塞: read 立即返回 EAGAIN，应用轮询
- 异步非阻塞: 提交请求即返回，完成后事件通知
- 异步阻塞: 实践中较少强调，语义上存在但工程上不常单独讨论

面试回答模板：

1. 先定义同步/异步是通知机制
2. 再定义阻塞/非阻塞是线程状态
3. 最后说明两者正交，不是同义词

知识扩展：
Java NIO、Netty、Go netpoll 都是把这四个概念拆开设计的典型案例。

### 1.18 到底什么是 Reactor

Reactor 是一种基于事件驱动和 I/O 多路复用的并发处理模式。

核心思想：

- 一个或多个 Reactor 线程阻塞在事件分离器 (select/poll/epoll)
- 当 fd 就绪后，分发给对应 Handler 执行业务逻辑

典型角色：

- Demultiplexer: 事件分离器 (epoll)
- Reactor: 事件循环与分发中心
- Handler: 具体事件处理逻辑

常见模型：

- 单 Reactor 单线程: 简单，适合低并发
- 单 Reactor 多线程: Reactor 负责分发，worker 处理业务
- 主从 Reactor 多线程: 主 Reactor 管 accept，从 Reactor 管读写与分发，适合高并发

知识扩展：
Reactor 关注“就绪事件”，Proactor 关注“完成事件”。二者是网络框架设计中的两大基本范式。

### 1.19 Select、Poll、Epoll 之间有什么区别

三者都是 I/O 多路复用机制，但在数据结构和复杂度上差异明显。

select：

- 使用位图保存 fd 集
- fd 数量有上限 (常见 1024，受 FD_SETSIZE 影响)
- 每次调用都要把全集从用户态拷贝到内核态并线性扫描

poll：

- 用数组描述 fd，无固定 1024 上限
- 仍需每次线性扫描全部 fd
- 仍存在重复拷贝与遍历成本

epoll (Linux)：

- 通过 `epoll_ctl` 维护兴趣列表，避免每次全量传参
- 就绪队列返回活跃 fd，减少无效遍历
- 支持 LT (水平触发) 和 ET (边缘触发)
- 在高并发大量空闲连接场景性能优势明显

常见误区：

- epoll 不是在任何规模都绝对更快，小连接数时优势不明显
- ET 模式性能高但更易写错，必须配合非阻塞与循环读写直到 EAGAIN

知识扩展：
epoll 是 Reactor 在 Linux 下最常见实现基础，进一步演进有 io_uring 等新机制，目标是减少系统调用与拷贝开销。

## 2. 计算机网络面试题

### 2.1 TCP 和 UDP 有什么区别

这个问题建议从四个维度回答：连接语义、可靠性、时延与开销、适用场景。

核心结论：

- TCP (Transmission Control Protocol) 是面向连接、可靠、有序、字节流的传输协议
- UDP (User Datagram Protocol) 是无连接、不保证可靠、不保证有序、面向报文的传输协议

#### 一、机制层面的本质差异

- 连接方式：
	- TCP 需要三次握手建立连接，四次挥手释放连接
	- UDP 无需建连，直接发送
- 可靠性：
	- TCP 通过序列号、确认应答 (ACK)、重传、超时控制保证可靠传输
	- UDP 默认尽力而为 (best effort)，丢包、乱序、重复都可能发生
- 有序性：
	- TCP 接收端按序重组，应用层通常读到有序数据
	- UDP 不维护全局顺序
- 流量与拥塞控制：
	- TCP 有滑动窗口、流量控制、拥塞控制 (慢启动、拥塞避免、快速重传等)
	- UDP 无内建拥塞控制 (是否限速由应用层自行设计)

#### 二、数据边界与开销差异

- 传输单位：
	- TCP 是字节流协议，不保留应用层消息边界
	- UDP 是报文协议，保留一次 send 对应一次报文边界
- 首部开销：
	- TCP 头部最小 20 字节 (不含选项)
	- UDP 头部固定 8 字节
- 时延特性：
	- TCP 因建连、重传、拥塞控制，平均时延通常更高但更稳
	- UDP 路径短、协议轻，实时性常更好，但抖动和丢包需上层兜底

#### 三、典型应用场景

- 适合 TCP 的场景：
	- 对正确性和完整性要求高的业务，如 HTTP/HTTPS、数据库连接、文件传输、邮件
- 适合 UDP 的场景：
	- 强实时、可容忍少量丢包的业务，如音视频通话、实时游戏、DNS 查询、直播推流

实践中很多“看起来像 UDP 的可靠通信”其实是在应用层自建可靠机制，例如重传、前向纠错 (FEC)、序号与抖动缓冲。

#### 四、面试高频误区

- 误区 1：UDP 不可靠就一定“差”
	- 纠正：UDP 是把复杂度上移到应用层，换取更低时延和更高灵活性
- 误区 2：TCP 一定比 UDP 慢很多
	- 纠正：小规模低丢包网络中差异未必巨大，但在高实时场景下 TCP 的重传与队头阻塞 (Head-of-Line Blocking) 影响更明显
- 误区 3：UDP 不安全
	- 纠正：安全不由 TCP/UDP 单独决定，可以在 UDP 之上叠加 DTLS 或 QUIC 的安全机制

#### 五、一个简明对比表

| 维度     | TCP               | UDP               |
| -------- | ----------------- | ----------------- |
| 连接     | 面向连接          | 无连接            |
| 可靠性   | 可靠传输          | 尽力而为          |
| 顺序     | 保序              | 不保序            |
| 传输语义 | 字节流            | 报文              |
| 拥塞控制 | 有                | 无 (应用层可实现) |
| 头部开销 | 较大              | 较小              |
| 典型场景 | Web、DB、文件传输 | 音视频、游戏、DNS |

知识扩展：
这个问题和 QUIC 强相关。QUIC 基于 UDP 实现了“连接管理 + 可靠传输 + 拥塞控制 + TLS”，本质是在用户态重构了传统 TCP 的很多能力，同时减少队头阻塞并加速连接建立，是现代网络协议演进的代表方向。
