## 1. 操作系统面试题

### 1.1 操作系统中的进程有哪几种状态

经典的进程状态模型是三态和五态，面试中最常用的是五态模型。

- 新建 (New): 进程刚被创建，PCB (Process Control Block) 已建立，但还未进入就绪队列
- 就绪 (Ready): 进程具备运行条件，只差 CPU 时间片
- 运行 (Running): 正在 CPU 上执行
- 阻塞 (Blocked/Waiting): 等待某个事件 (I/O 完成、锁释放、信号到达)
- 终止 (Terminated): 进程执行结束，等待系统回收资源

很多系统会细化为七态，在五态基础上增加：

- 就绪挂起 (Ready Suspended): 可运行但被换出到外存
- 阻塞挂起 (Blocked Suspended): 阻塞且被换出到外存

状态迁移核心逻辑：

- Running -> Blocked: 主动等待资源
- Running -> Ready: 时间片用完或被更高优先级任务抢占
- Blocked -> Ready: 等待事件到达
- Ready -> Running: 被调度器选中

知识扩展：
状态迁移与调度算法强相关。例如抢占式调度会显著增加 Running -> Ready 的切换频率，这会影响上下文切换成本与缓存命中率。

### 1.2 线程和进程有什么区别

核心结论：进程是资源分配单位，线程是 CPU 调度单位。

关键区别：

- 地址空间: 进程间隔离，线程共享所属进程地址空间
- 资源拥有: 进程拥有独立资源 (虚拟内存、文件描述符表等)，线程主要共享这些资源
- 切换开销: 进程切换通常比线程切换更重，因为涉及地址空间切换和 TLB (Translation Lookaside Buffer，用于加速虚拟地址到物理地址转换的缓存) 影响
- 通信方式: 进程常用 IPC，线程可直接读写共享内存 (但要同步)
- 故障影响: 一个进程崩溃通常不直接影响其他进程；线程崩溃可能导致整个进程异常

工程视角：

- 需要强隔离和稳定性时优先多进程
- 需要低延迟共享数据时优先多线程

知识扩展：
用户态线程和内核线程模型 (1:1、M:1、M:N) 会影响并发能力和阻塞行为，是理解 Go 协程调度器的基础。

### 1.3 什么是用户态和内核态

用户态和内核态是 CPU 权限级别的划分。

- 用户态 (User Mode): 权限受限，不能直接执行特权指令，不能直接访问硬件
- 内核态 (Kernel Mode): 拥有最高权限，可访问硬件、管理内存、调度进程

为什么要分态：

- 安全隔离: 防止应用直接破坏系统
- 稳定性: 应用故障不应直接破坏内核核心数据结构
- 受控访问: 通过系统调用统一访问内核服务

典型切换路径：

1. 用户程序发起系统调用 (如 read, write)
2. 触发陷入 (trap) 进入内核态
3. 内核执行服务逻辑
4. 返回用户态继续执行

代价：态切换伴随寄存器保存恢复和流水线影响，因此频繁小 I/O 需要通过缓冲减少系统调用。

知识扩展：
用户态/内核态切换与中断处理、系统调用优化 (如 vDSO、io_uring) 有直接关联。

### 1.4 进程之间的通信方式有哪些

常见 IPC (Inter-Process Communication) 方式：

- 管道 (pipe): 半双工，常用于父子进程
- 命名管道 (FIFO): 支持无亲缘关系进程
- 消息队列: 以消息为单位，支持优先级
- 共享内存: 性能最高，需自行同步 (锁/信号量)
- 信号量: 主要用于同步与互斥，不直接传递大数据
- 信号 (signal): 异步通知机制，语义简单
- 套接字 (socket): 最通用，可跨主机
- 内存映射文件 (mmap): 文件与内存映射，适合大数据共享

面试重点对比：

- 性能: 共享内存通常最快
- 可靠性和易用性: 消息队列更结构化
- 跨机能力: socket 唯一通用选择

知识扩展：
高并发服务常把“共享内存 + 无锁队列”用于同机低延迟通信，把“socket + 序列化协议”用于跨机 RPC。

### 1.5 什么是 CPU 缓存一致性协议 MESI

MESI 是多核系统下保证缓存一致性的经典协议，定义了缓存行的四种状态：

- M (Modified): 本核已修改，数据仅在本核缓存，和主内存不一致
- E (Exclusive): 仅本核持有，和主内存一致
- S (Shared): 多核共享，和主内存一致
- I (Invalid): 无效状态

核心目标：当某个核写入共享数据时，其他核上的旧缓存副本必须失效或更新，避免读到脏数据。

典型流程：

- 某核在 S 状态写入 -> 触发总线消息使其他核该行失效 -> 本核变 M
- 其他核读取该地址时，需要从拥有最新数据的缓存或内存获取

工程影响：

- 伪共享 (false sharing) 会导致频繁缓存行抖动，严重影响并发性能
- 高并发计数器常通过缓存行填充避免伪共享

知识扩展：
MESI 与内存模型 (如 x86 TSO) 和原子指令成本密切相关，是理解锁和无锁算法性能差异的关键。

### 1.6 进程的调度算法有哪些

常见调度算法可以分为非抢占和抢占两类。

- FCFS (First Come First Served): 先来先服务，简单但可能导致短作业饥饿
- SJF (Shortest Job First): 优先短作业，平均等待时间优，但需要预估运行时长
- SRTF (Shortest Remaining Time First): SJF 抢占版
- RR (Round Robin): 时间片轮转，响应快，适合分时系统
- Priority Scheduling: 按优先级调度，需防止低优先级饥饿
- Multilevel Queue: 不同队列对应不同任务类型
- Multilevel Feedback Queue: 经典通用方案，动态调整优先级，兼顾吞吐与响应

现代 Linux 的 CFS (Completely Fair Scheduler) 核心思想是“按虚拟运行时间公平调度”，不是传统固定时间片轮转。

知识扩展：
调度策略与系统目标相关。交互系统强调低延迟，批处理系统强调吞吐，实时系统强调截止期可预测性。

### 1.7 操作系统在进行线程切换时需要进行哪些动作

线程切换本质是保存当前上下文并恢复下一个线程上下文。

主要动作：

1. 保存当前线程 CPU 上下文 (通用寄存器、PC、SP、标志寄存器)
2. 更新线程状态并放入相应队列 (就绪队列或阻塞队列)
3. 调度器选择下一个可运行线程
4. 恢复目标线程上下文
5. 必要时处理地址空间相关结构 (同进程线程通常无需切页表，跨进程会更重)
6. 返回执行目标线程

性能成本来源：

- 上下文保存恢复
- 缓存局部性破坏
- TLB 影响 (跨进程更明显)

知识扩展：
减少切换开销的常见方法包括绑核、减少锁竞争、批量系统调用、协程化 I/O。

### 1.8 什么是软中断，什么是硬中断

- 硬中断 (Hardware Interrupt): 由外设触发 (网卡、磁盘、时钟)，异步打断 CPU 当前执行流
- 软中断 (Software Interrupt): 由软件机制触发的延后处理机制，常用于把重工作从硬中断上下文搬到更安全环境

为什么需要软中断：

- 硬中断上下文要求短小快速，不能做复杂阻塞逻辑
- 把后续处理下沉到软中断/内核线程，提高系统可抢占性和响应性

Linux 中常见关联机制：hardirq、softirq、tasklet、workqueue。

知识扩展：
网络高吞吐场景常依赖 NAPI 结合软中断批量处理包，降低中断风暴与上下文切换。

### 1.9 为什么要有虚拟内存

虚拟内存的核心价值是抽象、隔离和扩展。

- 地址空间隔离: 每个进程看到独立虚拟地址空间，增强安全性
- 按需分配: 不是一次性加载全部数据，减少物理内存压力
- 扩展可用空间: 通过换页机制让程序“看起来”可用更大内存
- 简化编程模型: 进程不需要关心真实物理地址布局

如果没有虚拟内存：

- 多进程互相干扰风险大
- 程序必须处理复杂物理内存管理
- 内存利用率与可移植性显著下降

代价：

- 地址翻译开销
- 缺页中断开销
- 过度换页会导致抖动 (thrashing)

知识扩展：
TLB (Translation Lookaside Buffer) 是虚拟内存性能关键，页表设计和大页机制都在围绕减少地址翻译成本。

### 1.10 什么是分段，什么是分页

两者都是内存管理机制，但切分维度不同。

- 分段 (Segmentation): 按逻辑意义划分，如代码段、数据段、栈段，段大小可变
- 分页 (Paging): 按固定大小页划分，页大小固定 (如 4KB)

对比：

- 分段便于体现程序逻辑，但会有外部碎片
- 分页易于物理内存管理，主要有内部碎片
- 现代系统多以分页为主，分段作用弱化 (x86-64 下更明显)

地址转换：

- 分段: 逻辑地址 = 段号 + 段内偏移
- 分页: 线性地址 = 页号 + 页内偏移

知识扩展：
页式管理与多级页表、反向页表、HugePage 等机制相关，决定了大内存场景的性能与页表开销。

### 1.11 CPU 使用率和 CPU 负载指的是什么，它们之间有什么关系

- CPU 使用率 (CPU Utilization): 某时间窗口内 CPU 忙碌时间占比
- CPU 负载 (Load Average): 一段时间内处于可运行态 (R) 和不可中断睡眠态 (D) 的平均任务数

两者区别：

- 使用率看“CPU 忙不忙”
- 负载看“系统排队长不长”

典型现象：

- 高使用率 + 高负载: CPU 可能是瓶颈
- 低使用率 + 高负载: 可能大量任务卡在 I/O 不可中断等待 (D 状态)
- 高使用率 + 低负载: 可能是少量计算密集任务吃满核

面试常用判断：

- 单核机器 load 接近 1 通常接近满载
- N 核机器 load 接近 N 通常接近满载

知识扩展：
排障时要联合看 load、us/sys/iowait、上下文切换、run queue 长度，避免只看单一指标误判。

### 1.12 常用的 Linux 指令

按面试高频场景整理如下。

系统与资源：

- `top` / `htop`: 实时查看 CPU、内存、进程

- `vmstat`: 看上下文切换、内存、I/O
- `free -h`: 看内存和 swap

进程与线程：

- `ps -ef` / `ps aux`: 查看进程
- `pgrep` / `pkill`: 按名称查杀进程

文件与文本：

- `ls`, `cp`, `mv`, `rm`, `find`
- `cat`, `less`, `head`, `tail -f`
- `grep`, `awk`, `sed`, `sort`, `uniq`, `cut`

网络排查：

- `ss -lntp` / `netstat -anp`: 看端口与连接
- `curl`: 调接口
- `ping`, `traceroute`, `dig` / `nslookup`

磁盘与 I/O：

- `df -h`: 磁盘容量
- `du -sh *`: 目录体积
- `iostat -x 1`: 磁盘 I/O 指标
- `lsof`: 查看文件句柄与端口占用

知识扩展：
生产环境定位问题通常是命令组合拳，例如 `top + pidstat + ss + iostat + lsof`，而不是单命令判断。

### 1.13 为什么网络 I/O 会被阻塞

网络 I/O 阻塞的根因是“当前读写条件不满足”。

- 读阻塞: 接收缓冲区没有数据可读
- 写阻塞: 发送缓冲区满，无法继续写入
- 连接阻塞: 三次握手或 DNS 解析尚未完成

从内核视角看：

- 进程发起阻塞式系统调用后，如果条件未满足，会被挂起并从运行队列移除
- 当网卡中断到来、协议栈处理完成，内核再唤醒对应进程

关键点：

- 阻塞不是“浪费 CPU 忙等”，而是让出 CPU
- 但线程被挂起会带来调度延迟和上下文切换成本

知识扩展：
这也是 I/O 多路复用出现的原因，用一个线程监听多个 fd，减少线程数量与阻塞开销。

### 1.14 I/O 模型有哪些

Unix/Linux 常见五种 I/O 模型：

- 阻塞 I/O (blocking I/O): 调用直到完成才返回
- 非阻塞 I/O (non-blocking I/O): 立即返回，未就绪返回 EAGAIN
- I/O 多路复用 (select/poll/epoll): 先等待“就绪事件”，再执行读写
- 信号驱动 I/O: 就绪时内核发信号通知进程
- 异步 I/O (AIO): 提交后立即返回，内核完成后通知“已完成”

面试易混点：

- 多路复用是“同步 I/O”，因为数据拷贝阶段通常仍由进程主动调用 read/write
- 异步 I/O 才是内核完成整个 I/O 后通知用户进程

知识扩展：
Reactor 基于 I/O 多路复用，Proactor 更贴近真正异步 I/O 完成通知模型。

### 1.15 同步和异步的区别

同步/异步描述的是“调用结果通知方式”。

- 同步: 调用方主动等待或轮询结果
- 异步: 被调方完成后主动通知调用方 (回调、信号、事件)

关键不是“是否阻塞线程”，而是“谁来推进结果获取”。

示例：

- 同步非阻塞: 不断轮询状态
- 同步阻塞: 一直等到返回
- 异步非阻塞: 提交后继续做别的，完成后收到通知

知识扩展：
同步/异步和阻塞/非阻塞是两个正交维度，很多误区来自把它们混为一谈。

### 1.16 阻塞和非阻塞的区别

阻塞/非阻塞描述的是“调用当下线程是否被挂起”。

- 阻塞: 条件不满足时，线程睡眠等待
- 非阻塞: 条件不满足时，立刻返回错误码或状态

影响：

- 阻塞编程简单，但大量连接需要大量线程
- 非阻塞需要事件循环或轮询，编程更复杂

知识扩展：
高并发网络服务通常使用非阻塞 fd + I/O 多路复用，把阻塞点收敛到 event loop。

### 1.17 同步、异步、阻塞、非阻塞的 I/O 的区别

可以用二维表理解：

- 维度 1: 同步/异步 (结果通知机制)
- 维度 2: 阻塞/非阻塞 (调用期间线程状态)

常见组合：

- 同步阻塞: 经典 read 阻塞直到数据到达
- 同步非阻塞: read 立即返回 EAGAIN，应用轮询
- 异步非阻塞: 提交请求即返回，完成后事件通知
- 异步阻塞: 实践中较少强调，语义上存在但工程上不常单独讨论

面试回答模板：

1. 先定义同步/异步是通知机制
2. 再定义阻塞/非阻塞是线程状态
3. 最后说明两者正交，不是同义词

知识扩展：
Java NIO、Netty、Go netpoll 都是把这四个概念拆开设计的典型案例。

### 1.18 到底什么是 Reactor

Reactor 是一种基于事件驱动和 I/O 多路复用的并发处理模式。

核心思想：

- 一个或多个 Reactor 线程阻塞在事件分离器 (select/poll/epoll)
- 当 fd 就绪后，分发给对应 Handler 执行业务逻辑

典型角色：

- Demultiplexer: 事件分离器 (epoll)
- Reactor: 事件循环与分发中心
- Handler: 具体事件处理逻辑

常见模型：

- 单 Reactor 单线程: 简单，适合低并发
- 单 Reactor 多线程: Reactor 负责分发，worker 处理业务
- 主从 Reactor 多线程: 主 Reactor 管 accept，从 Reactor 管读写与分发，适合高并发

知识扩展：
Reactor 关注“就绪事件”，Proactor 关注“完成事件”。二者是网络框架设计中的两大基本范式。

### 1.19 Select、Poll、Epoll 之间有什么区别

三者都是 I/O 多路复用机制，但在数据结构和复杂度上差异明显。

select：

- 使用位图保存 fd 集
- fd 数量有上限 (常见 1024，受 FD_SETSIZE 影响)
- 每次调用都要把全集从用户态拷贝到内核态并线性扫描

poll：

- 用数组描述 fd，无固定 1024 上限
- 仍需每次线性扫描全部 fd
- 仍存在重复拷贝与遍历成本

epoll (Linux)：

- 通过 `epoll_ctl` 维护兴趣列表，避免每次全量传参
- 就绪队列返回活跃 fd，减少无效遍历
- 支持 LT (水平触发) 和 ET (边缘触发)
- 在高并发大量空闲连接场景性能优势明显

常见误区：

- epoll 不是在任何规模都绝对更快，小连接数时优势不明显
- ET 模式性能高但更易写错，必须配合非阻塞与循环读写直到 EAGAIN

知识扩展：
epoll 是 Reactor 在 Linux 下最常见实现基础，进一步演进有 io_uring 等新机制，目标是减少系统调用与拷贝开销。

### 1.20 Linux 系统中文件的权限有哪些？请写出对应的数字及其含义

面试中建议按“权限对象 -> 权限位 -> 数字换算 -> 特殊权限 -> 实战示例”这条线回答。

#### 一、权限对象 (三组)

Linux 文件权限按三类主体划分：

- `u` (user): 文件所有者
- `g` (group): 文件所属组用户
- `o` (others): 其他用户

例如 `-rwxr-xr--` 可以拆成三组：`rwx` (u)、`r-x` (g)、`r--` (o)。

#### 二、基础权限位与数字对应

每组里有三种基础权限：

- `r` (read，读) = 4
- `w` (write，写) = 2
- `x` (execute，执行) = 1

每组权限数字就是把该组存在的权限值相加。

常见组合：

- `7 = 4 + 2 + 1 = rwx`
- `6 = 4 + 2 = rw-`
- `5 = 4 + 1 = r-x`
- `4 = r--`
- `0 = ---`

所以三位数字权限如 `755` 的含义是：

- 所有者 `7` -> `rwx`
- 所属组 `5` -> `r-x`
- 其他用户 `5` -> `r-x`

#### 三、文件与目录的权限语义差异

这是高频考点，必须区分：

- 对普通文件：
  - `r` 表示可读文件内容
  - `w` 表示可修改文件内容
  - `x` 表示可执行该文件
- 对目录：
  - `r` 表示可读取目录项列表 (能 `ls`)
  - `w` 表示可在目录中创建/删除/重命名目录项
  - `x` 表示可进入目录并访问其子项 (search 权限)

面试常见追问：目录有 `w` 但没有 `x`，通常仍无法正常进入和操作目录内容，因为缺少遍历能力。

#### 四、特殊权限位 (第四位数字)

除了常见三位 (ugo) 外，还可能出现第四位，表示特殊权限：

- `setuid` = 4
- `setgid` = 2
- `sticky bit` = 1

因此四位权限 `4755` 的含义是：

- 第一位 `4` 表示 `setuid`
- 后三位 `755` 表示基础权限

三个特殊位常见语义：

- `setuid` (作用于可执行文件): 进程以文件所有者身份运行
- `setgid`:
  - 对文件: 进程以文件所属组身份运行
  - 对目录: 新建文件默认继承目录组
- `sticky bit` (常见于目录，如 `/tmp`): 目录可写，但仅文件拥有者或 root 能删除自己的文件

#### 五、常见数字权限速记与示例

- `644` (`rw-r--r--`): 普通配置文件常见，所有者可写，其他人只读
- `600` (`rw-------`): 私钥等敏感文件
- `755` (`rwxr-xr-x`): 可执行程序和脚本常见
- `700` (`rwx------`): 仅所有者可访问的目录/脚本
- `777` (`rwxrwxrwx`): 几乎总是不推荐，风险高
- `1777` (如 `/tmp`): 全员可写 + sticky bit 防互删

示例命令：

```bash
# 给脚本添加执行权限 (所有者可读写执行，其他用户可读执行)
chmod 755 deploy.sh

# 私钥最小化授权
chmod 600 id_rsa

# 共享目录，组内协作 + 继承组
chmod 2775 shared_dir

# 公共临时目录语义
chmod 1777 my_tmp
```

知识扩展：
文件权限只是第一层访问控制。实际生产环境还会结合 umask、ACL (Access Control List)、SELinux/AppArmor 等强制访问控制机制共同生效，所以“权限看起来对但仍访问失败”常常是多层策略叠加导致。

### 1.21 说说 awk 和 sed 指令的用法，并且举一个使用案例

这道题建议从定位、语法、典型能力和实战排障四个层面回答。

#### 一、awk 是什么，适合做什么

`awk` 是按“行读取、按字段处理”的文本处理工具，特别适合结构化文本统计与报表生成。

核心特点：

- 默认按空白分隔字段，`$1`、`$2`... 表示第 1、第 2 列
- `BEGIN` 里做初始化，主块处理每一行，`END` 里做汇总输出
- 内置变量很多，如 `NR` (当前行号)、`NF` (当前行字段数)

常见用法：

```bash
# 打印第 1 列和第 4 列
awk '{print $1, $4}' access.log

# 统计文件总行数
awk 'END {print NR}' access.log

# 按第 1 列分组计数
awk '{cnt[$1]++} END {for (k in cnt) print k, cnt[k]}' access.log
```

#### 二、sed 是什么，适合做什么

`sed` 是流编辑器 (stream editor)，擅长“按规则替换、删除、插入、提取”。

核心特点：

- 面向行处理，默认把结果输出到标准输出
- 最常用命令是 `s/old/new/` (替换)
- 可结合正则表达式做批量文本改写

常见用法：

```bash
# 全局替换
sed 's/error/ERROR/g' app.log

# 删除空行
sed '/^$/d' app.log

# 只打印第 10 到 20 行
sed -n '10,20p' app.log

# 原地修改文件 (macOS 需要提供备份后缀)
sed -i '' 's/127.0.0.1/localhost/g' config.txt
```

#### 三、awk 和 sed 的分工对比

- `sed` 更偏“编辑器”：改内容、删内容、抽取区间
- `awk` 更偏“分析器”：拆字段、算统计、做聚合

面试时一句话总结：

- 只做替换清洗优先用 `sed`
- 需要字段计算和汇总优先用 `awk`

#### 四、实战案例 (定位 Nginx 访问量最高的 IP)

场景：要从 `access.log` 找出访问次数最多的 5 个客户端 IP，同时日志中可能夹杂空行或注释行。

```bash
# 第一步: 用 sed 清洗无效行 (空行和以 # 开头的注释行)
sed '/^$/d; /^#/d' access.log \
# 第二步: 用 awk 提取第 1 列 IP，并按 IP 计数
| awk '{cnt[$1]++} END {for (ip in cnt) print cnt[ip], ip}' \
# 第三步: 按访问次数降序排序，取前 5
| sort -nr | head -n 5
```

输出示例：

```text
1523 10.2.3.11
1498 10.2.3.9
1320 10.2.3.21
1188 10.2.3.6
1102 10.2.3.18
```

这类命令链在生产排障里非常常见，优点是无需写脚本即可快速得到可行动结论。

#### 五、常见误区

- 误区 1：`sed` 和 `awk` 只能二选一
  - 实际上两者常串联使用，先清洗再统计
- 误区 2：`sed -i` 在各平台完全一致
  - GNU/Linux 与 macOS 参数细节不同，macOS 常写 `-i ''`
- 误区 3：日志字段固定不变
  - 真正生产中建议先抽样确认日志格式，再写字段索引逻辑

知识扩展：
`awk`/`sed` 与 `grep`、`sort`、`uniq`、`xargs` 组合后可以形成高效文本处理流水线。这个能力与 SRE 日志分析、线上故障快速定位直接相关，也是面试里“工程落地能力”常见考察点。

### 1.22 深入探讨：到底什么是 Reactor (通俗版)

如果用一句最通俗的话概括：

Reactor 就像一个“总调度前台”，它不亲自干所有活，而是专门盯着“谁有事件来了”，然后把对应事件分发给合适的处理者去执行。

#### 一、先解决一个根问题：为什么会有 Reactor

在高并发网络服务里，最早的写法常是“一个连接一个线程”。当连接数很大时会出现三个问题：

- 线程太多，内存占用高
- 线程切换频繁，CPU 开销大
- 大量连接空闲时，线程仍在阻塞等待，资源利用率差

Reactor 的目标就是：

- 用更少的线程管理更多连接
- 通过事件驱动替代“每个连接一个阻塞线程”

#### 二、Reactor 的核心机制 (按执行顺序理解)

可以把它拆成 4 步：

1. 注册事件
  - 把 socket 的“我关心什么事件”注册到事件分离器 (如 epoll)
  - 典型事件：可读、可写、连接到来
2. 等待事件
  - Reactor 线程阻塞在 `epoll_wait` 这类调用上
  - 注意这里是“等事件”，不是“等某个具体连接的数据”
3. 分发事件
  - 哪个 fd 就绪了，就把该事件分发给对应 Handler
4. 处理事件
  - Handler 执行业务逻辑，必要时继续注册/修改后续关注事件

面试可直接用这句话：

Reactor 本质是“事件就绪通知 + 回调分发”的并发模型。

#### 三、三个角色怎么记最清楚

- Event Demultiplexer (事件分离器): 像“雷达”，负责发现哪个 fd 就绪了
- Reactor (事件循环器): 像“调度台”，负责把事件路由给正确处理器
- Handler (处理器): 像“工人”，真正执行业务动作 (读、写、编解码、业务处理)

很多框架的架构图看起来复杂，本质都在这三者上做拆分和扩展。

#### 四、三种常见 Reactor 架构

1. 单 Reactor 单线程

- 一个线程做 accept、读写、业务处理
- 优点：实现最简单
- 缺点：业务一旦耗时，整个事件循环被拖慢

2. 单 Reactor 多线程

- Reactor 线程只负责事件分发
- 业务处理交给 worker 线程池
- 优点：避免单线程业务阻塞
- 缺点：跨线程交互复杂度上升

3. 主从 Reactor 多线程

- 主 Reactor 负责 accept 新连接
- 从 Reactor 负责已连接 fd 的读写事件
- 业务再交给 worker 池
- 优点：吞吐和扩展性最好，是工程实践主流

```plaintext
┌─────────────────────────────────────────────────────────────┐
│                  主Reactor（Main Reactor）                  │
│  ┌─────────┐                                                │
│  │ epoll   │───监听端口──▶ accept新连接                     │
│  │ master  │                │                               │
│  └─────────┘                │                               │
│                             ▼                               │
│                    将新连接分配给子Reactor                  │
│                    (round-robin或负载均衡)                  │
└─────────────────────────────────────────────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
│   Sub-Reactor 1 │   │   Sub-Reactor 2 │   │   Sub-Reactor N │
│  ┌───────────┐  │   │  ┌───────────┐  │   │  ┌───────────┐  │
│  │ epoll     │  │   │  │ epoll     │  │   │  │ epoll     │  │
│  │ 处理连接1 │  │   │  │ 处理连接2 │  │   │  │ 处理连接N │  │
│  │ 的读写事件│  │   │  │ 的读写事件│  │   │  │ 的读写事件│  │
│  └───────────┘  │   │  └───────────┘  │   │  └───────────┘  │
│  ┌───────────┐  │   │  ┌───────────┐  │   │  ┌───────────┐  │
│  │ Worker    │  │   │  │ Worker    │  │   │  │ Worker    │  │
│  │ 线程池    │  │   │  │ 线程池    │  │   │  │ 线程池    │  │
│  └───────────┘  │   │  └───────────┘  │   │  └───────────┘  │
└─────────────────┘   └─────────────────┘   └─────────────────┘

优势：
• 主Reactor专注accept，极轻量
• 多个子Reactor并行处理I/O，利用多核
• 每个子Reactor可绑定独立CPU核心（CPU亲和性）
• 连接在不同Sub-Reactor间隔离，减少锁竞争
```

#### 五、为什么说 Reactor 是“同步 I/O 模型”

这是高频易错点。

- `epoll` 告诉你的只是“这个 fd 现在可读/可写了” (就绪通知)
- 真正把数据从内核拷到用户空间，仍是应用线程调用 `read/write` 去做

因此 Reactor 是“同步非阻塞 + 多路复用”的典型模式，而不是严格意义的异步完成通知模型。

#### 六、Reactor 的高频注意点 (面试常追问)

1. 业务逻辑不能阻塞事件循环

- Reactor 线程里只做轻量工作
- 重逻辑丢给线程池，否则会引发尾延迟恶化

2. ET (边缘触发) 模式必须循环读写到 EAGAIN

- 否则容易丢事件，连接看起来“卡住”

3. 连接关闭和异常处理要完整

- 对端关闭、RST、半包粘包、超时清理都要覆盖

4. 写事件要按需注册

- 常见优化是“有待发送数据时才监听可写事件”，避免空转

#### 七、一个贴近真实服务的简化流程

```text
主 Reactor:
  1. epoll_wait 等待新连接
  2. accept 后把连接分配给某个从 Reactor

从 Reactor:
  1. 监听连接上的可读/可写事件
  2. 可读 -> 读数据 -> 解码 -> 投递业务线程池
  3. 业务结果回写缓冲区
  4. 注册可写事件，发送完成后取消可写监听
```

这个流程就是 Netty、Redis、Nginx 等高性能网络程序背后的核心思想变体。

#### 八、面试回答收束模板

可以这样收尾：

- Reactor 通过 I/O 多路复用把“海量连接等待”收敛到少量事件循环线程
- 它用“事件分发”替代“每连接一线程”，提升并发能力和资源利用率
- 工程落地关键在于事件循环不阻塞、读写边界处理完整、线程池协作合理

知识扩展：
Reactor 与 Proactor 是两大并发网络模型。Reactor 关注“就绪事件”，Proactor 关注“完成事件”。同时，Reactor 与零拷贝、线程模型设计 (主从 + 线程池)、协议编解码 (半包粘包处理) 强相关，通常一起构成高性能网络服务的基础架构。

## 2. 计算机网络面试题

### 2.1 TCP 和 UDP 有什么区别

这个问题建议从四个维度回答：连接语义、可靠性、时延与开销、适用场景。

核心结论：

- TCP (Transmission Control Protocol) 是面向连接、可靠、有序、字节流的传输协议
- UDP (User Datagram Protocol) 是无连接、不保证可靠、不保证有序、面向报文的传输协议

#### 一、机制层面的本质差异

- 连接方式：
  - TCP 需要三次握手建立连接，四次挥手释放连接
  - UDP 无需建连，直接发送
- 可靠性：
  - TCP 通过序列号、确认应答 (ACK)、重传、超时控制保证可靠传输
  - UDP 默认尽力而为 (best effort)，丢包、乱序、重复都可能发生
- 有序性：
  - TCP 接收端按序重组，应用层通常读到有序数据
  - UDP 不维护全局顺序
- 流量与拥塞控制：
  - TCP 有滑动窗口、流量控制、拥塞控制 (慢启动、拥塞避免、快速重传等)
  - UDP 无内建拥塞控制 (是否限速由应用层自行设计)

#### 二、数据边界与开销差异

- 传输单位：
  - TCP 是字节流协议，不保留应用层消息边界
  - UDP 是报文协议，保留一次 send 对应一次报文边界
- 首部开销：
  - TCP 头部最小 20 字节 (不含选项)
  - UDP 头部固定 8 字节
- 时延特性：
  - TCP 因建连、重传、拥塞控制，平均时延通常更高但更稳
  - UDP 路径短、协议轻，实时性常更好，但抖动和丢包需上层兜底

#### 三、典型应用场景

- 适合 TCP 的场景：
  - 对正确性和完整性要求高的业务，如 HTTP/HTTPS、数据库连接、文件传输、邮件
- 适合 UDP 的场景：
  - 强实时、可容忍少量丢包的业务，如音视频通话、实时游戏、DNS 查询、直播推流

实践中很多“看起来像 UDP 的可靠通信”其实是在应用层自建可靠机制，例如重传、前向纠错 (FEC)、序号与抖动缓冲。

#### 四、面试高频误区

- 误区 1：UDP 不可靠就一定“差”
  - 纠正：UDP 是把复杂度上移到应用层，换取更低时延和更高灵活性
- 误区 2：TCP 一定比 UDP 慢很多
  - 纠正：小规模低丢包网络中差异未必巨大，但在高实时场景下 TCP 的重传与队头阻塞 (Head-of-Line Blocking) 影响更明显
- 误区 3：UDP 不安全
  - 纠正：安全不由 TCP/UDP 单独决定，可以在 UDP 之上叠加 DTLS 或 QUIC 的安全机制

#### 五、一个简明对比表

| 维度     | TCP               | UDP               |
| -------- | ----------------- | ----------------- |
| 连接     | 面向连接          | 无连接            |
| 可靠性   | 可靠传输          | 尽力而为          |
| 顺序     | 保序              | 不保序            |
| 传输语义 | 字节流            | 报文              |
| 拥塞控制 | 有                | 无 (应用层可实现) |
| 头部开销 | 较大              | 较小              |
| 典型场景 | Web、DB、文件传输 | 音视频、游戏、DNS |

知识扩展：
这个问题和 QUIC 强相关。QUIC 基于 UDP 实现了“连接管理 + 可靠传输 + 拥塞控制 + TLS”，本质是在用户态重构了传统 TCP 的很多能力，同时减少队头阻塞并加速连接建立，是现代网络协议演进的代表方向。

### 2.2 HTTP 和 HTTPS 有什么区别

面试建议从“安全目标、协议栈位置、握手过程、性能影响、部署成本”五个角度回答。

核心结论：

- HTTP (HyperText Transfer Protocol) 是明文传输的应用层协议，本身不提供机密性和完整性保护
- HTTPS = HTTP + TLS (Transport Layer Security)，在传输层之上增加加密、完整性校验和身份认证

#### 一、核心差异

- 安全性：
  - HTTP 明文传输，容易被窃听、篡改、中间人攻击
  - HTTPS 通过 TLS 加密数据并校验完整性，显著提升安全性
- 身份认证：
  - HTTP 默认不验证服务端身份
  - HTTPS 通过数字证书和 CA (Certificate Authority) 信任链验证服务端身份
- 默认端口：
  - HTTP 常用 80
  - HTTPS 常用 443
- URL 前缀：
  - HTTP 使用 `http://`
  - HTTPS 使用 `https://`

#### 二、HTTPS 是如何“比 HTTP 更安全”的

HTTPS 的安全收益主要来自 TLS 提供的三类能力：

- 机密性：对称加密保护传输内容，防止被窃听
- 完整性：消息认证码 (MAC) 或 AEAD 防篡改
- 身份认证：证书机制验证对端身份，降低中间人攻击风险

简化握手过程 (以 TLS 1.2/1.3 思路理解)：

1. 客户端发起握手，声明支持的加密套件
2. 服务端返回证书与协商参数
3. 客户端验证证书合法性 (域名、有效期、签发链)
4. 双方协商出会话密钥，后续使用对称加密传输应用数据

#### 三、性能与成本差异

- HTTPS 额外开销：
  - 首次握手增加 RTT (Round-Trip Time)
  - 加解密会消耗一定 CPU
- 现代优化后影响通常可控：
  - TLS 1.3 减少握手往返
  - Session Resumption、HTTP/2、HTTP/3 可降低延迟
  - 硬件加速与连接复用可进一步摊薄成本

工程实践里，HTTPS 早已是默认标准，安全收益远大于额外开销。

#### 四、常见误区

- 误区 1：用了 HTTPS 就绝对安全
  - 纠正：HTTPS 主要保护传输链路，无法解决业务逻辑漏洞、XSS、SQL 注入等应用层问题
- 误区 2：HTTPS 一定很慢
  - 纠正：在现代协议栈和优化手段下，性能影响通常很小，且可通过 CDN/连接复用进一步优化
- 误区 3：证书只要有就行
  - 纠正：证书还要关注域名匹配、过期时间、吊销状态与完整信任链

#### 五、一个简明对比表

| 维度         | HTTP       | HTTPS                   |
| ------------ | ---------- | ----------------------- |
| 安全性       | 明文传输   | TLS 加密传输            |
| 完整性保护   | 无内建保障 | 有 (MAC/AEAD)           |
| 身份认证     | 无         | 证书 + CA 信任链        |
| 默认端口     | 80         | 443                     |
| 抗中间人能力 | 弱         | 强 (依赖证书校验)       |
| 部署复杂度   | 低         | 较高 (证书、续期、配置) |

知识扩展：
这个问题和 HSTS (HTTP Strict Transport Security)、证书自动化 (如 ACME/Let's Encrypt)、以及 HTTP/3 (基于 QUIC) 强相关。面试中若能进一步说明“传输安全只是整体安全的一层”，并联动 Cookie 安全属性、CSP、同源策略，会体现更完整的安全体系理解。

### 2.3 详细说明一下 TCP 三次握手和四次挥手的过程及其中的注意点

这道题建议按“为什么需要 -> 报文流程 -> 状态迁移 -> 常见追问”来答，面试效果最好。

#### 一、三次握手 (建立连接)

TCP 建连的核心目标有三个：

- 双方确认对端可达 (发送与接收能力正常)
- 同步初始序列号 (Initial Sequence Number, ISN)
- 协商连接参数 (如 MSS、窗口扩大、SACK 等)

三次握手流程：

1. 第一次握手 (Client -> Server)
  - 客户端发送 `SYN=1, seq=x`
  - 客户端状态：`CLOSED -> SYN_SENT`
2. 第二次握手 (Server -> Client)
  - 服务端回复 `SYN=1, ACK=1, seq=y, ack=x+1`
  - 服务端状态：`LISTEN -> SYN_RCVD`
3. 第三次握手 (Client -> Server)
  - 客户端发送 `ACK=1, seq=x+1, ack=y+1`
  - 客户端进入 `ESTABLISHED`
  - 服务端收到该 ACK 后进入 `ESTABLISHED`

为什么不是两次握手：

- 两次只能保证“客户端能发、服务端能收并回”，不能保证“服务端发的数据客户端一定能收并确认”
- 三次中的最后一次 ACK 是服务端确认“客户端收到了我的 SYN+ACK”，避免历史失效连接请求导致的误建连

#### 二、四次挥手 (释放连接)

TCP 是全双工，两个方向的数据流要分别关闭，因此通常需要四次。

四次挥手流程 (主动关闭方记为 A，被动关闭方记为 B)：

1. 第一次挥手 (A -> B)
  - A 发送 `FIN=1, seq=u`，表示 A 不再发送数据
  - A 状态：`ESTABLISHED -> FIN_WAIT_1`
2. 第二次挥手 (B -> A)
  - B 回 `ACK=1, ack=u+1`
  - B 状态：`ESTABLISHED -> CLOSE_WAIT`
  - A 收到 ACK 后：`FIN_WAIT_1 -> FIN_WAIT_2`
3. 第三次挥手 (B -> A)
  - 当 B 数据发送完毕，发送 `FIN=1, seq=v`
  - B 状态：`CLOSE_WAIT -> LAST_ACK`
4. 第四次挥手 (A -> B)
  - A 回 `ACK=1, ack=v+1`
  - A 状态：`FIN_WAIT_2 -> TIME_WAIT`
  - B 收到 ACK 后：`LAST_ACK -> CLOSED`
  - A 等待 2MSL 后：`TIME_WAIT -> CLOSED`

为什么通常不是三次挥手：

- 第二次 ACK 和第三次 FIN 在被动关闭方通常不是同一时刻产生，因为 B 可能还有数据要发送
- 只有在极少数“B 无待发数据”的场景下，ACK 和 FIN 才可能合并

#### 三、关键注意点 (高频面试追问)

1. 为什么 TIME_WAIT 要等 2MSL

- 确保最后一个 ACK 有机会到达对端，若丢失可重传
- 让旧连接中的延迟报文在网络中自然消失，避免污染后续同 4 元组新连接

2. 为什么主动关闭方常出现大量 TIME_WAIT

- 主动关闭的一方进入 TIME_WAIT，这在网关、反向代理、短连接客户端上很常见
- 大量 TIME_WAIT 会占用本地端口与内核资源，但它是协议正确性的代价，不应简单粗暴关闭

3. SYN 洪泛 (SYN Flood) 与半连接队列

- 攻击者只发 SYN 不完成第三次握手，会消耗 `SYN_RCVD` 半连接资源
- 常见防护：SYN Cookie、调大 backlog、限速与防火墙策略

4. RST 和 FIN 的区别

- FIN 是“有序关闭”，尽量保证已发送数据完整交付
- RST 是“异常重置”，立即中断连接，未送达数据可能丢失

5. 抓包分析时应重点看什么

- 标志位 (SYN/ACK/FIN/RST)
- 序列号与确认号递增关系
- 重传、乱序、窗口大小变化
- RTT 与握手耗时

#### 四、一个简明状态流转图 (便于复盘)

```text
三次握手：
Client: CLOSED -> SYN_SENT -> ESTABLISHED
Server: LISTEN -> SYN_RCVD -> ESTABLISHED

四次挥手 (A 主动关闭)：
A: ESTABLISHED -> FIN_WAIT_1 -> FIN_WAIT_2 -> TIME_WAIT -> CLOSED
B: ESTABLISHED -> CLOSE_WAIT -> LAST_ACK -> CLOSED
```

#### 五、面试回答收束模板

可以这样总结：

- 三次握手的本质是“双方收发能力确认 + ISN 同步 + 参数协商”
- 四次挥手的本质是“TCP 全双工连接的双向独立关闭”
- TIME_WAIT、半连接队列、SYN Flood 防护是工程落地中的重点，不只是背流程

知识扩展：
这道题与滑动窗口、重传机制 (超时重传、快速重传)、拥塞控制 (慢启动/拥塞避免)、以及应用层长短连接策略强相关。理解握手挥手之后，再结合抓包工具分析 RTT、重传和窗口变化，能把“协议原理”真正落到“线上排障能力”。

### 2.4 具体说明 TCP 的粘包和拆包是什么。为什么会出现这两种现象？如何解决？

这个问题是 TCP 面试高频题，建议回答顺序是：先定义，再解释为什么必然发生，最后讲工程级解决方案。

#### 一、什么是粘包和拆包

先说结论：

- 粘包：应用层原本发了多条消息，接收端一次 `read` 读到了“多条消息拼在一起”的数据
- 拆包：应用层原本发了一条完整消息，接收端需要多次 `read` 才能读全

举例：

- 发送端连续发送两条业务消息：`HELLO` 和 `WORLD`
- 接收端可能一次读到 `HELLOWORLD` (粘包)
- 也可能先读到 `HEL`，再读到 `LOWORLD` (拆包)

关键认知：

TCP 是字节流协议，不保留应用层消息边界。它只保证字节有序、可靠到达，不保证“你一次 write 对应对方一次 read”。

#### 二、为什么会出现粘包和拆包

本质原因不是 TCP 出错，而是 TCP 的设计目标与应用层“按消息处理”的目标不同。

常见成因：

1. TCP 面向字节流，没有消息边界

- 内核看到的是连续字节序列，不知道你的“业务包”从哪开始到哪结束

2. 发送端合并发送

- Nagle 算法、发送缓冲聚合、应用连续小包写入，都可能让多条业务消息合并下发

3. 接收端读取时机和缓冲区大小不匹配

- `read` 取决于当时缓冲区已有字节数和你给的读取长度
- 缓冲区太小就会拆包，缓冲区里恰好多条消息就会粘包

4. 网络传输分段与重组是动态的

- MTU、拥塞控制、重传等机制会影响每次到达的数据块大小

一句话总结成因：

应用层按“消息”思考，TCP 按“字节流”传输，两者语义不一致就会出现粘包/拆包。

#### 三、如何解决 (核心是应用层协议设计)

TCP 层本身不解决“消息边界”，必须在应用层定义明确边界规则。常见方案有三类：

1. 固定长度协议

- 每条消息固定 N 字节
- 优点：实现简单
- 缺点：灵活性差，浪费带宽

2. 分隔符协议

- 用特殊分隔符标识一条消息结束 (如 `\n`)
- 优点：文本协议简单直观
- 缺点：消息体含分隔符时需转义，二进制场景不优雅

3. 长度字段协议 (最常用)

- 在消息头先写长度，再写消息体
- 接收端先读定长头，再按长度读取完整体
- 优点：通用、稳定、适合二进制协议，是主流工程方案

#### 四、长度字段方案示例 (通俗版)

约定协议格式：

- 前 4 字节：消息体长度 (uint32, big-endian)
- 后续 N 字节：消息体内容

示例：消息体是 `hello` (5 字节)

- 头部：`00 00 00 05`
- 全包：`00 00 00 05 68 65 6c 6c 6f`

无论底层 read 如何分片，接收端都能按“先头后体”组装出完整消息。

#### 五、一个简化 Go 伪代码示例 (解码侧)

```go
// decodeLoop 展示“长度字段协议”如何避免粘包/拆包问题
// buf 中可能一次来了半包、一包、或多包
func decodeLoop(buf []byte) {
    const headerLen = 4
    offset := 0

    for {
        // 1) 头都不够，等待更多数据
        if len(buf)-offset < headerLen {
            break
        }

        // 2) 读取消息体长度
        bodyLen := int(binary.BigEndian.Uint32(buf[offset : offset+headerLen]))

        // 3) 长度非法，直接丢弃或断开连接 (防御性编程)
        if bodyLen < 0 || bodyLen > 10*1024*1024 {
            // handle protocol error
            return
        }

        // 4) 消息体不完整，等待更多数据
        if len(buf)-offset < headerLen+bodyLen {
            break
        }

        // 5) 取出完整一帧
        frame := buf[offset+headerLen : offset+headerLen+bodyLen]
        handleMessage(frame)

        // 6) 继续解析下一个包 (兼容一次读到多包)
        offset += headerLen + bodyLen
    }

    // 剩余半包数据留待下次拼接再解
    // buf = buf[offset:]
}
```

这个模式同时解决两类问题：

- 拆包：半包先缓存，等齐再解
- 粘包：循环解析，一次 read 可拆出多帧

#### 六、面试常见误区与注意点

- 误区 1：关闭 Nagle 就能彻底解决粘包
  - 纠正：关闭 Nagle 只能影响发送聚合策略，不能提供消息边界
- 误区 2：以为每次 read 就是一条完整业务消息
  - 纠正：这是 UDP/消息队列思维，不适用于 TCP 字节流
- 误区 3：只考虑功能，不做长度合法性校验
  - 纠正：可能被恶意长度字段打爆内存，必须做上限校验

#### 七、面试回答收束模板

可以这样收尾：

- TCP 粘包/拆包不是 bug，而是字节流语义的自然结果
- 真正的解决方案在应用层协议边界设计
- 生产中主流是“长度字段 + 缓冲区累积 + 循环解码 + 异常长度防御”

知识扩展：
这个问题与 Netty 的编解码器设计强相关，如 LineBasedFrameDecoder、DelimiterBasedFrameDecoder、LengthFieldBasedFrameDecoder。它也和半包粘包处理、背压机制、零拷贝缓冲管理有关，是网络服务稳定性的核心基础能力。

### 2.5 HTTP 的返回码有哪些？请全部列举出来，并说明具体的含义。

- 1xx: 信息性响应 (请求已接收，继续处理)

- 成功类: 200, 201, 204, 206
- 重定向: 301, 302, 304, 307, 308
- 客户端错误: 400, 401, 403, 404, 405, 409, 413, 429
- 服务端错误: 500, 502, 503, 504

#### 七、常见误区

- 误区 1：401 和 403 是同一个意思
  - 纠正：401 是“未认证/认证失败”，403 是“已识别身份但无权限”
- 误区 2：302 和 307 完全等价
  - 纠正：307 明确要求保持原方法，302 在历史实现中可能改变方法
- 误区 3：所有失败都返回 500
  - 纠正：应区分客户端错误和服务端错误，返回更精确状态码便于排障与治理

知识扩展：
HTTP 状态码与缓存策略 (304, ETag, Last-Modified)、限流熔断 (429, 503)、反向代理网关治理 (502, 504) 强相关。面试中若能把状态码和具体排障路径绑定起来 (例如 502 查上游可用性、504 查超时链路、429 查限流规则)，会明显体现工程实战能力。

### 2.6 说说 TCP 拥塞控制的步骤

这道题建议按“目标 -> 步骤 -> 触发条件 -> 工程观察”来答，层次更完整。

核心结论：

- TCP 拥塞控制的目标是避免网络进入持续拥塞，同时尽可能提高吞吐。
- 经典流程通常分为四个阶段：慢启动 (Slow Start)、拥塞避免 (Congestion Avoidance)、快速重传 (Fast Retransmit)、快速恢复 (Fast Recovery)。

#### 一、为什么需要拥塞控制

TCP 发送端并不知道路径上每一跳的实时承载能力。如果只追求高速发送，会导致：

- 路由器队列持续堆积，RTT 上升
- 缓冲区溢出，丢包增加
- 重传进一步加剧拥塞，进入恶性循环

因此发送端维护拥塞窗口 `cwnd` (congestion window)，并与接收端窗口 `rwnd` 一起决定可发送的数据上限：

- 在途数据上限 = `min(cwnd, rwnd)`

面试高频点：

- `rwnd` 是流量控制 (保护接收端)
- `cwnd` 是拥塞控制 (保护网络)

#### 二、拥塞控制四大步骤

##### 1) 慢启动 (Slow Start)

慢启动并不“慢”，它是指数增长阶段。

- 初始 `cwnd` 较小 (现代实现常见约 10 MSS，具体与内核实现有关)
- 每收到 ACK，`cwnd` 增长
- 近似表现为每个 RTT 翻倍

退出条件：

- `cwnd` 增长到慢启动阈值 `ssthresh`
- 或者出现丢包事件

##### 2) 拥塞避免 (Congestion Avoidance)

当 `cwnd >= ssthresh` 后，进入线性增长阶段。

- 常见近似是每个 RTT 增加约 1 MSS
- 目的是更平滑地探测可用带宽，避免冲击队列

##### 3) 快速重传 (Fast Retransmit)

如果发送端收到 3 个重复 ACK (dupACK)，通常判定某段丢失，不再等待超时，立即重传该段。

价值：

- 比超时重传更快
- 显著降低恢复时延

##### 4) 快速恢复 (Fast Recovery)

在快速重传后，发送端不必把窗口降到最小，而是进入快速恢复。

以 Reno 思路为例：

1. `ssthresh = cwnd / 2`
2. `cwnd = ssthresh + 3 * MSS`
3. 每收到额外重复 ACK，`cwnd` 再增 1 MSS
4. 收到新 ACK 后，`cwnd = ssthresh`，进入拥塞避免

这体现了 TCP 的核心思想之一：检测到拥塞后快速退让，但尽量维持管道不被完全清空。

#### 三、超时重传 (RTO) 时如何处理

如果触发的是超时而不是重复 ACK，通常意味着拥塞更严重。经典处理更保守：

- `ssthresh = cwnd / 2`
- `cwnd` 降到较小值 (教材常写 1 MSS)
- 重新进入慢启动

可以总结为：

- dupACK 丢包 -> 温和退让 (快速恢复)
- RTO 超时 -> 强烈退让 (回慢启动)

#### 四、一个完整的窗口演进示例

假设：

- 初始 `cwnd = 1 MSS`
- `ssthresh = 16 MSS`

演进过程：

1. 慢启动：`1 -> 2 -> 4 -> 8 -> 16`
2. 拥塞避免：`17 -> 18 -> 19 -> ...`
3. 若在 `cwnd = 20` 时出现 3 个重复 ACK：
  - `ssthresh = 10`
  - 进入快速重传 + 快速恢复
4. 收到新 ACK 后回到拥塞避免，在 `ssthresh` 附近线性增长

这就是经典 AIMD (Additive Increase Multiplicative Decrease)：

- 加性增大：逐步试探可用带宽
- 乘性减小：拥塞后快速降载

#### 五、工程实践中的拥塞控制算法

经典面试流程通常基于 Tahoe/Reno 讲解，但生产中常见：

- CUBIC (Linux 常见默认): 适合高带宽高时延场景
- BBR: 以带宽和最小 RTT 建模，不只依赖丢包信号

面试建议表达：

- “慢启动/拥塞避免/快速重传/快速恢复”是理解 TCP 拥塞控制的基础框架
- 现代算法是在该框架上优化吞吐、时延和公平性

#### 六、可观测性与排障示例

```bash
# 观察连接级别的拥塞和重传指标 (字段随内核版本略有差异)
ss -ti dst <server_ip>

# 抓包分析重复ACK、重传、RTT 波动
tcpdump -i eth0 -nn 'tcp and host <server_ip>'
```

重点观察：

- 重传率是否上升
- RTT 是否先升后降并呈周期性
- 吞吐是否出现明显锯齿

#### 七、常见误区

- 误区 1：把拥塞控制等同于流量控制
  - 纠正：二者作用对象不同，前者面向网络，后者面向接收端
- 误区 2：慢启动一定是“慢慢增加”
  - 纠正：慢启动是指数增长，增长速度很快
- 误区 3：丢包一定是带宽不足
  - 纠正：也可能是突发队列溢出、链路抖动、设备缓存策略导致

#### 八、面试回答收束模板

可以这样收尾：

- TCP 通过 `cwnd` 动态调速，避免把网络压垮
- 经典步骤是慢启动、拥塞避免、快速重传、快速恢复
- 处理策略本质是 AIMD，在吞吐和稳定性之间做平衡
- 工程上可结合 `ss` 和抓包判断当前是否进入拥塞状态

知识扩展：
拥塞控制与滑动窗口、RTT 估计、RTO 计算、队列管理 (RED/CoDel)、以及 QUIC 强相关。QUIC 把拥塞控制放在用户态演进，很多策略与 TCP 同源，但迭代速度更快，也更易在业务侧做 A/B 优化。

### 2.7 HTTP 1.0 和 2.0 和 3.0 有什么区别？

这道题建议按“传输层依赖 -> 连接与并发模型 -> 头部与压缩 -> 队头阻塞 -> 工程落地”来答。

核心结论：

- HTTP/1.0 是早期文本协议，默认短连接，并发能力弱
- HTTP/2 在 TCP 之上引入二进制分帧和多路复用，显著提升同连接并发效率
- HTTP/3 将传输层切换为 QUIC (UDP)，把连接管理、可靠传输和拥塞控制前移到用户态，重点解决 TCP 层队头阻塞

#### 一、协议分层与传输基座差异

- HTTP/1.0: 基于 TCP，明文请求行为主 (后续常与 TLS 组合为 HTTPS)
- HTTP/2: 仍基于 TCP + TLS (业界常见 h2 over TLS)
- HTTP/3: 基于 QUIC (UDP) + TLS 1.3，握手与加密深度耦合

面试高频点：

- HTTP/2 解决的是“应用层多路复用效率”问题，但仍受 TCP 队头阻塞影响
- HTTP/3 进一步解决“传输层队头阻塞”和握手时延问题

#### 二、连接与并发模型

1. HTTP/1.0

- 默认短连接 (每个请求通常新建 TCP)
- 可通过 `Connection: keep-alive` 扩展长连接，但没有标准化良好的多路复用
- 浏览器通常通过“多开 TCP 连接”提升并发，成本是握手和拥塞窗口重复建立

2. HTTP/2

- 单 TCP 连接内支持多 Stream 并发
- 二进制分帧 (frame) + 流标识 (stream id) + 优先级调度
- 减少连接数，提升链路利用率

3. HTTP/3

- 单 QUIC 连接内支持多 Stream，并且每个流独立重传与排序
- 某个流丢包不会阻塞其他流交付，降低页面关键资源尾延迟

#### 三、头部压缩与带宽利用

- HTTP/1.0: 纯文本头部，无专门头压缩机制，冗余较大
- HTTP/2: 使用 HPACK 压缩头部，减少重复 Header 传输开销
- HTTP/3: 使用 QPACK (针对 QUIC 多路复用特性优化)，降低头阻塞风险

#### 四、队头阻塞 (Head-of-Line Blocking) 对比

1. HTTP/1.0

- 应用层基本按请求顺序处理，同连接阻塞明显

2. HTTP/2

- 应用层已多路复用，但底层仍是单 TCP 字节流
- 一旦 TCP 丢包，后续字节要等待重传，导致所有流受影响

3. HTTP/3

- 基于 QUIC 的流级别重传与交付，丢包影响被限制在单个流内
- 对弱网、移动网络和高丢包场景改善更明显

#### 五、握手与时延差异

- HTTP/1.0 / HTTP/2 (HTTPS): 典型是 TCP 握手 + TLS 握手，多次 RTT 后才传业务数据
- HTTP/3: QUIC 集成 TLS 1.3，支持 1-RTT 建连与会话恢复场景下的 0-RTT (有重放风险，需要业务侧幂等控制)

#### 六、可操作示例 (面试可直接说)

```bash
# 使用 curl 强制不同协议版本进行对比测试

# HTTP/1.0
curl -I --http1.0 https://example.com

# HTTP/2
curl -I --http2 https://example.com

# HTTP/3 (本机 curl 需支持 HTTP/3)
curl -I --http3 https://example.com

# 查看协商后的应用层协议 (ALPN)
openssl s_client -connect example.com:443 -alpn h2,http/1.1
```

观测建议：

- 对比首包时间 (TTFB)、总加载时间、丢包场景下的页面稳定性
- 在高并发短资源场景，HTTP/2/3 相比 HTTP/1.0 优势通常更明显

#### 七、一个简明对比表

| 维度           | HTTP/1.0 | HTTP/2       | HTTP/3                 |
| -------------- | -------- | ------------ | ---------------------- |
| 传输层         | TCP      | TCP          | QUIC (UDP)             |
| 数据格式       | 文本     | 二进制分帧   | 二进制分帧             |
| 多路复用       | 基本无   | 有           | 有                     |
| 头部压缩       | 无       | HPACK        | QPACK                  |
| 传输层队头阻塞 | 明显     | 仍存在       | 显著缓解               |
| 典型收益       | 实现简单 | 降低连接开销 | 弱网与移动场景时延更优 |

#### 八、常见误区

- 误区 1：HTTP/2 已经彻底解决队头阻塞
  - 纠正：HTTP/2 只解决应用层串行问题，TCP 丢包导致的传输层队头阻塞仍在
- 误区 2：HTTP/3 一定全面快于 HTTP/2
  - 纠正：低丢包、低时延内网下优势可能不显著，收益强依赖网络质量与终端支持
- 误区 3：上了 HTTP/3 就不需要 TLS
  - 纠正：QUIC 天生与 TLS 1.3 绑定，加密是协议组成部分而不是可选项

#### 九、面试回答收束模板

可以这样总结：

- HTTP/1.0 的瓶颈是连接复用和协议效率不足
- HTTP/2 用二进制分帧和多路复用提升了应用层并发能力，但仍受 TCP 丢包连带影响
- HTTP/3 借助 QUIC 把优化下沉到传输层，重点改善弱网与高丢包场景下的时延与稳定性

知识扩展：
这个问题与 TLS 1.3、ALPN 协商、CDN 边缘加速、以及 QUIC 拥塞控制算法 (如 CUBIC/BBR) 强相关。面试里把“协议机制差异”与“真实网络场景收益”关联起来，能体现从原理到工程落地的完整理解。

### 2.8 既然有 HTTP 协议，为什么还要有 RPC

这道题建议从“抽象层级差异 -> 性能与工程效率 -> 治理能力 -> 适用边界”来回答，面试表达会更完整。

核心结论：

- HTTP 是应用层通用传输协议规范，重点是“如何在网络上传内容”
- RPC (Remote Procedure Call) 是远程调用编程模型，重点是“如何像调用本地函数一样调用远端服务”
- 两者不是互斥关系，RPC 往往运行在 HTTP/2 或 HTTP/3 之上 (例如 gRPC)

#### 一、先厘清一个常见误解：HTTP 和 RPC 不是同一层问题

- HTTP 主要定义请求/响应语义、方法、状态码、头部等传输约定
- RPC 主要定义服务接口、参数序列化、调用语义、超时重试、错误模型、服务治理

一句话概括：

- HTTP 回答“消息怎么传”
- RPC 回答“服务怎么调”

因此“有 HTTP 为什么还要 RPC”的本质是：通用传输能力不等于高效的服务调用抽象。

#### 二、为什么在微服务里 RPC 往往更有优势

1. 接口契约更强，跨团队协作成本更低

- RPC 常配合 IDL (Interface Definition Language) 进行强约束定义 (如 Protobuf)
- 可以自动生成客户端/服务端桩代码，减少手写样板代码与字段对齐错误

2. 序列化更紧凑，通信效率更高

- 典型 JSON + HTTP 在可读性上有优势，但体积通常更大、解析开销更高
- 二进制协议 (如 Protobuf) 通常在带宽和 CPU 上更高效，适合服务间高频调用

3. 流式通信和双向通信能力更完善

- 现代 RPC 框架常直接支持 unary、client streaming、server streaming、bidirectional streaming
- 在日志订阅、实时推送、长连接交互等场景下比传统短请求更自然

4. 服务治理能力通常是“内建”的

- 包含服务发现、负载均衡、重试、熔断、限流、超时传播、链路追踪、元数据透传
- 如果只用裸 HTTP，这些能力通常要在业务侧或网关层自行拼装，维护复杂度更高

#### 三、一个工程化对比示例

场景：订单服务调用库存服务扣减库存。

- 纯 HTTP 方案：
  - 你需要手写 URL、方法、Header、JSON 编解码、错误码映射、重试逻辑
  - 团队之间接口字段变更时，容易出现文档和实现偏差
- RPC 方案：
  - 先定义 `DeductStock(Request) -> Response` 的接口契约
  - 通过代码生成拿到强类型客户端，像本地函数一样调用
  - 超时、重试、负载均衡、Tracing 可统一通过框架或中间件配置

可以把 RPC 理解为“在 HTTP 之上再封装一层调用语义和治理能力”。

#### 四、那是不是所有场景都该用 RPC

不是。面试中要强调“按边界选型”。

- 更适合 HTTP/REST 的场景：
  - 对外开放 API (浏览器、第三方开发者接入)
  - 追求可读性和生态兼容性 (curl、Postman、网关策略、缓存体系)
- 更适合 RPC 的场景：
  - 内部微服务高频调用
  - 对延迟、吞吐、强类型契约、统一治理有更高要求

生产中常见做法是“内 RPC、外 HTTP”：

- 内部服务网格或微服务之间用 RPC 提升效率
- 对外仍提供 HTTP/REST 或 GraphQL 接口以兼顾通用性

#### 五、常见误区

- 误区 1：RPC 比 HTTP 更底层
  - 纠正：RPC 是调用模型，HTTP 是传输协议，很多 RPC 实现本身就跑在 HTTP/2 上
- 误区 2：用了 RPC 就不需要 API 设计
  - 纠正：RPC 也需要良好的领域建模、版本兼容、错误码语义和幂等设计
- 误区 3：RPC 一定比 HTTP 快很多
  - 纠正：性能收益依赖负载类型、序列化方式、连接复用和网络质量，小规模场景差异未必显著

#### 六、面试回答收束模板

可以这样总结：

- HTTP 解决的是“通用传输”问题，RPC 解决的是“服务调用抽象与治理”问题
- 两者并非替代关系，RPC 往往建立在 HTTP/2、HTTP/3 等传输之上
- 在微服务内部，RPC 常因强契约、高性能和治理能力更有优势；在对外开放接口场景，HTTP 仍是主流

知识扩展：
这个问题与 API 网关、服务发现、服务网格 (Service Mesh)、序列化协议 (Protobuf/JSON)、以及分布式容错策略 (重试、熔断、限流、幂等) 强相关。理解“传输协议”和“调用模型”的分层关系，是做后端架构选型和性能优化的基础。

### 2.9 详细说明一下 gRPC 的原理及其应用，再说说你对 RPC 的理解

这道题建议按“RPC 本质 -> gRPC 运行原理 -> 工程能力与应用场景 -> 边界与误区”来回答，这样既有协议细节，也有架构落地视角。

核心结论：

- RPC (Remote Procedure Call) 的本质是把“网络通信”抽象成“函数调用”
- gRPC 是一套基于 HTTP/2 + Protobuf 的高性能 RPC 框架，重点解决服务间高效通信与统一治理
- RPC 不是某个具体协议，而是一种调用模型；gRPC、Dubbo、Thrift 都是不同实现路径

#### 一、先说我对 RPC 的理解 (面试可先给结论)

RPC 的核心价值不是“能远程调用”这件事本身，而是把分布式系统中反复出现的问题标准化：

- 接口契约标准化 (IDL)
- 序列化协议标准化
- 调用语义标准化 (超时、重试、错误码、幂等约束)
- 服务治理标准化 (发现、负载均衡、熔断、限流、追踪)

可以把 RPC 理解为“分布式调用操作系统”。它让业务代码聚焦领域逻辑，而不是反复手写网络细节。

#### 二、gRPC 的运行原理 (从调用链看)

1. 契约定义阶段

- 使用 Protobuf 定义 service 和 message (即 IDL)
- 通过 protoc 生成客户端 stub 和服务端 skeleton

2. 发起调用阶段

- 业务侧调用本地 stub 方法，参数被序列化为 Protobuf 二进制
- gRPC 将请求封装为 HTTP/2 stream 上的帧进行传输

3. 服务端处理阶段

- 服务端框架解码请求并路由到对应方法实现
- 执行业务逻辑后，将响应对象再次序列化并回传

4. 返回与治理阶段

- 客户端反序列化响应，得到强类型结果
- 同时可统一挂载拦截器 (interceptor) 处理鉴权、日志、Tracing、重试、超时等横切逻辑

一句话总结链路：

IDL 定义 -> 代码生成 -> 序列化传输 -> 服务执行 -> 反序列化返回。

#### 三、gRPC 为什么常被认为“快且稳”

1. HTTP/2 多路复用

- 单连接可并发多个 stream，减少连接建立和队头等待开销

2. Protobuf 二进制编码

- 相比 JSON 通常更小、更快，降低网络带宽和序列化 CPU 成本

3. 长连接与流式能力

- 天然支持 unary、server streaming、client streaming、bidirectional streaming
- 在实时推送、日志订阅、Agent 状态回传等场景更自然

4. 明确的错误模型与元数据机制

- 使用状态码和 metadata 统一错误语义与上下文透传，便于治理和排障

#### 四、gRPC 的典型应用场景

- 微服务内部高频调用 (订单、库存、支付之间)
- 跨语言服务协作 (Go/Java/Python 通过同一 proto 协作)
- 低延迟或高吞吐内部链路
- 需要流式传输的场景 (实时风控、日志流、监控流)

常见架构实践：

- 内部服务间使用 gRPC
- 对外开放接口继续提供 HTTP/REST (便于浏览器和第三方生态接入)

#### 五、一个简化 proto 示例 (用于说明契约驱动)

```proto
syntax = "proto3";

package stock;

service StockService {
  // 扣减库存
  rpc Deduct(DeductRequest) returns (DeductResponse);
}

message DeductRequest {
  string sku_id = 1;
  int32  count  = 2;
}

message DeductResponse {
  bool success = 1;
  string reason = 2;
}
```

这个定义生成的客户端代码，调用形态接近本地函数，但底层仍是网络传输。

#### 六、常见误区与边界

- 误区 1：gRPC 可以完全替代 HTTP/REST
  - 纠正：对外开放 API、浏览器直连、生态工具兼容上，HTTP/REST 仍有明显优势
- 误区 2：用了 gRPC 就不需要做接口版本管理
  - 纠正：仍需遵守向后兼容规则 (字段编号不可复用、谨慎删除字段、保留 reserved)
- 误区 3：RPC 一定比 REST 快很多
  - 纠正：性能收益取决于负载特征、消息大小、连接复用和治理策略，不是绝对结论

边界提醒：

- gRPC 对浏览器原生支持有限，通常需要 gRPC-Web 或网关转译
- 在简单 CRUD 外部 API 场景，REST 的可读性和调试便利性可能更优

#### 七、面试回答收束模板

可以这样总结：

- RPC 是一种把远程调用抽象为本地函数调用的分布式通信模型
- gRPC 通过 HTTP/2 + Protobuf + 代码生成，把性能、类型安全和治理能力做了工程化整合
- 在内部微服务场景，gRPC 往往更高效；在开放生态和浏览器友好场景，HTTP/REST 仍是主流

知识扩展：
这个问题与服务注册发现、负载均衡策略 (客户端负载均衡或服务网格)、链路追踪 (OpenTelemetry)、以及分布式容错设计 (超时、重试预算、熔断、幂等) 强相关。理解 RPC 的本质后，再看 Dubbo、Thrift、QUIC RPC 等框架，会更容易把握它们的共性和差异。
