## 1. Redis 集群的实现原理是什么

### Redis 集群的核心特征

- 无中心架构：所有节点地位平等，每个节点都存储完整的集群元数据（槽位映射、节点状态、集群配置），无单点故障风险。
- 数据分片存储：采用 16384 个哈希槽（Hash Slot）实现数据分片，每个主节点负责一部分哈希槽的读写，实现水平扩容。
- 主从高可用：每个主节点可配置 1-N 个从节点，从节点实时同步主节点数据，主节点故障时自动触发故障转移，从节点晋升为主节点，保证集群可用性。
- 去中心化通信：节点间通过 Gossip 协议交换集群元数据，自动完成节点发现、故障状态同步、槽位映射更新，无需第三方组件协调。
- 智能请求路由：支持客户端缓存槽位映射表，直接路由请求到对应节点，同时提供重定向机制保证请求正确性。

### 核心数据分片原理

哈希槽是 Redis Cluster 最核心的设计，决定了数据分布规则、扩容缩容的平滑性。

#### 哈希槽的核心规则

- 集群预分配16384 个固定的哈希槽，编号范围 0-16383，是 Redis Cluster 数据分片的最小单位。
- 每个 key 通过固定算法映射到一个哈希槽：CRC16(key) mod 16384，其中 CRC16 是 16 位循环冗余校验算法，输出值范围 0-65535，取模后得到 0-16383 的槽位号。
- 每个主节点负责管理一段连续 / 非连续的哈希槽，集群正常对外服务的必要前提是 16384 个哈希槽 100% 被分配，无槽位缺失。

#### 为什么哈希槽数量固定为 16384？

- 心跳消息体极致优化：节点间心跳消息会携带自身的槽位信息，采用位图（bitmap）存储，16384 个槽位的位图大小为 16384 / 8 = 2KB，大幅降低网络传输开销；若采用 65536 个槽，位图大小为 8KB，会显著增加节点间通信的带宽压力。
- 集群规模适配：Redis 官方建议集群最大节点数不超过 1000 个，16384 个槽位足够均匀分配给 1000 个节点，满足扩容缩容的粒度需求，更多槽位无实际业务价值。
- 哈希算法适配：CRC16 算法输出为 16 位整数，mod 16384 可保证哈希值均匀分布，无明显哈希倾斜问题。

#### 哈希槽和一致性哈希对比

| 对比维度       | 哈希槽                                                           | 一致性哈希                                                   |
| -------------- | ---------------------------------------------------------------- | ------------------------------------------------------------ |
| 数据迁移粒度   | 以哈希槽为单位，仅需迁移变更节点负责的槽位数据，迁移范围精准可控 | 节点增减时，需重新映射环上相邻节点的大量数据，迁移范围不可控 |
| 元数据管理     | 槽位映射用位图存储，体积小，节点间同步开销极低                   | 需维护完整的节点哈希环，元数据体积大，同步开销高             |
| 扩容缩容平滑性 | 可逐槽迁移数据，业务无感知，支持精细化扩缩容                     | 节点变更会引发大范围数据重定向，易出现业务抖动               |
| 元数据一致性   | 槽位归属唯一，无哈希冲突，元数据收敛速度快                       | 哈希环变更易出现临时数据不一致，收敛速度慢                   |

#### 多 key 操作支持：哈希标签 (Hash Tag)

默认情况下，Redis Cluster不支持跨槽位的多 key 操作（如 MSET/MGET、事务、Lua 脚本），会直接报错。通过哈希标签可强制多个 key 映射到同一个槽位，解决该问题。

核心规则：若 key 包含 {xxx} 格式的标签，仅对 {} 内的字符串计算 CRC16 哈希值，{} 外的内容不参与哈希计算。

### 集群节点通信机制：Gossip 协议

Redis Cluster 采用无中心设计，节点间通过Gossip 流言协议完成元数据同步，实现节点发现、状态同步、故障信息广播，最终所有节点达成一致的集群视图。

#### 节点通信基础

- 双端口设计：每个 Redis 集群节点开放两个端口，生产环境必须同时放行
  - 客户端服务端口：默认 6379，对外提供客户端读写服务；
  - 集群总线端口：固定为客户端端口 + 10000（如 16379），采用二进制私有协议，仅用于节点间内部通信。
- 通信模式：节点间两两建立 TCP 长连接，定期发送心跳消息，无需第三方中间件。

#### Gossip 核心消息类型与作用

| 消息类型 | 核心作用                                               | 触发场景                                        |
| -------- | ------------------------------------------------------ | ----------------------------------------------- |
| MEET     | 通知新节点加入集群，握手建立通信                       | 执行 CLUSTER MEET ip port 命令时                |
| PING     | 心跳检测，携带自身节点状态与部分集群元数据             | 节点每秒随机选择多个节点发送，检测节点存活状态  |
| PONG     | PING/MEET 消息的响应，返回自身元数据，用于广播节点状态 | 收到 PING/MEET 消息后，或主动广播自身状态变更时 |
| FAIL     | 广播节点客观下线状态，通知所有节点更新故障节点信息     | 节点被标记为客观下线时                          |
| PUBLISH  | 跨节点广播消息，实现集群级别的 Pub/Sub 功能            | 客户端执行 PUBLISH 命令时                       |

#### Gossip 协议工作流程

1. 新节点加入：客户端执行CLUSTER MEET <新节点ip> <新节点端口>，现有节点向新节点发送 MEET 消息，新节点响应 PONG，完成握手。
2. 元数据扩散：握手完成后，节点间通过 PING/PONG 消息不断交换自身掌握的集群元数据，包括所有节点的 IP、端口、ID、角色、负责槽位、状态等。
3. 集群视图收敛：经过多次消息交换，最终集群内所有节点都获得完整、一致的集群元数据，完成节点发现与集群组建。

### 客户端请求路由与重定向机制

Redis Cluster 通过槽位映射实现请求路由，核心分为智能客户端直连模式、节点重定向模式两种，保证客户端请求能精准到达负责对应槽位的主节点。

- 智能客户端直连模式（生产环境推荐）
  - 核心逻辑：客户端启动时，主动向集群任意节点拉取全量的槽位 - 节点映射表，缓存到本地内存。
  - 请求流程：客户端执行命令时，先通过CRC16(key) mod 16384计算 key 对应的槽位，再通过本地缓存的映射表找到对应的主节点，直接向该节点发送请求，性能最优，无额外网络开销。
- 节点重定向机制：当客户端请求的 key 对应的槽位，不在当前接收请求的节点负责范围内时，节点会返回重定向错误，引导客户端访问正确的节点。核心分为 MOVED 永久重定向、ASK 临时重定向两种
  - MOVED 永久重定向
    - 触发场景：槽位的归属已经永久变更，当前节点不再负责该槽位，且集群所有节点已同步该元数据。
    - 响应格式：MOVED <槽位号> <目标节点ip>:<端口>
    - 客户端处理逻辑：收到 MOVED 响应后，更新本地缓存的槽位映射表，然后向目标节点重新发送请求，后续相同槽位的请求直接路由到新节点。
  - ASK 临时重定向
    - 触发场景：槽位正在从源节点迁移到目标节点，部分 key 已迁移到目标节点，部分仍在源节点，槽位归属尚未最终变更。
    - 处理流程
      - 客户端向源节点发送请求，源节点发现 key 对应的槽位正在迁移，且当前节点不存在该 key，返回ASK <槽位号> <目标节点ip>:<端口>响应。
      - 客户端收到 ASK 响应后，不会更新本地槽位映射缓存，先向目标节点发送ASKING命令，再发送原始请求。
      - 目标节点收到 ASKING 命令后，会临时允许客户端访问正在导入的槽位，执行请求并返回结果。
    - 核心设计目的：保证槽位迁移过程中，客户端的请求不会丢失，同时避免迁移过程中客户端缓存被临时变更污染。

### 高可用与自动故障转移机制

Redis Cluster 通过主从复制 + 去中心化故障检测 + 自动选举机制，实现集群的高可用，主节点故障时可自动完成故障转移，业务无感知。

#### 集群主从复制基础

- 每个主节点可配置 1-N 个从节点，从节点通过 Redis 原生主从复制机制，实时同步主节点的全量数据与增量命令，与单机主从复制逻辑完全一致。
- 默认配置下，从节点为只读模式，仅用于数据备份与故障转移替补，不对外提供读写服务，需客户端手动执行 READONLY 命令后，才会接受读请求。

#### 故障检测机制：主观下线（PFail）与客观下线（Fail）

集群通过两步检测机制，精准判断节点是否故障，避免网络抖动导致的误判。

- 主观下线（PFail，Possible Fail）
  - 触发条件：节点 A 定期向节点 B 发送 PING 心跳消息，若超过cluster-node-timeout（默认 15000 毫秒）未收到节点 B 的 PONG 响应，节点 A 会将节点 B 标记为 PFail 主观下线状态。
  - 核心说明：主观下线是单个节点的局部判断，不代表集群整体认可节点故障。
- 客观下线（Fail）
  - 触发条件：节点 A 通过 Gossip 消息，将节点 B 的 PFail 状态广播给集群其他节点；当集群中超过半数的主节点，都标记节点 B 为 PFail 状态时，节点 B 会被正式标记为 Fail 客观下线状态。
  - 核心说明：客观下线是集群的集体判断，只有主节点拥有投票权，从节点不参与主观下线的投票统计，这是面试高频易错点。
  - 关键配置：cluster-node-timeout是故障检测的核心参数，生产环境建议设置为 15000-30000 毫秒，过小易引发误判，过大导致故障恢复时间过长。
- 自动故障转移完整流程：当主节点被标记为客观下线后，集群自动触发故障转移
  1. 从节点资格审核：下线主节点的所有从节点，会检查自身的复制状态，只有满足主从断开时间 < cluster-node-timeout * cluster-slave-validity-factor的从节点，才有资格参与选举，保证数据足够完整，避免数据丢失。
  2. 选举发起：资格达标的从节点，会将自身的config epoch（集群逻辑时钟，单调递增）加 1，然后向集群中所有主节点发起投票请求，申请晋升为新的主节点。
  3. 投票仲裁：主节点收到投票请求后，会校验请求的合法性（配置纪元、从节点所属主节点等），每个主节点在同一个配置纪元内，只能给一个从节点投票，优先投票给复制偏移量最大（数据最完整）的从节点。
  4. 新主节点当选：发起选举的从节点，若获得超过半数的主节点投票，则成功当选为新的主节点。
  5. 槽位接管与广播：新主节点将原下线主节点负责的所有哈希槽，全部转移到自己名下，然后通过 PONG 消息向集群广播自己的新主节点身份，所有节点更新槽位映射表。
  6. 集群状态恢复：客户端更新槽位映射后，请求自动路由到新主节点，集群恢复正常服务；原下线的主节点若恢复上线，会自动变为新主节点的从节点，同步数据。

#### 集群扩缩容核心原理

Redis Cluster 的扩缩容核心是哈希槽的迁移，以槽为单位进行数据迁移，保证扩缩容过程中业务无感知，集群持续可用。

- 集群扩容流程（新增主节点）
  - 准备新节点：启动新的 Redis 节点，开启集群模式配置，加入集群网络。
  - 节点加入集群：在集群任意节点执行CLUSTER MEET <新节点ip> <新节点端口>，新节点通过 Gossip 协议完成集群元数据同步。
  - 槽位分配与迁移：从现有主节点中，迁移部分哈希槽到新主节点。
  - 槽位归属更新：槽位所有 key 迁移完成后，向集群所有节点广播槽位的新归属节点，更新槽位映射表。
  - 从节点配置：为新主节点添加从节点，保证高可用。
- 集群缩容流程（下线主节点）
  - 槽位迁出：将待下线主节点负责的所有哈希槽，全部迁移到集群其他主节点。
  - 节点移除：待所有槽位迁移完成后，在集群所有节点执行CLUSTER FORGET <待下线节点id>，移除节点信息。
  - 节点下线：关闭待下线的 Redis 进程，完成缩容。

## 2. Gossip 协议的原理及应用

### Gossip 协议的核心特征

- 去中心化传播：不依赖中心节点维护全局状态，节点之间通过随机通信逐步扩散信息，天然具备高可用特性。
- 最终一致性：单次消息交换不要求所有节点立即一致，但经过多轮传播后，集群状态会逐步收敛到一致视图。
- 轻量级扩散：每个节点只需要周期性与少量随机节点通信，就能把本地信息传播到整个网络，通信成本低，扩展性强。
- 抗局部故障：即使部分节点不可达，剩余节点仍可继续传播状态，系统不会因为单点失联而整体停摆。
- 容忍短暂不一致：Gossip 的设计目标不是强一致，而是通过冗余传播和概率收敛实现高可用元数据同步。

### Gossip 协议的核心原理

Gossip 协议模拟人类社会中的流言传播机制。节点只需把自己知道的信息告诉少量邻居，邻居再继续扩散，最终整个网络都会知道同一件事。

#### 基本传播模型

- 每个节点维护一份本地状态视图，包含自身状态和从其他节点学到的集群元数据。
- 节点按照固定周期随机选择若干目标节点发送消息，把本地最新状态带给对方。
- 接收方在回复中也会带上自己知道的新信息，双方通过一次交互完成信息互补。
- 重复多轮之后，新状态会像流言一样在集群中逐步扩散，最终实现全网收敛。

#### 典型传播过程

1. 节点 A 检测到节点 B 状态变化，例如疑似下线或槽位归属变化。
2. 节点 A 在后续 Gossip 消息里把这个变化告诉节点 C 和节点 D。
3. 节点 C 和节点 D 再把相同状态继续转发给更多节点。
4. 经过多轮传播，所有节点都获得相同的状态结论，集群视图完成收敛。

#### 为什么 Gossip 适合分布式系统

- 扩展性好：每轮只和少量节点通信，节点数量增加时通信压力不会线性爆炸。
- 无中心瓶颈：不需要中心协调器维护拓扑和元数据，避免单点故障和中心性能瓶颈。
- 容错性强：局部丢包、延迟或节点宕机只会拖慢收敛，不会直接破坏整体协议。
- 工程实现简单：节点只要维护本地状态和重传逻辑，就能实现大规模状态同步。

### Gossip 协议在 Redis Cluster 中的应用

Redis Cluster 并不是直接把 Gossip 当作业务数据广播协议使用，而是用它来传播集群元数据和节点状态，核心目标是让所有节点最终看到一致的集群拓扑。

#### Redis 中 Gossip 传播的内容

- 节点身份信息：节点 ID、IP、端口、角色 (master 或 slave)。
- 集群拓扑信息：每个主节点负责的哈希槽范围，以及槽位迁移状态。
- 节点健康状态：是否疑似下线 (PFail)、是否客观下线 (Fail)、复制关系是否正常。
- 配置纪元信息：用于故障转移和元数据版本管理，避免过期信息覆盖新信息。

#### Redis 中 Gossip 的通信载体

- Redis Cluster 使用集群总线进行节点间通信，承载 PING、PONG、MEET、FAIL 等内部消息。
- 节点会在心跳消息中夹带部分集群元数据，而不是每次都发送全量状态，降低通信开销。
- 消息采用增量传播方式，优先传播变化信息，例如新节点加入、槽位迁移、节点故障等。

#### Redis 中 Gossip 的核心作用

- 节点发现：新节点加入集群后，通过 Gossip 很快学到其他节点信息，完成拓扑初始化。
- 状态同步：节点健康状态、复制信息、槽位归属等内容通过 Gossip 不断扩散，形成统一视图。
- 故障广播：当某个节点被判定为 Fail 后，其他节点会通过 Gossip 快速获知，尽快切换路由。
- 元数据收敛：集群扩缩容、故障转移、槽位迁移完成后，所有节点最终会通过 Gossip 达成一致。

### Redis Cluster 中 Gossip 的工作机制

#### 消息交换策略

- 节点不会轮询所有其他节点，而是随机抽样部分节点进行通信。
- 这种方式可以在较低带宽下，把状态变化持续传播到全网。
- 如果某条信息没有被所有节点一次性收到，后续心跳仍会继续携带它，直到收敛完成。

#### 信息扩散与老化机制

- 节点会优先传播最近发生变化的元数据，确保最新状态更快扩散。
- 过时状态会随着新一轮 Gossip 被覆盖，避免旧信息长期残留。
- 这种设计本质上是以概率方式保证收敛，而不是依赖严格的中心仲裁。

#### 与故障检测的关系

- 节点先基于本地超时判断得到 PFail，再通过 Gossip 把该判断扩散出去。
- 当足够多主节点都看到同一个故障信号后，节点才会被升级为 Fail。
- 所以 Gossip 不只是广播管道，也是 Redis Cluster 故障状态最终一致的重要基础。

### Gossip 协议与其他分布式协议的对比

| 对比维度   | Gossip 协议                      | 传统中心化协议                 |
| ---------- | -------------------------------- | ------------------------------ |
| 架构形态   | 去中心化，节点自治               | 依赖中心协调器                 |
| 扩展能力   | 节点增多时仍能平滑扩展           | 中心节点容易成为瓶颈           |
| 一致性模型 | 最终一致性                       | 可实现更强的一致性，但代价更高 |
| 容错能力   | 局部故障不影响整体传播           | 中心故障影响面大               |
| 工程复杂度 | 通信模式简单，状态收敛靠概率传播 | 需要额外的协调、选主或仲裁机制 |

### 典型应用场景

- 分布式数据库集群：用于传播节点状态、分片信息和故障信息，例如 Redis Cluster。
- 服务发现系统：用于把服务实例的上线、下线和健康状态扩散给其他节点。
- 监控与成员管理：用于维护大规模节点的 membership 列表和活性探测。
- 多副本同步辅助：作为补充机制帮助传播副本状态或版本变更信息。

### 面试回答要点

- Gossip 的本质是通过随机、周期性、双向的状态交换，实现分布式系统的最终一致。
- 它适合传播元数据和成员状态，不适合直接承载强一致业务写入。
- Redis Cluster 使用 Gossip 解决的是节点发现、状态同步、故障广播和拓扑收敛问题，而不是数据读写本身。
- Gossip 的优势是无中心、可扩展、容错强，代价是短时间内可能存在视图不一致。

### 知识扩展

- 与 Redis Cluster 的关系：Gossip 是 Redis Cluster 节点发现和元数据同步的基础。
- 与 PFail 和 Fail 的关系：节点故障先本地感知，再通过 Gossip 扩散并收敛为集群共识。
- 与哈希槽迁移的关系：槽位迁移完成后，新的槽位归属需要依赖 Gossip 广播到全网。
- 与主从复制的关系：主从复制解决数据同步，Gossip 解决集群成员与拓扑信息同步。

## 3. Redis 中常见的数据类型有哪些？

### Redis 数据类型的核心特征

- 面向场景建模：Redis 不是只有简单的 Key-Value 存储，而是提供多种抽象结构，便于直接表达计数、队列、排行榜、标签、地理位置、位运算等业务模型。
- 统一访问入口：所有类型都通过 key 访问，但不同类型拥有独立的命令族和时间复杂度特征。
- 底层自动编码：同一种逻辑类型会根据元素数量、元素长度动态选择底层编码 (encoding)，在内存占用和操作性能之间平衡。
- 高性能基因：绝大多数操作在 O(1) 或 O(logN) 复杂度内完成，且单线程模型避免了复杂锁竞争。
- 组合能力强：可以通过 Lua、事务、Pipeline 和过期策略把多种类型组合成完整业务能力。

### 常见数据类型总览

Redis 面试里高频类型包括 String、List、Hash、Set、ZSet，以及面向特定场景的 Bitmap、HyperLogLog、GEO、Stream。

#### 类型速览表

| 数据类型          | 典型业务场景                     | 常用命令                              | 典型复杂度                        |
| ----------------- | -------------------------------- | ------------------------------------- | --------------------------------- |
| String            | 缓存对象、计数器、分布式锁 token | GET、SET、INCR、MGET                  | O(1)                              |
| List              | 消息队列、时间线、任务缓冲       | LPUSH、RPOP、BLPOP、LRANGE            | 头尾 O(1)，按索引 O(N)            |
| Hash              | 用户资料、配置对象、字段级更新   | HSET、HGET、HMGET、HINCRBY            | O(1)                              |
| Set               | 标签去重、共同好友、抽奖池       | SADD、SMEMBERS、SINTER、SUNION        | 单操作 O(1)，集合运算与元素数相关 |
| Sorted Set (ZSet) | 排行榜、延迟队列、权重排序       | ZADD、ZRANGE、ZREVRANK、ZSCORE        | O(logN)                           |
| Bitmap            | 签到、活跃统计、状态位存储       | SETBIT、GETBIT、BITCOUNT、BITOP       | 位操作 O(1)，统计与位图长度相关   |
| HyperLogLog       | UV 统计、近似基数统计            | PFADD、PFCOUNT、PFMERGE               | 近似统计，常量级内存              |
| GEO               | 附近的人、门店检索、距离计算     | GEOADD、GEORADIUS、GEOSEARCH、GEODIST | 基于有序集合索引                  |
| Stream            | 消息流、消费组、可回溯队列       | XADD、XREAD、XGROUP、XACK             | 追加近 O(1)，读取与范围相关       |

### 1) String

#### 核心原理

- String 是 Redis 最基础类型，值可以是字符串、数字或二进制内容。
- 底层依赖 SDS (Simple Dynamic String)，支持 O(1) 长度获取、预分配和惰性释放，避免 C 字符串频繁扩容问题。
- INCR、DECR 等原子计数操作本质是服务端单线程串行执行，无需应用层加锁。

#### 常见应用

- 缓存 JSON 序列化对象。
- 计数器 (访问量、点赞数、限流令牌桶计数)。
- 分布式锁 (SET key value NX EX seconds)。

#### 命令示例

```bash
# 缓存对象并设置过期
SET user:1001 '{"name":"alice","level":3}' EX 3600

# 原子计数
INCR page:view:home

# 分布式锁 (加锁)
SET lock:order:9001 req-uuid-001 NX PX 30000
```

#### Lua 解锁示例 (防误删)

```lua
-- 只有 value 匹配时才删除，避免误删其他客户端锁
if redis.call('GET', KEYS[1]) == ARGV[1] then
  return redis.call('DEL', KEYS[1])
else
  return 0
end
```

### 2) List

#### 核心原理

- List 是按插入顺序维护的双端链式逻辑结构，适合头尾插入和弹出。
- 新版本 Redis 在实现上采用 quicklist (ziplist/listpack + linked list 组合思想)，兼顾内存和遍历性能。
- 阻塞弹出命令 (BLPOP/BRPOP) 可直接构建轻量消息队列。

#### 常见应用

- 简单异步任务队列。
- Feed 流时间线 (最新内容头插)。
- 固定长度日志缓冲 (LPUSH + LTRIM)。

#### 命令示例

```bash
# 生产者写入任务
LPUSH queue:email '{"to":"a@example.com","tpl":"welcome"}'

# 消费者阻塞消费
BRPOP queue:email 10

# 保留最近 100 条日志
LPUSH log:api '/order/create'
LTRIM log:api 0 99
```

### 3) Hash

#### 核心原理

- Hash 是 field-value 的映射结构，适合存储对象型数据。
- 小对象时常用紧凑编码 (listpack)，字段数量或值长度增长后会升级为哈希表编码。
- 相比把整个对象放入 String，Hash 支持字段级读写，降低网络与序列化开销。

#### 常见应用

- 用户信息、商品信息、配置中心对象。
- 字段级计数 (HINCRBY)。

#### 命令示例

```bash
HSET user:1001 name alice age 28 city shanghai
HGET user:1001 name
HMGET user:1001 name city
HINCRBY user:1001 login_count 1
```

### 4) Set

#### 核心原理

- Set 是无序且元素唯一的集合。
- 适合做去重与集合运算 (交集、并集、差集)。
- 底层会根据元素类型和数量在 intset 与 hashtable 等编码之间切换。

#### 常见应用

- 用户标签去重。
- 共同关注、共同好友。
- 黑名单、白名单。

#### 命令示例

```bash
SADD tag:article:1 redis cache distributed
SADD tag:article:2 redis interview

# 交集 (两个文章共同标签)
SINTER tag:article:1 tag:article:2
```

### 5) Sorted Set (ZSet)

#### 核心原理

- ZSet 在 Set 唯一性的基础上为每个元素增加 score。
- 底层通常由 hash table + skiplist 组合实现：hash table 提供成员到分值的快速定位，skiplist 负责按分值有序查询与范围操作。
- 因为支持有序范围检索，ZSet 是排行榜类场景首选。

#### 常见应用

- 实时排行榜 (积分榜、热度榜)。
- 延迟队列 (score 存时间戳)。
- 限流窗口统计。

#### 命令示例

```bash
ZADD rank:game 1200 user:1 980 user:2 1350 user:3

# Top 2
ZREVRANGE rank:game 0 1 WITHSCORES

# 查询用户排名
ZREVRANK rank:game user:1
```

### 6) Bitmap

#### 核心原理

- Bitmap 本质不是独立数据结构，而是基于 String 的位操作能力。
- 一个 bit 表示一个状态，极其节省空间，适用于海量布尔状态场景。

#### 常见应用

- 用户签到 (第 n 天是否签到)。
- 在线状态、活跃状态压缩存储。

#### 命令示例

```bash
# 用户 1001 在第 5 天签到
SETBIT sign:2026-04:1001 5 1

# 统计当月签到天数
BITCOUNT sign:2026-04:1001
```

### 7) HyperLogLog

#### 核心原理

- HyperLogLog 用概率算法估算去重基数 (cardinality)，标准误差约 0.81%。
- 核心优势是固定且极小的内存开销 (每个 key 约 12 KB 量级)。
- 适合 UV、独立访客统计，不适合要求精确去重明细的场景。

#### 命令示例

```bash
PFADD uv:2026-04-03 user:1 user:2 user:3
PFCOUNT uv:2026-04-03
```

### 8) GEO

#### 核心原理

- GEO 底层依赖 ZSet，把地理位置编码为 geohash 后按 score 存储。
- 支持附近检索、距离计算、按距离排序。

#### 命令示例

```bash
GEOADD store:city 121.4737 31.2304 shanghai_store 116.4074 39.9042 beijing_store
GEODIST store:city shanghai_store beijing_store km
GEOSEARCH store:city FROMLONLAT 121.47 31.23 BYRADIUS 10 km WITHDIST
```

### 9) Stream

#### 核心原理

- Stream 是 Redis 5.0 引入的消息流结构，支持消息持久化、消费组、ACK、待处理列表 (PEL)。
- 相比 List，Stream 对多消费者协作和消息可回溯更友好。

#### 常见应用

- 订单事件流。
- 异步任务分发与失败重试。

#### 命令示例

```bash
# 生产消息
XADD stream:order * order_id 9001 status created

# 创建消费组
XGROUP CREATE stream:order group_order 0 MKSTREAM

# 消费组读取
XREADGROUP GROUP group_order consumer_1 COUNT 10 BLOCK 2000 STREAMS stream:order >

# 处理完成 ACK
XACK stream:order group_order 1712200000000-0
```

### 生产配置与选型建议

```conf
# redis.conf 关键配置示例 (按业务压测结果调整)
maxmemory 4gb
maxmemory-policy allkeys-lru
appendonly yes
appendfsync everysec
hash-max-listpack-entries 512
hash-max-listpack-value 64
zset-max-listpack-entries 128
zset-max-listpack-value 64
```

- 以读多写少缓存为主：优先 String + 过期策略。
- 以对象字段局部更新为主：优先 Hash，避免整对象反序列化。
- 以排序检索为主：优先 ZSet。
- 以可靠消息处理为主：优先 Stream + 消费组，而不是简单 List 队列。

### 客户端交互代码示例 (Go)

```go
package main

import (
  "context"
  "fmt"

  "github.com/redis/go-redis/v9"
)

func main() {
  ctx := context.Background()
  rdb := redis.NewClient(&redis.Options{Addr: "127.0.0.1:6379"})

  // String
  _ = rdb.Set(ctx, "counter:pv", 1, 0).Err()
  _ = rdb.Incr(ctx, "counter:pv").Err()

  // Hash
  _ = rdb.HSet(ctx, "user:1001", "name", "alice", "age", 28).Err()

  // ZSet
  _ = rdb.ZAdd(ctx, "rank:game", redis.Z{Score: 100, Member: "user:1001"}).Err()

  rank, _ := rdb.ZRevRank(ctx, "rank:game", "user:1001").Result()
  fmt.Println("rank:", rank)
}
```

### 面试回答要点

- String、Hash、List、Set、ZSet 是最核心的五大类型，面试回答必须带上典型场景和复杂度。
- Bitmap、HyperLogLog、GEO、Stream 是高阶类型，体现你对 Redis 场景化建模能力。
- 不要只背命令，要说明底层编码 (SDS、quicklist、skiplist、listpack) 与性能权衡。
- 类型选型本质是空间、时间复杂度、业务语义三者平衡。

### 知识扩展

- 与内存淘汰策略的关系：不同类型 value 的体积和访问模式会影响 LRU/LFU 淘汰效果。
- 与持久化的关系：AOF/RDB 会影响不同类型写入吞吐与恢复时间。
- 与事务和 Lua 的关系：可通过 Lua 把多类型操作封装为原子步骤，避免并发竞态。
- 与 Redis Cluster 的关系：多 key 操作涉及跨槽位约束，类型设计需结合 hash tag 规划 key。
