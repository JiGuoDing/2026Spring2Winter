## 1. Kubernetes 基础面试题

### 1.1 描述如何在 Kubernetes 中创建一个 Pod，并给出示例配置文件

这道题建议从四个层次回答：资源定义、提交流程、控制面与节点面执行链路、以及可观测与排障。

核心结论：

- 在 Kubernetes 中创建 Pod 的本质是向 API Server 提交一个声明式对象 (Pod YAML)
- 后续由控制平面和节点组件协同把“期望状态”收敛为“实际运行状态”
- 面试里不要只说 `kubectl apply -f`，要讲清 Scheduler、Kubelet、CRI、CNI 的协作路径

#### 一、创建 Pod 的标准步骤

1. 编写 Pod 资源清单 (YAML)
2. 使用 `kubectl apply -f` 提交到 API Server
3. API Server 完成认证、鉴权、准入控制后写入 etcd
4. Scheduler 监听到 Pending Pod，为其选择 Node 并回写 `spec.nodeName`
5. 目标节点上的 Kubelet 通过 CRI (containerd/runc) 拉镜像并启动容器
6. Kubelet 调用 CNI 插件完成网络配置 (veth、网桥/overlay、IP 分配)
7. Pod 进入 Running，`kubectl get pod -o wide` 可看到节点与 IP

#### 二、可落地示例 (Pod YAML)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-demo
  namespace: default
  labels:
    app: nginx-demo
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      imagePullPolicy: IfNotPresent
      ports:
        - containerPort: 80
          name: http
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
  restartPolicy: Always
```

说明：

- `requests/limits` 会影响调度和 cgroups 配额，是面试高频点
- 单独创建 Pod 常用于学习与调试，生产通常用 Deployment 管理副本与滚动升级

#### 三、常用命令行操作 (创建 + 验证 + 调试)

```bash
# 1) 创建 Pod
kubectl apply -f nginx-pod.yaml

# 2) 查看调度和运行状态
kubectl get pod nginx-demo -o wide

# 3) 查看事件 (调度失败、拉镜像失败等最先看这里)
kubectl describe pod nginx-demo

# 4) 进入容器验证进程
kubectl exec -it nginx-demo -- sh

# 5) 查看日志
kubectl logs nginx-demo
```

#### 四、底层机制 (面试加分点)

1. 为什么 Pod 会先 Pending

- Pod 刚写入 etcd 时还没有绑定节点，直到 Scheduler 完成打分与绑定才会进入下一阶段

2. Kubelet 如何创建容器

- Kubelet 不直接操作 runc，而是调用 CRI Runtime (常见 containerd)
- containerd 再调用 runc 按 OCI 规范创建容器 namespace 与 cgroups

3. 网络是何时完成的

- 容器进程创建前后，Kubelet 通过 CNI `ADD` 动作为 Pod 创建网络命名空间接口
- 常见实现会创建 veth pair 并挂到主机侧网桥或 overlay 设备

4. 为什么 Pod IP 在重建后会变

- Pod 是短生命周期对象，删除重建后通常会获得新 IP
- 稳定访问应通过 Service 提供虚拟 IP 与负载均衡

#### 五、常见问题与排障路径

- `ImagePullBackOff`:
  - 原因常见于镜像名错误、仓库鉴权失败、节点无法访问镜像仓库
  - 先查 `kubectl describe pod` 的 Events，再查节点网络与镜像拉取凭据
- `Pending` 长时间不调度:
  - 常见于资源不足、NodeSelector/Taint 不匹配、PVC 未绑定
  - 重点看调度事件中的 predicate/filter 失败原因
- `CrashLoopBackOff`:
  - 容器启动后反复退出，常见于启动命令错误、配置缺失、探针配置不合理
  - 结合 `kubectl logs --previous` 与 `describe` 事件定位

#### 六、面试收束模板

可以这样总结：

- 创建 Pod 是一次声明式提交，真正落地依赖 API Server、Scheduler、Kubelet、CRI、CNI 的控制闭环
- YAML 负责表达期望状态，控制器与节点组件负责持续逼近期望
- 线上问题优先看事件流与状态流转，而不是只看容器日志

知识扩展：
Pod 创建流程与 Deployment/ReplicaSet 强相关，因为生产环境通常不用裸 Pod 管理生命周期；与 Service 强相关，因为 Pod IP 不稳定需要稳定服务发现；与 NetworkPolicy 强相关，因为连通性故障常出在 CNI 策略与策略生效链路。

### 1.2 解释什么是 Kubernetes，并描述其主要组件及其作用

这道题不要只回答“容器编排平台”。面试里建议按四层结构展开：定义与设计目标、控制平面组件、节点组件、以及一次完整请求如何在组件间流转。

核心结论：

- Kubernetes (K8s) 是一个面向容器化应用的声明式集群操作系统
- 它通过 API + 控制器模式，把用户声明的期望状态持续收敛为实际状态
- 其本质不是“启动容器工具”，而是一套分布式控制平面 + 节点执行平面的协同系统

#### 一、什么是 Kubernetes (本质与设计思想)

Kubernetes 最核心的思想有三点：

1. 声明式 API
   用户提交的是“我要什么” (Desired State)，不是“每一步怎么做”。
2. 控制器闭环
   Controller 通过 Watch 监听对象变化，持续 Reconcile (对齐期望与现实)。
3. 最终一致性
   分布式系统里不追求强同步一步到位，而是通过不断重试和状态收敛达到稳定结果。

可以把 K8s 理解为：

- etcd 存储“系统真相”
- API Server 提供统一入口与准入治理
- 各控制器和节点代理不断把现实世界拉回声明目标

#### 二、主要组件与作用 (控制平面)

##### 1. kube-apiserver

作用：集群唯一权威入口，负责认证、鉴权、准入控制、对象校验、版本化 API 输出。

关键点：

- 所有组件几乎都通过它读写状态 (而不是直接写 etcd)
- 通过 Watch 机制向控制器和客户端推送资源变化
- 典型高频问题包括 QPS 限流、缓存命中、长连接 Watch 中断

##### 2. etcd

作用：存储集群全量元数据与状态的分布式 KV 数据库。

关键点：

- 基于 Raft 共识保证一致性
- 保存 Pod、Deployment、ConfigMap、Lease 等关键对象
- 面试排障会涉及 compaction、defrag、snapshot、成员健康度

`etcdctl` 示例 (只读健康检查)：

```bash
# 需要在控制平面节点或具备证书环境执行
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/peer.crt \
  --key=/etc/kubernetes/pki/etcd/peer.key \
  endpoint health
```

##### 3. kube-scheduler

作用：为未绑定节点的 Pod 选择最合适 Node。

核心流程：

- 过滤 (Filter)：剔除不满足资源/亲和性/污点容忍的节点
- 打分 (Score)：对候选节点按策略打分
- 绑定 (Bind)：把 `spec.nodeName` 回写到 Pod

关键点：

- 调度失败不是“容器问题”，而是调度约束问题
- 排障优先看 Pod Events 中的 `FailedScheduling`

##### 4. kube-controller-manager

作用：运行一组控制器，负责系统对象的持续对账。

常见控制器：

- Deployment/ReplicaSet Controller (副本数收敛)
- Node Controller (节点心跳与驱逐)
- EndpointSlice Controller (服务后端维护)
- Job/CronJob Controller (批任务编排)

本质：控制器 = Watch -> Diff -> Act，失败后按重试队列继续收敛。

#### 三、主要组件与作用 (节点平面)

##### 1. kubelet

作用：节点代理，负责把 PodSpec 转成真实运行容器。

关键职责：

- 监听分配到本节点的 Pod
- 调用 CRI 启停容器
- 调用 CNI 配网
- 调用 CSI 挂载存储
- 上报 Node/POD 状态与探针结果

##### 2. container runtime (containerd/runc)

作用：真正创建容器进程与隔离环境。

关键点：

- containerd 负责镜像与容器生命周期管理
- runc 按 OCI 规范创建 namespace 与 cgroups

##### 3. kube-proxy

作用：实现 Service 的四层转发能力。

关键点：

- iptables 模式：写 NAT 规则完成后端分发
- IPVS 模式：使用内核虚拟服务，规则规模大时性能更稳

##### 4. CNI 插件 (如 Calico/Cilium/Flannel)

作用：为 Pod 分配 IP 并打通跨节点网络连通性。

关键点：

- Pod 创建时执行 CNI `ADD`
- Pod 删除时执行 CNI `DEL`
- NetworkPolicy 生效也依赖 CNI 数据面实现 (iptables/eBPF)

#### 四、一次典型请求链路 (从提交到运行)

```plaintext
kubectl apply -f deploy.yaml
  ↓
API Server (认证/鉴权/准入)
  ↓
etcd 持久化对象
  ↓
Controller Manager 发现副本差异，创建 Pod
  ↓
Scheduler 为 Pod 选择 Node
  ↓
kubelet 拉镜像并调用 CRI/CNI/CSI
  ↓
Pod Running + kube-proxy 更新 Service 转发
```

这个链路说明了一个关键面试点：K8s 是“控制平面驱动的数据面执行系统”，每个组件都在闭环中有明确职责。

#### 五、常见面试追问与排障视角

1. 为什么 API Server 负载高会影响全局

- 因为它是唯一控制入口，控制器、调度器、kubelet 都依赖它

2. 为什么 etcd 抖动会导致集群异常

- 状态持久化与读取延迟上升，控制循环会整体变慢甚至失败

3. 为什么 Pod 运行慢不一定是镜像问题

- 可能卡在调度、网络初始化、卷挂载、探针失败等阶段

4. 为什么 kube-proxy 问题会表现为“服务偶发不通”

- Service 转发表不一致、规则未及时刷新、后端健康状态波动都可能导致间歇故障

常用命令示例：

```bash
# 看控制平面组件状态
kubectl get componentstatuses

# 看 Pod 调度与节点信息
kubectl get pod -A -o wide

# 看关键事件流
kubectl get events -A --sort-by=.lastTimestamp

# 聚焦某个Pod调度和容器启动细节
kubectl describe pod <pod-name> -n <ns>
```

#### 六、面试收束模板

可以这样总结：

- Kubernetes 是声明式容器编排与集群控制系统，核心是“API + 控制器 + 最终一致性”
- 控制平面负责决策和状态收敛 (API Server、etcd、Scheduler、Controller Manager)
- 节点平面负责执行和上报 (kubelet、runtime、kube-proxy、CNI)
- 生产排障应按“提交 -> 持久化 -> 调度 -> 启动 -> 网络/存储 -> 服务转发”的链路逐段定位

知识扩展：
Kubernetes 组件协作与 Operator 模式强相关，因为 Operator 本质是定制 Controller 扩展控制平面能力；与 Service Mesh 强相关，因为流量治理会叠加在 kube-proxy/CNI 之上；与可观测体系强相关，因为控制环路问题需要通过 metrics、events、logs、traces 联合定位。

## 2. Docker 基础面试题

### 2.1 请解释 Docker 的基本概念及其核心组件

这道题建议按“概念 -> 组件 -> 内核机制 -> 执行链路 -> 排障视角”回答，不要只停留在“Docker 是容器工具”。

核心结论：

- Docker 是一个围绕 OCI 规范构建的容器打包、分发、运行平台
- 它通过镜像分层、容器运行时和宿主机内核隔离机制，把应用及依赖标准化交付
- 容器不是轻量虚拟机，本质是宿主机上的受限进程 (受 Namespace/Cgroups/Capabilities 等约束)

#### 一、Docker 的基本概念

##### 1. 镜像 (Image)

- 镜像是只读模板，包含应用代码、运行时依赖、启动命令等
- 镜像采用分层结构 (Layer)，复用层可显著降低构建与分发成本
- 常见格式符合 OCI Image 规范

##### 2. 容器 (Container)

- 容器是镜像的运行实例
- 运行时在镜像只读层之上叠加可写层 (Writable Layer)
- 容器生命周期短、可重建，配置和数据应尽量外置 (Volume/Config/Secret)

##### 3. 仓库 (Registry)

- Registry 用于存储和分发镜像 (如 Docker Hub、Harbor)
- 企业场景通常需要私有仓库、镜像签名、漏洞扫描与准入拦截

##### 4. Dockerfile

- Dockerfile 是镜像构建脚本，描述“如何从基础镜像构建业务镜像”
- 面试高频点是多阶段构建 (multi-stage build) 和最小化镜像

#### 二、Docker 核心组件与作用

##### 1. Docker CLI

作用：用户入口，用于执行 `docker build/run/pull/push/exec/logs` 等命令。

##### 2. Docker Engine (dockerd)

作用：守护进程，负责镜像管理、容器生命周期、网络与存储编排。

关键点：

- 对上接收 CLI/API 请求
- 对下调用 containerd/runc 执行容器
- 管理本地镜像层、容器元数据、网络与卷

##### 3. containerd

作用：容器生命周期管理器，负责镜像拉取、快照管理、任务管理。

关键点：

- 通过 snapshotter 管理镜像层与可写层
- 调用低层运行时 (通常 runc) 创建容器进程

##### 4. runc

作用：OCI Runtime 实现，真正执行 `clone/unshare/setns/cgroup` 等系统调用创建受限进程。

关键点：

- runc 本身不管理集群，也不做镜像分发
- 它负责把 OCI bundle 变成一个运行中的容器进程

##### 5. Registry + Distribution

作用：完成镜像分发链路，包括层上传、层下载、Manifest 管理与标签解析。

#### 三、底层 Linux 内核机制 (面试加分点)

Docker 的隔离与限制主要依赖以下内核能力：

1. Namespace (隔离视图)

- `pid`：隔离进程号空间
- `net`：隔离网络栈 (网卡、路由、iptables 视图)
- `mnt`：隔离挂载点
- `uts`：隔离主机名
- `ipc`：隔离进程间通信对象
- `user`：隔离用户与权限映射

2. Cgroups (资源配额)

- CPU 配额与权重
- 内存上限与 OOM 行为
- I/O 限速
- PIDs 数量限制

3. UnionFS / OverlayFS (分层文件系统)

- 下层只读镜像层 + 上层可写层合并为容器视图
- Copy-on-Write 降低磁盘占用与构建成本

4. Capabilities + Seccomp + LSM

- Capabilities：细粒度拆分 root 权限
- Seccomp：限制可调用系统调用集合
- AppArmor/SELinux：强制访问控制

#### 四、一次 `docker run` 的执行链路

```plaintext
docker run nginx:1.27
  ↓
CLI 调用 Docker Engine API
  ↓
dockerd 检查本地镜像，不存在则从 Registry 拉取
  ↓
containerd 创建容器元数据与快照层
  ↓
runc 创建 Namespace/Cgroups 并启动容器主进程
  ↓
Docker 网络驱动配置 veth/bridge/NAT
  ↓
容器进入 Running，stdout/stderr 接入日志驱动
```

这个链路可以很好回答“Docker 到底做了什么”而不是只说“起了一个容器”。

#### 五、可落地示例

##### 1. 多阶段 Dockerfile 示例

```dockerfile
# Stage 1: 构建阶段
FROM golang:1.22 AS builder
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o app ./cmd/server

# Stage 2: 运行阶段 (最小化镜像)
FROM gcr.io/distroless/static:nonroot
WORKDIR /app
COPY --from=builder /src/app /app/app
USER nonroot:nonroot
ENTRYPOINT ["/app/app"]
```

说明：

- 多阶段构建可显著减小镜像体积
- distroless 减少攻击面，适合生产环境

##### 2. 常用命令与排障命令

```bash
# 构建镜像
docker build -t demo/app:1.0.0 .

# 启动容器并映射端口
docker run -d --name app -p 8080:8080 demo/app:1.0.0

# 查看容器进程与资源限制
docker inspect app

# 查看日志
docker logs -f app

# 进入容器排障
docker exec -it app sh

# 查看实时资源使用
docker stats app
```

#### 六、常见误区与排障视角

1. 误区：容器和虚拟机是同一层隔离

- 不准确。容器共享宿主机内核，隔离边界弱于硬件虚拟化

2. 误区：镜像越小越好

- 不完整。应在“体积、安全、可观测、可调试”之间平衡

3. 误区：容器内数据天然可靠

- 错误。可写层随容器销毁而丢失，持久化必须依赖 Volume

4. 常见故障排障顺序

- 启动失败：先看 `docker logs`，再看 `docker inspect` 的 entrypoint/cmd
- 资源问题：看 `docker stats` 和 cgroup 限制
- 网络问题：看端口映射、bridge 网络、宿主机防火墙/NAT 规则

#### 七、面试收束模板

可以这样总结：

- Docker 的本质是基于 OCI 标准的容器交付与运行平台
- 核心组件是 CLI、dockerd、containerd、runc 与 Registry
- 关键内核机制是 Namespace、Cgroups、OverlayFS 与安全策略
- 生产落地要关注镜像治理、运行时安全、资源限制与可观测排障链路

知识扩展：
Docker 与 Kubernetes 强相关，因为 K8s 的 Pod 运行最终依赖 CRI 运行时和同一套内核隔离机制；与镜像供应链安全强相关，因为镜像签名、SBOM、漏洞扫描直接影响生产风险；与 Service Mesh 强相关，因为容器网络与 sidecar 注入都建立在容器网络命名空间和 CNI 数据面之上。

### 2.2 Docker 是如何进行容器间的资源隔离的？并且请具体描述分析其底层逻辑

这道题建议按"隔离维度 -> 底层内核机制 -> 系统调用链路 -> 实战验证 -> 排障视角"回答，不要只停留在"用 Namespace 和 Cgroups"。面试要体现你对 Linux 内核系统调用、数据结构和运行时行为的具体理解。

核心结论：

- Docker 的容器隔离不是单一机制，而是 Namespace (视图隔离) + Cgroups (资源限制) + UnionFS (文件系统隔离) + 安全机制 (Capabilities/Seccomp/LSM) 的协同
- 底层由 runc 通过 `clone/unshare/setns` 等系统调用完成内核对象的创建与绑定
- 理解隔离的边界很重要：容器共享宿主机内核，属于逻辑隔离而非硬件级虚拟化

#### 一、Namespace：隔离视图 (What you can see)

Namespace 是 Linux 内核提供的资源视图隔离机制，让进程只能看到属于自己的一组资源。

| Namespace | 系统调用参数 | 隔离资源 |
|-----------|-------------|----------|
| PID       | `CLONE_NEWPID` | 进程号空间 |
| Network   | `CLONE_NEWNET` | 网络设备、端口、路由表、iptables |
| Mount     | `CLONE_NEWNS`  | 挂载点 |
| UTS       | `CLONE_NEWUTS` | 主机名与域名 |
| IPC       | `CLONE_NEWIPC` | 信号量、消息队列、共享内存 |
| User      | `CLONE_NEWUSER`| UID/GID 映射 |
| Cgroup    | `CLONE_NEWCGROUP`| Cgroup 根目录视图 (Linux 4.6+) |
| Time      | `CLONE_NEWTIME` | 容器独立时钟 (Linux 5.6+) |

底层实现逻辑：

- 内核为每个 Namespace 维护独立的数据结构 (如 `struct pid_namespace`、`struct net`)
- `clone()` 创建新进程时通过 flags 指定要独立的 Namespace
- `unshare()` 让已有进程脱离当前 Namespace 并进入新 Namespace
- `setns()` 让进程加入一个已存在的 Namespace

实战验证命令：

```bash
# 查看某容器进程所属的 Namespace
# 以容器内 PID 1 为例 (替换 <container_pid> 为实际宿主机 PID)
ls -l /proc/<container_pid>/ns
# 输出示例：
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 cgroup -> cgroup:[4026531835]
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 ipc -> ipc:[4026532892]
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 mnt -> mnt:[4026532890]
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 net -> net:[4026532895]
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 pid -> pid:[4026532893]
# lrwxrwxrwx 1 root root 0 Apr 26 10:00 uts -> uts:[4026532891]
```

```bash
# 进入容器的 Network Namespace 排障
nsenter -t <container_pid> -n ip addr
nsenter -t <container_pid> -n iptables -t nat -L -n -v

# 进入容器的 PID Namespace 排障 (可看到容器内进程视图)
nsenter -t <container_pid> -p -r ps aux
```

关键面试点：

- 两个容器可以拥有相同的 PID 1，因为它们处于不同的 PID Namespace
- 容器内 `lo` 网卡和 `eth0` 是独立的，因为 Network Namespace 隔离了内核协议栈
- User Namespace 允许容器内 root (UID 0) 映射到宿主机普通用户 (如 UID 100000)，降低特权逃逸风险

#### 二、Cgroups：限制资源 (What you can use)

Cgroups (Control Groups) 是 Linux 内核的资源限制与统计机制，Docker 通过它限制容器的 CPU、内存、IO 等资源使用。

底层架构：

- Cgroups v1：按子系统独立挂载 (cpu、memory、blkio 等)，路径 `/sys/fs/cgroup/<subsystem>/docker/<container_id>`
- Cgroups v2：统一层级，路径 `/sys/fs/cgroup/docker/<container_id>`，使用 BPF 和 eBPF 增强
- 内核通过 `cgroup_subsys_state` 结构将进程与资源限制策略关联

核心子系统与面试要点：

##### 1. CPU 子系统

- `cpu.shares`：相对权重 (默认 1024)，用于 CPU 时间片竞争
- `cpu.cfs_quota_us` / `cpu.cfs_period_us`：硬限制 (如配额 100000/周期 100000 = 1 核)
- `cpuset.cpus`：绑定到特定物理 CPU 核心

```bash
# 查看容器的 CPU Cgroup 限制 (Docker Cgroups v1 路径示例)
cat /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_quota_us
cat /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_period_us
# 若 quota = 50000, period = 100000，表示限制使用 0.5 核
```

##### 2. Memory 子系统

- `memory.limit_in_bytes`：内存硬上限
- `memory.soft_limit_in_bytes`：软限制 (不强制 OOM，但系统紧张时优先回收)
- `memory.kmem.limit_in_bytes`：内核内存限制 (防止内核级 OOM)
- OOM Killer 行为：当容器内存超过 limit，内核会杀死容器内评分最高的进程 (通常是占用内存最多的)

```bash
# 查看容器的内存限制
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes

# 查看容器是否发生过 OOM
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.oom_control
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.failcnt
```

##### 3. Pids 子系统

- `pids.max`：限制容器内进程/线程数，防止 fork 炸弹

```bash
# 查看容器的 PID 限制
cat /sys/fs/cgroup/pids/docker/<container_id>/pids.max
cat /sys/fs/cgroup/pids/docker/<container_id>/pids.current
```

Docker 运行时的资源限制示例：

```bash
# 启动一个资源受限的容器
docker run -d \
  --name limited-app \
  --cpus="1.5" \
  --memory="512m" \
  --memory-swap="512m" \
  --pids-limit=100 \
  --blkio-weight=300 \
  nginx:1.27

# --cpus="1.5" 底层映射为 cpu.cfs_quota_us=150000 / cpu.cfs_period_us=100000
# --memory="512m" 底层映射为 memory.limit_in_bytes=536870912
# --pids-limit=100 底层映射为 pids.max=100
```

#### 三、文件系统隔离：OverlayFS 与 Rootfs

容器需要独立的文件系统视图，但又要避免镜像层被修改。

底层逻辑：

- Docker 使用 OverlayFS (或 aufs/zfs/btrfs) 实现联合挂载
- 容器启动时，内核将多个只读层 (lowerdir) 和一个可写层 (upperdir) 合并为统一视图 (merged)
- 写操作触发 Copy-on-Write：修改只读层文件时，内核将其复制到可写层再修改

挂载结构：

```plaintext
lowerdir=/var/lib/docker/overlay2/l/XXX:/var/lib/docker/overlay2/l/YYY  (镜像层)
upperdir=/var/lib/docker/overlay2/<container_id>/diff                   (容器可写层)
workdir=/var/lib/docker/overlay2/<container_id>/work                    (工作目录)
merged=/var/lib/docker/overlay2/<container_id>/merged                   (容器根目录视图)
```

验证命令：

```bash
# 查看容器的 OverlayFS 挂载详情
docker inspect -f '{{.GraphDriver.Data}}' <container_id>

# 查看宿主机上的挂载点
mount | grep overlay | grep <container_id>

# 直接查看容器在宿主机上的可写层文件
ls /var/lib/docker/overlay2/<container_id>/diff
```

面试要点：

- OverlayFS 让容器看似拥有完整文件系统，实际数据可能分散在多个层
- 大量写操作会导致 Copy-on-Write 放大，影响 IO 性能 (数据库类应用建议用 Volume)
- 容器删除后，upperdir (可写层) 也会被清理，因此重要数据必须外挂 Volume

#### 四、安全加固机制：Capabilities + Seccomp + LSM

隔离不等于安全，Docker 还通过以下机制缩小攻击面：

##### 1. Linux Capabilities

- 传统 root 权限被拆分为多个细粒度 Capability (如 `CAP_NET_ADMIN`、`CAP_SYS_PTRACE`)
- 默认 Docker 容器会丢弃大部分危险 Capability，只保留 14 个安全的基础 Capability
- 可通过 `--cap-add` 和 `--cap-drop` 精细控制

```bash
# 查看容器当前的 Capability
docker inspect -f '{{.HostConfig.CapAdd}} {{.HostConfig.CapDrop}}' <container_id>

# 启动一个禁用所有 Capability 的容器 (最严格)
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE nginx:1.27

# 查看进程的 Capability 位图 (宿主机视角)
cat /proc/<container_pid>/status | grep Cap
# CapInh: 继承的 Capability
# CapPrm: 允许的 Capability
# CapEff: 生效的 Capability
# CapBnd: 边界 Capability (上限)
```

##### 2. Seccomp (Secure Computing Mode)

- 通过 BPF 过滤器限制容器进程可发起的系统调用
- Docker 默认启用 seccomp profile，禁止约 44 个危险系统调用 (如 `mount`、`umount2`、`pivot_root` 等)
- 可通过 `--security-opt seccomp=profile.json` 自定义

```bash
# 查看容器的 seccomp 状态
docker inspect -f '{{.HostConfig.SecurityOpt}}' <container_id>

# 以无 seccomp 限制运行 (不推荐生产环境)
docker run --security-opt seccomp=unconfined nginx:1.27
```

##### 3. LSM (Linux Security Modules)：AppArmor / SELinux

- AppArmor (Ubuntu/Debian 默认)：基于路径的强制访问控制
- SELinux (RHEL/CentOS 默认)：基于标签 (label) 的强制访问控制
- Docker 默认启用 AppArmor profile `docker-default`，限制容器对宿主机文件的访问

```bash
# 查看容器的 AppArmor 策略
docker inspect -f '{{.AppArmorProfile}}' <container_id>

# 查看进程的 AppArmor 状态
cat /proc/<container_pid>/attr/current
# 输出示例：docker-default (enforce) 表示正在强制执行 docker-default 策略
```

#### 五、一次容器启动时的隔离链路 (runc 视角)

当 `docker run` 执行时，runc 通过以下步骤建立完整隔离环境：

```plaintext
runc create <container-id>
  ↓
1. 创建 Namespace：调用 unshare(CLONE_NEWNS|CLONE_NEWPID|CLONE_NEWNET|...)
  ↓
2. 设置 Cgroups：向 /sys/fs/cgroup/ 各子系统写入限制参数，并将容器 PID 加入 tasks 文件
  ↓
3. 挂载 Rootfs：通过 mount(MS_BIND|MS_REC, ...) 挂载 OverlayFS 到容器根目录
  ↓
4. 写入 UID/GID 映射：配置 /proc/<pid>/uid_map 和 gid_map (User Namespace)
  ↓
5. 应用 Seccomp 过滤器：通过 prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, ...) 加载 BPF 规则
  ↓
6. 应用 Capabilities：通过 capset() 系统调用设置进程的 Capability 位图
  ↓
7. pivot_root 切换根目录：将容器根目录切换为 merged 层，旧根挂载到 /.pivot_root 后卸载
  ↓
8. execve() 启动用户进程：容器主进程开始执行 entrypoint/cmd
```

面试要点：

- `pivot_root` 与 `chroot` 的区别：`pivot_root` 更安全，它改变了进程的根挂载点和当前工作目录，而 `chroot` 只是改变了根目录查找路径
- 容器内执行 `ps aux` 只能看到 PID Namespace 内的进程，是因为 `/proc` 挂载在新的 PID Namespace 下

#### 六、常见问题与排障路径

1. 资源限制未生效：
   - 检查 Cgroups v1/v2 混合环境是否导致配置路径错误
   - 检查宿主机内核是否编译了对应的 Cgroup 子系统 (`grep CGROUP /boot/config-$(uname -r)`)

2. 容器内存 OOM：
   - 先看 `docker stats` 确认是否达到 limit
   - 再看宿主机 `dmesg -T | grep -i "killed process"` 找到 OOM Killer 日志
   - 若 `memory.failcnt` 持续增长，说明频繁触发限制

3. 网络不通但端口已映射：
   - 进入容器 Network Namespace：`nsenter -t <pid> -n`
   - 检查容器内 iptables、路由表、网卡状态
   - 检查宿主机是否开启 ip_forward：`cat /proc/sys/net/ipv4/ip_forward`

4. 权限不足 (Permission denied)：
   - 检查 Capabilities：`cat /proc/<pid>/status | grep Cap`
   - 检查 AppArmor/SELinux 日志：`dmesg | grep -i apparmor`
   - 检查文件系统挂载选项是否包含 `noexec`

5. 容器内看到宿主机进程：
   - 说明 PID Namespace 未正确创建，常见于特权模式 (`--privileged`) 或 `hostPID: true` 的 Pod

#### 七、面试收束模板

可以这样总结：

- Docker 的资源隔离是 Namespace (视图隔离) + Cgroups (资源限制) + OverlayFS (文件系统隔离) + 安全机制 (Capabilities/Seccomp/LSM) 的组合拳
- Namespace 通过内核数据结构复制实现视图分离，Cgroups 通过内核调度器与资源计数器实现硬限制
- runc 通过 `clone/unshare/setns`、`mount`、`pivot_root`、`prctl` 等系统调用完成这些隔离的落地
- 容器隔离是逻辑隔离，共享宿主机内核，与虚拟机有本质区别
- 排障时应按 Namespace -> Cgroups -> 文件系统挂载 -> 安全策略的层次逐层定位

知识扩展：
Docker 容器隔离与 Kubernetes 强相关，因为 K8s 通过 SecurityContext 将 Namespace/Capabilities/SELinux 等配置下发到 Pod；与 CRI 运行时强相关，因为 containerd/runc 才是实际执行隔离系统调用的组件；与 eBPF 可观测强相关，因为内核级排障工具 (如 bcc、bpftrace) 可以穿透 Namespace 边界观察容器内行为；与云安全强相关，因为容器逃逸 (Container Escape) 的本质就是突破 Namespace/Cgroups 边界获取宿主机权限。
